## Role in Authority Stack — Vendor Reference Authority

FQH-5026 is the **authoritative vendor reference for the Five-Question Cognitive Hierarchy** as the primary Phase 1 UX acceptance criterion for Trackaroo®. Consolidates, in a single vendor-facing table, the five questions every user must be able to answer **at any time without deliberation**, together with the subsystem responsible for each answer and the failure condition that constitutes a breach of the acceptance criterion.

Issued as a component of the Phase 1 vendor pack alongside RFT-5026, VRB-5126, VEG-5026, VGD-5126, PSB-5026, and SFD-5026. **Does not create new obligations — it consolidates direction established in UXS-5726 §3 and PRD-5126 §10.** In any conflict, the primary authority stack resolves upward.

---

## 2. Governing Principle

The Five-Question Cognitive Hierarchy is the **primary UX acceptance criterion for Phase 1**. Every design and implementation decision — across navigation, hazard awareness, emergency readiness, group communication, and first aid — must be evaluated against this framework. **The five questions are not aspirational. They are a hard acceptance floor.**

All five questions must be answerable in:
- **A fully offline state**
- **Under stress**
- **With gloves**
- **In low-light conditions**
- **On first use**

This is the operative condition set for acceptance. User testing must be conducted across **all six archetypes and all major screen states (TQP-5026)**.

## 3. Five-Question Cognitive Hierarchy — Reference Table

Authority sources: **UXS-5726 §3 · PRD-5126 §10 · BTF-5126 §3 · ESF-5026 · SFD-5026**

| # | Question | Subsystem Responsibility | Failure Mode | Governing Authority |
|---|---|---|---|---|
| **1** | **Where am I?** | Navigation Engine, Offline Map | Position uncertainty; map blank or stale | UXS-5726 §3; PRD-5126 §10 |
| **2** | **Where am I going?** | Route Planning, Breadcrumb Trail | No active route; lost trail context | UXS-5726 §3; PRD-5126 §10 |
| **3** | **What surrounds me?** | HazTrack™, Overlay System | Hidden hazards; stale data not disclosed to user | UXS-5726 §3; PRD-5126 §10; OSM-5026 |
| **4** | **How do I get back?** | BackTrack™, Return Navigation | Breadcrumb loss; retrace unavailable offline; trail load >3 seconds; BackTrack™ not accessible within ≤3 taps from any navigation screen state | UXS-5726 §3; BTF-5126 §3; PRD-5126 §10 |
| **5** | **How do I call for help?** | SOS Module (≤2 tap) | SOS inaccessible or buried in UI; ≤2 tap discipline violated; non-dispatch posture not maintained | UXS-5726 §3; PRD-5126 §10; ESF-5026; SFD-5026 |

> **CONSTRAINT — Primary Acceptance Criterion:**
> Any design decision that impairs a user's ability to answer any of these five questions **without deliberation** is a failure against the primary acceptance criterion. All five questions must be answerable in a fully offline state, under stress, with gloves, in low-light conditions, and on first use.
> 
> **Five-question hierarchy failure is a named rejection trigger in PRD-5126 §14.4.** Failure at any gate halts release progression and requires written resolution from the Project Director before development may continue.
> 
> Authority: UXS-5726 §3; PRD-5126 §10; VGD-5126 rejection triggers.

> **NOTE — Question 4 Extended Obligations:** BTF-5126 §3 defines extended compliance for BackTrack™. Must reconstruct exact reverse trail from locally stored breadcrumb data, **fully offline, within ≤3 seconds of user initiation**. Must be accessible within **≤3 taps from any navigation screen state without exception**.

> **NOTE — Question 5 Extended Obligations:** SFD-5026 (SOS Flow Diagram, 29 March 2026) is the governing authority for SOS activation sequence, the **3-stage log discipline**, the **six-element confirmation screen**, **non-dispatch posture**, and all SOS copy discipline. Vendors must read SFD-5026 in full before proposing any SOS architecture or UX approach.

## 4. Acceptance & Enforcement

Vendor proposals, architecture documents, and delivery gate submissions will be evaluated against the five-question hierarchy as a **primary acceptance lens**. Demonstrated understanding of each question, its responsible subsystem, and its failure condition is required across:
- VRB-5126 §4 (Response Section 1 — Technical Architecture)
- Each delivery gate (VGD-5126 §7)

**Acceptance authority:** Project Director.
**Validation authority:** TQP-5026.
**Enforcement authority:** VGD-5126.
All gate clearances require **written sign-off from the Project Director**.
