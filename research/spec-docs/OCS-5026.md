# OCS-5026 · Operations Console Specification

**Source:** `04_SUPPORTING GUIDES FRAMEWORK DOCUMENTS/Trackaroo OCS-5026 Operations Console.pdf` (20 pages)
**Date issued:** 30 Mar 2026 · Ian Moore — Founder & Project Director
**Subtitle:** Phase 1 Operator Tooling — Web-Based Internal Administration Platform

> "OCS-5026 defines the Operations Console Specification for trackaroo® Phase 1. **Web-based internal administration platform** providing Trackaroo Systems Pty Ltd with operator tooling necessary to manage/monitor/maintain all Phase 1 data systems from GA (13 Nov 2026) onwards."

**The operations console is a confirmed Phase 1 deliverable.** NOT a consumer-facing product. NOT a public portal. **Internal operator tool, accessible only to authorised Trackaroo Systems personnel.**

Without operations console, following Phase 1 obligations cannot be discharged: PCR supersession governance (OSM-5026 §10.3) · difficulty grade authorised contributor review (OSM-5026 §6.3) · HazTrack™ feed status monitoring (HFG-5026 §8) · TrackIQ™ pipeline operational oversight · user account administration for Beta testing + post-launch support.

---

## §2 Authority
OCS-5026 issued by Project Director, constitutes formal Phase 1 specification. All authority resides with PD. No modification without written authorisation recorded in CAR-5026.

OCS-5026 operates **subordinate to UXS-5726 / PRD-5126 / FSD-5126 / TQP-5026 / CDG-5126**. Sits alongside specialist domain docs (OSM-5026, HFG-5026, FRM-5126, TAA-5126), drawing functional requirements directly from them. Conflicts resolve upward through Section 4 authority hierarchy.

---

## §3 Scope

OCS-5026 governs:
- Functional specification of all Phase 1 ops console modules
- Role-based access control (RBAC) model + authentication requirements
- Staged delivery model + gate conditions for console readiness
- Survival Core data isolation constraint applying to all console surfaces
- System audit log requirements
- Vendor response obligation **V-15** introduced by this document

OCS-5026 does NOT govern: mobile application · companion website · consumer-facing product · TrackIQ™ scoring algorithm (FSD-5126) · HazTrack™ feed schema (HFG-5026). Console provides operational visibility + governance tooling over those systems; does NOT redefine them.

Operations console is a **separate web application**, hosted independently of trackaroo® companion website. **NOT publicly accessible.**

## §5 Constraints

> **CONSTRAINT — Survival Core Isolation:** Ops console must NEVER surface Survival Core data to any console user or role. Breadcrumb data, SOS event logs, local event logs are classified Local-Only — Non-Syncable under CDG-5126 + MUST never be accessible, searchable, exportable, or visible via any console interface, API endpoint, or query path. Any console architecture providing any pathway to these data categories constitutes automatic non-compliance with CDG-5126 §4 + is a named rejection trigger.

> **CONSTRAINT — Internal Tool:** Console is an internal operator tool. MUST NOT be accessible to public, to delivery partner personnel (except under time-limited arrangement in §6.1), or to any party not holding active console role assigned by PD.

> **CONSTRAINT — Grade Workflow Discipline:** No grade value may be written to production data set by any automated process, pipeline output, or TrackIQ™ scoring result **without passing through Grade Review Queue in §6.3** + receiving explicit Authorised Contributor or PD approval. Automated grade assignment bypassing this workflow = violation of OSM-5026 §6.3.

> **CONSTRAINT — Clinical Review Gating:** No First Aid Reference content (any tier) may be activated via Content Release Flag (§6.7) without written clinical review completion + written PD sign-off. Content version that has not completed clinical review must NEVER be deployed to production regardless of business urgency.

## §6 Content

### §6.0 Access Control Model
Console operates **3-role RBAC** model. All roles authenticate via **Firebase Authentication** with role claims managed server-side via Firebase custom claims. Role assignment + revocation = exclusive authority of PD. No self-service role elevation permitted.

| Role | Module Access | Restrictions |
|---|---|---|
| **Project Director** | Full access to all modules. Sole authority for: First Aid Reference content release; Authorised Contributor assignment/revocation; account suspension; full audit log access; system configuration. | None — highest privilege role |
| **Operations** | PCR Management; HazTrack™ Feed Management; TrackIQ™ Pipeline Operations; User & Account Administration (read + tier adjustment only — no suspension); Beta & Alpha Tester Management | No access to First Aid Reference content release. No account suspension. Audit log: own actions only |
| **Authorised Contributor** | Track Data & Difficulty Grade Administration module ONLY — grade review queue within declared domain | No access to any other module. Audit log: own actions only. No user data access |

> **Authorised Contributor** = **data governance role**, NOT employee category. Assigned by PD to any individual granted authority to review + approve difficulty grade changes for declared track domain. Role carries no other console privilege. Delivery partner access to Beta & Alpha Tester Management module during test phases granted under **time-limited Operations role**, assigned by PD + revoked at Beta-Ready gate clearance.

#### Authentication & Session Rules
- MFA mandatory for PD + Operations roles
- Sessions expire after **8 hours of inactivity** for all roles
- Failed login attempts rate-limited — max 5 before temporary lockout
- All login events (success + failure) recorded in system audit log
- Role changes take effect on next session — active sessions NOT immediately invalidated on role change

Console delivered in **2 stages** aligned to Phase 1 gate schedule.

| Gate | Date | Required Console Modules | Rationale |
|---|---|---|---|
| **Alpha** | 22 Aug 2026 | **Stage 1:** PCR Management (full); User & Account Administration (full); Beta & Alpha Tester Management (full); System Audit Log (all actions covered) | Alpha introduces human testers — PCR submissions will occur · Tester account management must be operational · Audit log must be active from first console use |
| **Beta-Ready MVP** | 30 Oct 2026 | **Stage 2 — All remaining:** Track Data & Difficulty Grade Administration; TrackIQ™ Pipeline Operations; HazTrack™ Feed Management; First Aid Reference Content Administration. Full RBAC operational. All 3 roles assignable. | Beta-Ready represents full Phase 1 scope delivery. Full console readiness required before GA clearance can be granted |

Written sign-off required at Alpha (Stage 1) + Beta-Ready (all modules). Console module that fails gate validation halts relevant gate clearance until remediation + revalidation.

### §6.2 PCR Management Module
**Most operationally critical console component.** OSM-5026 §10.3 defines supersession resolution — PCRs resolved by supersession only, never by deletion or TTL expiry. CDG-5126 §5.5 enforces immutability — original submitted record preserved in original form in all storage contexts.

| Capability | Description | Roles |
|---|---|---|
| **Active PCR Queue View** | Live view of all active PCR submissions. Filterable by category (PCR-OBS/CLO/INF/SRF/WTR/HAZ), submission date, geographic bbox. Displays: pcr_id, category, timestamp, coordinates, description, resolution state, sync status | PD, Operations |
| **90-Day Unconfirmed-Age Alert Queue** | Dedicated queue surfacing all PCRs exceeded 90 days without supersession event. Per OSM-5026 §10.4. **No automated action — informational only** | PD, Operations |
| **PCR Supersession (Operator)** | Operator may initiate supersession event for demonstrably incorrect/requiring resolution PCR. Creates new superseding record with: supersession type = `operator_review`; operator identity (anonymised); timestamp; reason code. Original PCR archived, NOT deleted. Reason codes: Duplicate, Incorrect Location, Condition Resolved, Data Quality | PD, Operations |
| **PCR Suppression** | In exceptional circumstances, PCR may be suppressed — sets suppressed flag + removes from public display. Record never deleted. Requires reason entry + logged as PD action | PD only |
| **PCR History View** | Full submission + supersession history for given track point coordinate. All PCRs ever submitted at that point, chronological. **Immutable — no edit capability** | PD, Operations |
| **PCR Sync Status Dashboard** | View of queued vs synced PCR submissions. Informational only — no manual sync trigger (sync device-initiated per CDG-5126 §5.3) | PD, Operations |

> **NOTE:** PCR data in console MUST NOT expose PII. `reporter_archetype` (anonymised archetype token) = only reporter attribute visible. PCR export = Phase 2 item. All console actions on PCR records recorded in system audit log.

### §6.3 Track Data & Difficulty Grade Administration Module
OSM-5026 §6.3: difficulty grade changes require **verified grading review by authorised contributor**. No grade change may be applied to production data set without passing through this console workflow.

| Capability | Description | Roles |
|---|---|---|
| **Track Register** | Searchable register of all tracks. Displays: track ID, current grade, grade colour per OSM-5026 §6.2, last authorised review date, reviewer identity (role token), verification shield state. Read-only for Operations. Edit-eligible only via Grade Review Queue | PD, Ops, AC (own domain) |
| **Grade Review Queue** | Queue of pending grade review submissions. Displays: track ID, current grade, proposed grade, submitting contributor, date, supporting data summary. Reviewer approves or rejects. Approval writes new immutable grade record with reviewer identity + timestamp | PD, AC (own domain) |
| **Grade Review Approval** | On approval: new grade record written with grade value, reviewer role token, timestamp, source (`contributor_review`). Previous grade archived — NOT deleted. Grade propagates to production map tiles via pipeline trigger | PD, AC (own domain) |
| **Grade Review Rejection** | On rejection: reason entered + logged. Submission closed. No grade change applied. Submitting contributor notified via console notification. Logged in audit | PD, AC (own domain) |
| **Contributor Management** | PD can: assign Authorised Contributor role; declare domain (geographic region or track category); set expiry date; revoke at any time. Role changes take effect on next login | PD only |
| **Ungraded Track View** | Filtered view of all tracks with Unknown/Ungraded (Grey) status. Informational only — supports prioritisation of grading efforts | PD, Operations |

### §6.4 TrackIQ™ Pipeline Operations Module
TrackIQ™ scoring pipeline (ingest → DEM enrichment → scoring → tile publish) = vendor-built + vendor-operated at infrastructure level. Pipeline Operations Module provides Trackaroo Systems with **operational visibility + break-glass intervention capability**. Monitoring + oversight surface — NOT a pipeline configuration tool.

| Capability | Description | Roles |
|---|---|---|
| **Pipeline Status Dashboard** | Real-time status of each pipeline stage (Ingest, DEM Enrichment, Scoring, Tile Publish). Each stage: status (running/idle/error), last successful run timestamp, last run duration, error count | PD, Operations |
| **Pipeline Run History** | Chronological log of all pipeline runs: run ID, start/completion timestamps, stage-level outcomes, error details. **Min 90-day retention** | PD, Operations |
| **Error Alert Surface** | Any pipeline stage entering error state surfaces visible alert. Alert displays: stage name, error type, timestamp, last successful run reference. Persists until next successful completion | PD, Operations |
| **Manual Re-run Trigger** | **Break-glass** capability for PD to trigger manual pipeline re-run for specific stage or full sequence. Logged in audit with timestamp + triggering user role | **PD only** |
| **Scoring Governance Confirmation** | Read-only view: scoring algorithm version in production; last version change date; **vendor attestation that no AI inference or adaptive logic is present**. Refreshed at each pipeline deployment | PD, Operations |

> **NOTE:** Pipeline configuration changes (algorithm updates, DEM source changes, tile publish parameters) = delivery partner actions requiring formal amendment + PD written approval before deployment. NOT actionable from console.

### §6.5 HazTrack™ Feed Management Module
HFG-5026 governs feed eligibility via Five-Pillar Filter. Feed Management Module gives Trackaroo Systems operational visibility over all integrated feeds + limited break-glass cache refresh capability. **Does NOT permit feed configuration changes — those require delivery amendment.**

| Capability | Description | Roles |
|---|---|---|
| **Feed Status Dashboard** | Live status view of all integrated feeds. Per feed: source name, authority body, feed type, last successful ingest timestamp, TTL compliance status (per OSM-5026 §8), ingest cadence (expected vs actual), active error state | PD, Operations |
| **Feed Ingest Latency Alert** | Alert triggered when feed has not ingested within 150% of expected cadence. Informational — does NOT alter overlay display or TTL behaviour | PD, Operations |
| **Manual Cache Refresh Trigger** | Break-glass to trigger manual cache refresh for specific feed. Logged in audit. Does NOT bypass TTL governance | **PD only** |
| **Feed Eligibility Register** | Read-only register of all feeds ever evaluated under Five-Pillar Filter. Displays: feed name, evaluation date, Pillar 1–5 status, eligibility outcome, integration status. **Immutable** | PD, Operations |
| **Prohibited Feed Alert** | If feed endpoint returns data not matching registered authority-origin schema → system generates alert + halts ingest for that feed until discrepancy resolved | PD, Operations |

### §6.6 User & Account Administration Module
Manages registered user accounts for post-launch support, tier management, compliance. Operates on Firebase Firestore user records + App Store/Google Play subscription data. **Must NEVER surface Survival Core data.**

| Capability | Description | Roles |
|---|---|---|
| **User Lookup** | Search by: account ID (Firebase UID), email, archetype. Returns: account ID (hashed for display), registration date, current tier (Free/Plus/Pro), subscription status, archetype, flag status | PD, Operations |
| **Tier Adjustment** | Manual account tier adjustment. Use cases: Beta tester provisioning; support resolution; complimentary access. Logged with: console user role, account ID, previous tier, new tier, reason code, timestamp. Reason codes: Beta Tester, Support Resolution, Complimentary, Error Correction | PD, Operations |
| **Account Flag** | Flag account for review without suspending. Flag visible to PD + Operations. Flagged accounts appear in dedicated review queue. Flag reason required | PD, Operations |
| **Account Suspension** | Suspend user account. Prevents login + access to all app features. Logged with reason code, timestamp, duration (temporary or indefinite). Suspension notification sent to account email. Reversible by PD only | **PD only** |
| **Subscription Status View** | Read-only view of current subscription status from IAP provider. Displays: subscription tier, renewal date, platform, billing issue flags. **Console cannot modify subscription state directly** | PD, Operations |

> **NOTE:** Email addresses = personal information under APPs. Display limited to PD + Operations roles. Any export requires explicit PD authorisation, documented in audit log. APPs compliance review (LE-04, LRB-5026) governs data handling for all personal information accessible via console.

### §6.7 First Aid Reference Content Administration Module
FRM-5126 governs First Aid Reference tier model + clinical review gate. **No First Aid Reference content may be exposed to any user without completed clinical approval + written PD clearance.** Module = release control mechanism for approved content versions.

| Capability | Description | Roles |
|---|---|---|
| **Content Status View** | Read-only of current approved + deployed content version for each tier (Universal Baseline, Plus Clinical Depth, Pro Operational Layer). Displays: content version ID, clinical review completion date, clinical reviewer name (role token), PD approval date, deployment date | PD, Operations |
| **Content Version Register** | Immutable register of all content versions ever approved + deployed. Full approval chain per version: clinical reviewer attestation, PD sign-off, deployment timestamp. **Min 5-year retention** | PD, Operations |
| **Content Release Flag** | Mechanism for PD to activate new approved content version for production. Requires: clinical reviewer attestation reference; PD acknowledgement; reason for version change. **Only PD may activate** | **PD only** |
| **Content Hold** | Mechanism to place hold on content version pending review or recalled. Prevents deployment. Hold reason required. Logged in audit | **PD only** |

> **RELEASE GATE:** No First Aid Reference content (any tier) may be activated via Content Release Flag without written clinical review completion + written PD sign-off. Alpha gate for Universal Baseline: completed clinical review by qualified wilderness paramedic or emergency nurse; written PD clearance. Beta-Ready for Plus + Pro: same.

### §6.8 Beta & Alpha Tester Management Module
Alpha + Beta testing across all 6 archetypes requires structured tester account management. Module enables Trackaroo Systems to provision/monitor/deprovision test accounts without requiring delivery partner involvement for individual account actions. **Alpha gate readiness requirement.**

| Capability | Description | Roles |
|---|---|---|
| **Tester Register** | Register of all invited Alpha + Beta testers. Per tester: account ID (hashed), assigned archetype, assigned tier, test phase (Alpha/Beta), invite date, activation status, last active date. Filterable by archetype + test phase | PD, Operations |
| **Tester Account Provisioning** | Provision new tester account: assign archetype, tier, test phase, access expiry date. Generates console-recorded invite reference. Delivery partner access to this module = time-limited under temporary Operations role assigned + revoked by PD | PD, Operations |
| **Tester Adjustment** | Adjust archetype/tier for active tester. Logged with reason. Use case: archetype coverage gap → tester reassigned | PD, Operations |
| **Tester Access Revocation** | Revoke tester access at any time. Account set to expired. Logged with reason code + timestamp | PD, Operations |
| **Test Coverage Dashboard** | Informational view: tester count + activation status by archetype + tier. Identifies coverage gaps. No automated action | PD, Operations |

### §6.9 System Audit Log
**Every console action that modifies record state, changes role assignment, releases content, or triggers system event must be recorded in system audit log.** Evidentiary record of all console governance activity. **Immutable after write.**

| Field | Type | Required | Description |
|---|---|---|---|
| `audit_id` | UUID | Yes | Unique identifier at log write time. Immutable |
| `timestamp` | ISO 8601 UTC | Yes | Server-side timestamp. Immutable |
| `console_user_role` | Enum | Yes | Role of console user taking action. **No personal name stored** |
| `module` | String | Yes | Console module in which action occurred |
| `action_type` | String | Yes | Action taken (e.g. `pcr_supersession`, `grade_approval`, `tier_adjustment`, `content_release`) |
| `record_id` | UUID/String | Conditional | Reference to affected record. Null for system-level actions |
| `previous_state` | JSON | Conditional | State before action. Required where record state modified |
| `new_state` | JSON | Conditional | State after action. Required where record state modified |
| `reason_code` | String | Conditional | Reason code where required by module specification |
| `reason_text` | String | Optional | Free-text reason entry where reason_code present |

**Retention + access rules:**
- **Min audit log retention: 7 years** from date of entry
- Audit log NOT exportable by Operations or Authorised Contributor roles
- **PD may export full audit log in JSON or CSV format.** Export event is itself logged
- Audit log Firestore collection must be isolated with **write-once security rules** permitting appends only
- Operations + Authorised Contributor roles have read access to **their own audit entries only**
