## §3 Phase 1 Strategic Position

> "Phase 1 is a fully operational competitive MVP. It is the **maximum defensible launch** — survival-grade, offline-first, satellite-ready in architecture, and compliance-driven from Day 1."

Phase 1 must:
- Compete at category-leading feature density across all 6 declared archetypes
- Operate fully offline — zero connectivity dependency for any Survival Core flow
- Deliver complete Survival Core capability without compromise
- Deliver complete Experience Layer capability within UXS-5726 constraints
- Remain architecturally expandable to Phase 2/3 without structural rework

**Competitive baseline** — must match/exceed: Hema Maps, Newtracs, GAIA GPS, equivalents.

**Grant narrative dimensions:** Public safety outcomes · Remote worker welfare · Emergency services burden reduction · Search & rescue cost avoidance · Underserved geographic populations.

## §4 Survival Core Requirements (Immutable, offline-first)

Survival Core invariants — must hold AT ALL TIMES:
- **Offline-first** — no connectivity dependency under any condition
- **Deterministic** — same input → same output every time
- **Non-adaptive** — no behavioural variation based on telemetry / density / inference
- **Non-inferential** — no predictive modelling / algorithmic confidence scoring
- **Non-escalatory** — no automated escalation logic of any kind
- **Crash-survivable** — Survival Core data must persist through unexpected termination
- **Firebase-independent** — Survival Core data persistence may NOT depend on any cloud service

### Survival Core components (§4.1)

| Subsystem | Function |
|---|---|
| Offline navigation engine | Point-to-point + turn-by-turn guidance without connectivity |
| BackTrack™ breadcrumb logging | Continuous, immutable, crash-survivable position logging |
| HazTrack™ overlays | Cached authority-origin hazard event display |
| SOS (≤2 tap) | Emergency state logging + distress breadcrumb activation |
| Deterministic TTL logic | Fixed-threshold data lifecycle management |
| Local logging | Immutable, crash-survivable event record for all safety-critical events |

### §4.1.1 Navigation requirements
**Must:** Fully offline within all downloaded regions · Warm start ≤3s · Cold start ≤6s · Initial map tile render ≤2s · Overlay toggle ≤1s · Point-to-point + turn-by-turn from downloaded bundle · Heading confidence display from GNSS.
**Must not:** Auto-reroute to any data signal/hazard/state · Recalculate active route autonomously · Modify breadcrumb capture based on nav state/speed · Imply impassable/closed/dangerous from system data alone · Introduce any server/cloud dependency for route execution.

### §4.1.2 Declarative Hazard Routing Constraint (5-step consent model)
Where HazTrack™ overlay intersects active route:
1. **Detect intersection** — passive, local geometry only · no connectivity
2. **Surface notification** — calm, non-alarming · no alarm language
3. **Offer alternate route** if available — must not pre-apply/activate
4. **Await user decision** — 3 options: accept alternate · dismiss · continue original · no countdown · no default action · no auto-selection
5. **Confirm and log** — explicit confirmation before any route change · log immutable · confirmation step counts in ≤2 tap SOS rule

### §4.1.3 Breadcrumb Capture Standard (Dual-Trigger Logic)
- **Standard mode:** 15m distance OR 20s time — whichever first
- **Distress mode (SOS active):** 5m distance OR 5s time — whichever first

**All breadcrumbs must be:** Immutable · Ordered (strict chronological) · Local-only in Phase 1 · Crash-survivable · Firebase-independent.

> **CONSTRAINT:** Capture intervals are fixed and deterministic. Cannot be dynamically altered by any system component, Experience Layer feature, navigation state, or telemetry signal. **BackTrack™ retrace trails must render in ≤3s** of user initiation. BackTrack™ is deterministic recall — NOT adaptive routing.

### §4.1.4 HazTrack™ Feed Failure Behaviour
Feed unavailability is PRIMARY operating condition, not edge case.

| Condition | Required behaviour |
|---|---|
| Feed unavailable | Cached overlay remains visible · freshness indicator updates · NO alert/error/notification |
| Feed timeout or error | Silent failure · freshness indicator only |
| Cached data present | Display with freshness indicator · never suppress due to feed failure |
| No cached data in region | Overlay empty · MUST NOT imply all-clear/safe |
| Feed resumes | Updates silently · NO notification |

### §4.1.5 Hazard Governance Inheritance
HazTrack™ must comply at all times:
- Ingest **authority-origin data sources only**: BOM, AFAC member agencies, state/territory emergency services
- Apply deterministic TTL · no adaptive TTL behaviour
- NOT alter routing decisions or auto-reroute in response to any hazard event
- NOT change/influence difficulty colour grading on any TrackIQ™ track
- NOT influence breadcrumb capture interval/fidelity
- NOT trigger automated escalation logic or SOS pathways
- NOT imply surveillance, monitoring, or real-time agency oversight

> **Cross-system acceptance:** A HazTrack™ feed event of ANY type — live or cached, any severity, any category — MUST produce **zero change** in TrackIQ™ difficulty grade, verification shield state, or scoring output. Verified by integration testing with live + cached feed injection.

### §4.1.6 SOS Discipline & User Feedback Model

> "SOS is the most consequential interaction in trackaroo®. Its behavioural model is non-negotiable and must be satisfied from every application screen state without exception."

SOS must:
- Be accessible within **≤2 taps from ANY active app screen state** (lock screen excluded)
- Function fully offline with **zero network dependency**
- Generate **immutable local log** immediately on activation (timestamp, coordinates, device state)
- UI response within **≤1 second of final confirmation tap**
- Trigger distress breadcrumb mode immediately (5m / 5s)
- Remain **device-resident** — no external transmission in Phase 1

SOS must NOT:
- Transmit any data externally in Phase 1
- Simulate dispatch or imply rescue services notified
- Escalate or trigger automated external actions
- Use alarm language in any confirmation or feedback state

**SOS feedback elements (every one is a discrete acceptance criterion):**
| Element | Standard |
|---|---|
| Emergency state logged | Confirmation logged locally with clear signal |
| Timestamp | Exact UTC timestamp at activation moment, in user's local time |
| Coordinates | Exact coordinates in format user can communicate to others |
| Distress mode active | Confirmation 5m/5s capture is now operating |
| Phase 1 transmission limitation | Calm statement: external transmission NOT available in Phase 1; available Phase 2 via satellite |
| User responsibility | Clear: user must seek help through available means — system has NOT contacted anyone |

> **CONSTRAINT:** SOS-triggered distress breadcrumb mode activates immediately upon SOS confirmation **for all users on all tiers — including Free tier**. No tier boundary may restrict, delay, or degrade SOS behaviour. SOS is Survival Core, tier-agnostic.
>
> The tier-gated feature is **manual distress density recording** — user's ability to enable elevated capture without SOS. That is Experience Layer, Plus/Pro only. These are SEPARATE features.

ESF-5026 governs SOS liability boundary. ESF-5026 legal review = named release gate.

### §4.1.7 Local Logging Requirements
Local log = immutable evidentiary record for all safety-critical events. Survival Core component.

Must record:
- SOS activation/deactivation events (timestamp, coordinates, device state, battery level)
- App startup/shutdown
- Connectivity state changes
- HazTrack™ route intersection notifications (timestamp, feed type, user decision)
- TrackMate™ group join/leave events (timestamp, anonymised member ID)

Must:
- Immutable — no modification/deletion after initial write
- Crash-survivable
- **30-day minimum retention** from creation
- Stored independently of Firebase
- Readable by user on request (basic log viewer = Phase 1 requirement)

### §4.1.8 TTL Governance Delegation
- All TTL thresholds **fixed at build time** — not configurable at runtime
- No Experience Layer component may alter TTL threshold
- TTL must not vary based on connectivity/telemetry/data signal
- Adaptive TTL behaviour = named rejection trigger

### §4.1.9 Map Region Bundle Download Standards
Primary mechanism for establishing offline capability. User-initiated. Must meet:
- Completion verification before marking offline-ready
- Graceful recovery from interrupted downloads (resume/restart option)
- User-visible progress + ETA
- Storage availability check before initiation
- Integrity validation (checksum) before marking available
- Offline readiness indicator (which regions are downloaded + age)

## §5 Experience & Capability Layer Requirements

> All Experience Layer components MUST degrade gracefully when offline. Non-critical features may become unavailable but MUST NOT generate error states that interrupt Survival Core flows.

### §5.1 TrackMate™ — Full Phase 1 Requirements

TrackMate™ = peer-to-peer group comms + location module · "practical digital alternative to two-way radio" · NOT a monitoring system, rescue tool, or automated distress mechanism.

TrackMate™ must:
- Short-range peer-to-peer messaging + location sharing without network infrastructure
- Long-range comm for separated convoy members via **optional user-owned hardware**
- Operate without cellular/internet — no TrackMate™ function may create network dependency for any Survival Core flow
- Be **transport-transparent** — no manual transport selection required
- Sustain battery alongside active nav — must not materially compromise ≥10-hour endurance
- Support invite-only group creation + role-based membership
- **Hard-cap active group membership at 8 members**
- Maintain offline message queue
- Enforce CAL state flags: satReady, queueEnabled, offlineBeacon

#### §5.1.1 Transport Stack Acceptance Criteria
| Mechanism | Role/Tier | Range | Battery | Vendor response |
|---|---|---|---|---|
| BLE Mesh | Tier 1 Primary (continuous bg) | ~100m + multi-hop | Very low (bg, must not be material drain) | Flutter/Dart feasibility, SDK, battery |
| Wi-Fi Direct | Tier 1 Fallback (on-demand) | ~200m line-of-sight | Medium (on-demand only) | Auto-activation logic, battery |
| LoRa (optional peripheral) | Tier 2 Long-range (auto-detected) | 5–15km open / reduced complex | Very low at hardware level | Auto-detect GoTenna/Meshtastic, Flutter integration |
| Satellite | Phase 2 only | Global | TBD | CAL must architecturally accommodate, no Phase 1 artefacts |

Message delivery latency (indicative, vendor-proposable): direct BLE ≤3s · two-hop ≤8s.

#### §5.1.2 Member Cap & Group Health Envelope
Member cap at 8 — system must return clear non-alarming notification, NOT silently fail.

**Group Health Envelope** = **visual indicator ONLY**, not a monitoring/surveillance system. Must:
- Display simple binary: all members present in envelope / member outside expected envelope
- Derive from last-known position + elapsed time since last contact only — NO inference/prediction
- NOT imply monitoring/surveillance/automated distress assessment
- NOT trigger any escalation pathway or SOS flow
- Degrade gracefully when member positions unavailable

#### §5.1.3 LoRa Hardware Onboarding Disclosure (release-gated)
| Requirement | Standard |
|---|---|
| Hardware expectation disclosure | First-use onboarding clearly explains long-range needs compatible LoRa module; without it, TrackMate™ operates short-range only |
| Range context | Communicate ~100–200m short-range / 5–15km long-range with hardware |
| No LoRa detected first use | UI confirms short-range-only mode without alarm language |
| LoRa hardware detected | App acknowledges range extension, confirms active transport, updates visible range indicator |
| No implied long-range capability | UI must never present TrackMate™ as long-range tool to user with no LoRa hardware |

**WFD-5126 flag:** Four wireframe states required before TrackMate™ onboarding development: (1) first-use onboarding with LoRa hardware disclosure; (2) short-range-only mode; (3) LoRa detected + active; (4) LoRa disconnected mid-session.

#### §5.1.4 Presence & Ping Governance
- Event-triggered + deterministic
- Fallback ping max 120s interval during active session only
- No persistent scanning loops
- 2 missed presence intervals → shift member marker to Last Known state · NO escalation pathway · NO alarm notification

#### §5.1.5 TrackMate™ Prohibitions
- Adaptive bandwidth logic / telemetry-based transport weighting
- Monitoring posture (explicit/implied) — in any UI, copy, or behaviour
- Automated distress triggering from Group Health Envelope state or any signal
- Any transport mechanism introducing network dependency into any Survival Core path
- Wi-Fi Direct held continuously active as background
- Satellite transmission artefacts/references/implied capability in Phase 1

### §5.2 TrackIQ™ Deterministic Scoring
**Must:** Score from deterministic telemetry only: elevation change, surface type, gradient, distance · Apply OSM-5026 colour-coded difficulty system · Accept explicit user validation signals · Display verification shields per OSM-5026 freshness model (independent of difficulty colour).
**Must not:** Infer obstruction/closure from user behaviour/stopping · Predict route condition from any data signal · Auto-detect hazard from stopping behaviour/telemetry density · Escalate difficulty based on data richness/absence · Introduce AI classification · Alter difficulty in response to HazTrack™ event.

#### §5.2.1 Deterministic Telemetry Use
Permitted inputs: GPS, accelerometer, gyroscope, altimeter.
All telemetry must be: session-bound (not retained across sessions) · not weighted across users · not used for predictive modelling · not used for adaptive recalculation.

#### §5.2.2 Stop-Detection Prompt
- Triggered by fixed time or distance only (thresholds in FSD-5126)
- Rule-based — no inference about cause of stop
- Non-inferential, non-diagnostic, non-escalatory — neutral language only, user action only

#### §5.2.3 Overlay State Matrix — 8 valid states only
| # | Difficulty Layer | Verification Shield | PCR | Meaning |
|---|---|---|---|---|
| 1 | Difficulty known | No Shield | None | Track graded; not yet verified |
| 2 | Difficulty known | Grey Shield | None | Graded & verified; verification stale |
| 3 | Difficulty known | Gold Shield | None | Graded & verified; verification current+fresh |
| 4 | Difficulty unknown (Grey) | No Shield | None | No authoritative grade data; not yet verified |
| 5 | Difficulty known | No Shield | PCR active | Graded; not verified; active condition report |
| 6 | Difficulty known | Grey Shield | PCR active | Graded; stale verification; active PCR |
| 7 | Difficulty known | Gold Shield | PCR active | Graded; fresh verification; active PCR |
| 8 | Difficulty unknown (Grey) | No Shield | PCR active | Ungraded; active condition report |

> **CONSTRAINT:** Difficulty colour must NEVER represent freshness/verification state. Freshness shields must NEVER represent terrain difficulty. PCR markers must NEVER use shield icon system (LIR-04). No overlay layer may mutate any other — PCR must not alter difficulty/shield, HazTrack™ must not alter PCR (LIR-05). Three layers always independent.

#### §5.2.4 PCR (Point Condition Report) Framework
6 PCR categories: PCR-OBS (Obstruction), PCR-CLO (Closure), PCR-INF (Infrastructure), PCR-SRF (Surface), PCR-WTR (Water Crossing), PCR-HAZ (Localised Hazard).

Acceptance:
- Supersession-only resolution — no TTL expiry · no clock-based removal · any other mechanism = **RT-19**
- In-navigation submission — without leaving active map view
- PCR offline queue — locally queued when offline, sync on reconnection · never silently discarded
- No automatic PCR trigger — explicit user action only · automatic triggering = **RG-10**
- PCR marker visual compliance — calm/non-alarming · WCAG 2.1 AA · glove-compatible
- Layer independence — LIR-04 + LIR-05

### §5.3 Six Activity Archetype Modules
| Archetype | Primary environment | Key UX priorities |
|---|---|---|
| 4WD Explorers & Remote Long-Range Travellers | Outback, unsealed roads, multi-day | Track condition, POIs, fire/flood overlays, convoy support |
| Bushwalkers & Hikers | National parks, wilderness, alpine, multi-day | High-fidelity trail data, elevation, BackTrack™, First Aid |
| Mountain Bikers / eBike / Motorbike | Trail networks, fire roads, technical | TrackIQ™ grading, one-handed UX, glove-compatible SOS |
| Remote Professionals | Mining, resources, conservation, emergency services, agriculture | High reliability, connectivity clarity, TrackMate™, compliance logs |
| Fish & Hunt | Remote lakes, rivers, restricted access | Land tenure overlays, water feature detail, seasonal, discreet UI |
| Snow & Alpine | Ski resorts, backcountry snowfields, alpine | Reduced-visibility UI, avalanche overlays, high contrast, glove-compatible |

Archetypes may adjust: visual presentation, defaults, overlay relevance, onboarding flow.
Archetypes may NOT adjust: breadcrumb capture, SOS access/tap count, hazard ingestion, TTL logic, routing determinism.

### §5.4 First Aid Reference System
"First Aid Reference" = mandated feature name across all surfaces. Reference tool — NOT medical device. Based on ARC + St John Ambulance Australia. Universal Baseline = DRSABCD, primary survey, 6 core reference cards — **all tiers (Free/Plus/Pro), no paywall, no auth, no upgrade prompts, ≤2 taps from any state**, persistent disclaimer.

| Tier | Access | Content |
|---|---|---|
| Universal Baseline | Free/Plus/Pro | DRSABCD, primary survey, 6 cards |
| Clinical Depth Layer | Plus/Pro | Extended wound mgmt, anaphylaxis, hypothermia, heat stroke, snake bite (PIT), spinal |
| Professional Layer | Pro only | Professional incident log, START triage, controlled incident export |

**RELEASE GATE:** Two-stage clinical review: (1) Universal Baseline clinically cleared before Alpha; (2) All tier content clinically reviewed + approved before Beta-Ready.

### §9.1 Intelligence & Adaptive Logic Definitions
**Intelligence includes:** predictive modelling · statistical smoothing · behavioural trend inference · telemetry density weighting · cross-session learning.
**Adaptive logic includes:** dynamic interval modification (incl. breadcrumb intervals) · real-time scoring recalibration · behaviour-based routing mutation · any response varying based on inference/prediction/AI.

> Rule-based threshold triggers (e.g. stop-detection prompts) are deterministic fixed-threshold responses — they DO NOT constitute adaptive behaviour.

### §9.3 Phase 2 Scaffolding Scope Constraint
Phase 2 scaffolding in Phase 1 limited EXCLUSIVELY to:
1. **BackTrack™ Emergency Escrow data schema** (non-executable)
2. **CAL satReady transport flag** (declared but not activated)
3. **CAL satellite transport architectural pathway** (documented in FSD-5126, not executable)

NOT a general licence to pre-build Phase 2. Anything beyond requires explicit Project Director approval + separate costing.

### §9.4 WFD-5126 Build Gate
> No subsystem development may commence where WFD-5126 wireframe coverage for that subsystem is not complete and formally approved. Absolute constraint, no exceptions.
