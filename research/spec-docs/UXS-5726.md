## §1 Executive Summary & Behavioural Position

trackaroo® Phase 1 must:
- **Feel complete + confidence-inspiring** to users operating in fully offline state
- **Execute all critical flows without any connectivity dependency**
- Deliver competitive functionality across navigation, hazard awareness, emergency readiness
- Preserve survival-grade determinism in all safety-critical execution paths
- Avoid adaptive or inferential behaviour in all Phase 1 systems

> Safety is the **core pillar**. Experience & Intelligence Layer enhances — but **never mutates** — Survival Core logic. Separation immutable + enforced at every level of design, development, QA.
>
> **Clarification on adaptive behaviour:** Rule-based threshold triggers (e.g. stop-detection prompts) are deterministic fixed-threshold responses. They do NOT constitute adaptive behaviour. Adaptive behaviour refers EXCLUSIVELY to system responses that vary based on inference, prediction, telemetry density, or AI classification.

### §1.3 Liability Position & Scope Boundary

**What trackaroo® IS:**
trackaroo® is a **decision-support platform** for people travelling/operating in remote or low-signal outdoor environments. It supports human judgement. It does NOT replace human judgement, emergency services, or rescue systems. **Offline-first information tool — NOT emergency dispatch system.**

**What trackaroo® is NOT (absolute scope exclusions):**
- NOT a real-time emergency response or dispatch tool Phase 1
- NOT a monitoring/surveillance system — for user, group members, or any third party
- NOT a substitute for emergency services (Triple Zero / 000 in Australia)
- NOT an endorsement of safe conditions — trackaroo® **never** implies a route/location/situation is safe
- NOT a medical device — First Aid Reference is read-only reference, not diagnostic/therapeutic
- NOT a substitute for qualified first aid training
- **NOT a guaranteed satellite communication system in Phase 1** — no satellite transmission Phase 1
- NOT a substitute for formal WHS incident reporting obligations for Remote Professional archetype users

### §1.3.3 User Responsibility Model
trackaroo® operates on **user-sovereignty principle**. Decisions remain with the individual at all times. The platform provides information; the user acts on it. Reflected in ToS, Privacy Policy, all in-app copy touching hazard/SOS/emergency contexts.

### §1.4 Prohibited Claims Register
Maintained by Trackaroo Systems. **No delivery partner, legal advisor, content author, or vendor may use prohibited claims** in any user-facing, App Store-facing, or marketing copy. Prohibited claims specified in §1.4.1 (SOS & Emergency) · §1.4.2 (HazTrack™ Notifications) · §1.4.3 (First Aid Reference) · §1.4.4 (TrackMate™ Group Health Envelope) · §1.4.5 (General Platform).

## §3 The Five-Question Cognitive Hierarchy

> **Primary UX Acceptance Criterion for Phase 1.** Every design decision — navigation, hazard awareness, emergency readiness, group comms, first aid — must be evaluated against this framework.

| # | Question | Subsystem Responsibility | Failure Mode |
|---|---|---|---|
| **1** | **Where am I?** | Navigation Engine · Offline Map | Position uncertainty · map blank or stale |
| **2** | **Where am I going?** | Route Planning · Breadcrumb Trail | No active route · lost trail context |
| **3** | **What surrounds me?** | HazTrack™ · Overlay System | Hidden hazards · stale data not disclosed |
| **4** | **How do I get back?** | BackTrack™ · Return Navigation | Breadcrumb loss · return path unavailable |
| **5** | **How do I call for help?** | SOS Module (≤2 tap) | SOS inaccessible · buried in UI · delayed by network |

> **RELEASE GATE:** Any design decision impairing user's ability to answer any of these 5 questions **without deliberation** = failure against primary acceptance criterion. **All 5 must be answerable** in fully offline state · under stress · with gloves · low-light · on first use.

## §4 Dual-Layer Architecture & Behavioural Separation

### §4.1 Layer Definitions

| Layer | Components | Behavioural Rule |
|---|---|---|
| **Survival Core** | Nav engine, BackTrack™, HazTrack™ overlays, SOS, local logging, deterministic TTL logic | Offline-first · deterministic · non-adaptive · no network dependency · no mutation by E&I Layer |
| **Experience & Intelligence Layer** | TrackIQ™, TrackMate™, archetypes, visual overlays, First Aid Reference, companion website | May degrade gracefully offline · must NEVER block/delay/mutate any Survival Core flow |

### §4.2 Immutable Separation Rules
- E&I Layer must NOT modify/delay/conditionally suppress any Survival Core output
- No E&I Layer feature may introduce network dependency into any Survival Core path
- Survival Core must execute **identically** regardless of state of any E&I Layer component
- Scaffolded Phase 2/3 features within E&I Layer must be **inert, non-executable, confirmed through QA**

> **CONSTRAINT — Architectural Constraint:** Dual-layer separation = **safety requirement**, NOT design preference. Any architecture that blurs boundary — through shared state, conditional rendering, unified execution paths — is a **rejection trigger**.

## §5 Offline-First Supremacy

> **CONSTRAINT — Offline-First Rule:** Every Survival Core feature must be **fully functional in zero-connectivity state**. No Survival Core flow may degrade/block/prompt for connectivity. User must NEVER be presented with loading state, connectivity prompt, or feature unavailability message within Survival Core.

### §5.2 Connectivity State Communication
App must clearly + consistently communicate current connectivity to user. 4 states in UI at all times:
- **Fully offline** — no network, no satellite
- **Satellite placeholder only** — no cellular or Wi-Fi · Inactive in Phase 1
- **Cellular or Wi-Fi connected**
- **TrackMate™ mesh active** (peer-to-peer link established)

Connectivity status displayed via **CAL status indicator**. Indicator must be visible at all times in navigation view without user action.

### §5.3 Data Freshness & Offline Confidence
- Every data layer displayed must carry freshness indicator
- Freshness disclosure = calm, non-alarming language
- Timestamp of most recent data sync accessible within ≤2 taps from main nav view
- Stale data must NEVER be presented without clear freshness indicator

## §6 Survival Core — Behavioural Constraints

### §6.1 Navigation Engine
- Must render maps in fully offline state using pre-downloaded vector tiles
- Must display user's current GNSS position without any network request
- Must NOT present "no map data" state within a pre-downloaded region
- Must NOT auto-reroute in response to any hazard event (HazTrack™ or otherwise) without explicit user initiation
- Route recalculation requires explicit user action — never silent or automated

### §6.2 Breadcrumb Logging
- Log breadcrumb points deterministically at configured interval
- Breadcrumb logging must NOT be suspended/degraded/affected by any E&I Layer state
- Breadcrumb data persisted locally using storage **independent of Firebase Firestore**
- Breadcrumb TTL logic deterministic + fixed — no adaptive retention

### §6.3 Local Logging & TTL
- All Survival Core data (position history, breadcrumbs, SOS events, session records) persisted locally
- Local storage must NOT be managed by or dependent on Firebase Firestore
- TTL rules deterministic + explicitly documented in FSD-5126
- Data must survive app crash, force close, device restart without loss

## §7 SOS Access & Emergency Preparedness

### §7.1 The ≤2 Tap Rule

> **CONSTRAINT — Absolute SOS Constraint:** SOS function must be accessible within **2 taps from any screen** in the application at any time. **Absolute, non-negotiable, cannot be relaxed** by any downstream document, delivery partner proposal, or design decision. Violation = rejection trigger.

### §7.2 SOS Trigger Discipline
- Accessible with **gloved hand** using min 44×44pt touch targets (60×60pt recommended for SOS control)
- Functions in fully offline state without any network dependency
- Functions in fully dark or bright screen — high contrast required
- Must NOT require device unlock, authentication, or app navigation to initiate
- Confirmation step (to prevent accidental activation) must NOT add more than 1 additional tap — total remains ≤2

### §7.3 SOS Feedback Language
- Must NEVER imply help has been dispatched
- Must NEVER imply rescue is underway
- Must confirm what action HAS occurred (e.g. "SOS signal logged", "position recorded")
- Must include timestamp + coordinates of SOS event
- **Reassurance-first framing**

> All SOS feedback copy, logging behaviour, any implied emergency service interaction must conform exactly to approved language + behaviour specified in **ESF-5026**. SOS liability boundary + user responsibility model owned by Trackaroo Systems. No modification permitted without written PD authorisation.

### §7.4 Non-Transmission & Non-Dispatch Posture (Authority: ESF-5026)
Behavioural constraints derived from ESF-5026 (absolute):
- **Zero transmission** — SOS + all Survival Core paths generate zero outbound packets Phase 1
- **Zero escalation logic** — no pathway for device to escalate to emergency services or external systems · trackaroo® is NOT a dispatch system Phase 1
- **SOS event log immutability** — all SOS log entries write-once, tamper-resistant, stored locally independent of Firebase
- **Phase 2 Escrow pathway** — BackTrack™ Emergency Escrow architectural pathway scaffolded in Phase 1 with **no structural barrier** to Phase 2 uplift, but must remain inert, non-executable, confirmed through TQP-5026 validation

## §8 HazTrack™ Hazard Awareness System

### §8.1 Behavioural Rules
- Render from locally cached data in fully offline state
- Notifications use calm, non-alarming language at all times
- Notifications must NEVER instruct user to take specific action
- Must always disclose freshness of hazard data displayed
- System must NEVER auto-reroute in response to hazard event
- Hazard overlays must NOT visually dominate base navigation map

### §8.2 Notification Tone
> Governing principle: **informing, not directing**. User retains full decision authority at all times.
>
> Hazard notification copy, data freshness disclosures, any implied duty-of-care must conform exactly to approved language in **ESF-5026 + HFG-5026**. Liability posture owned by Trackaroo Systems.

### §8.3 Overlay Data Systems & Visual Semantics (Authority: OSM-5026)
**OSM-5026 = sole + exclusive authority** for all overlay visual semantics. 4 independent data systems: Hazard Feed Data, Environmental Feed Data, Track Verification Data, Point Condition Report (PCR) framework. All visual encoding, TTL thresholds, layer independence rules, PCR governance defined in OSM-5026.

### §8.4 Point Condition Report (PCR) Behavioural Rules
- User-submitted discrete physical-state reports (fallen tree, locked gate, washed-out crossing) — **separate data layer** from Track Verification Shield · must NEVER share visual encoding
- Resolved by **supersession only** — no TTL expiry, no automated clock-based removal · remains active until subsequent user confirms cleared/submits new PCR same point/original reporter self-resolves
- Submission UI must allow report without navigating away from active map — interaction completable within navigation view · **6 categories** in OSM-5026 §10.2 (PCR-OBS/CLO/INF/SRF/WTR/HAZ)
- Markers use calm, non-alarming iconography · WCAG 2.1 AA contrast · glove-compatible min touch targets · must NOT obscure difficulty grade or track verification shield

## §9 BackTrack™ Return Navigation
- Must be accessible within **3 taps** from any navigation view
- Must reconstruct return path from locally stored breadcrumb data without any network request
- Return path clearly distinguished from forward route on map
- Must remain functional if breadcrumb trail is partial/gapped — displaying available portion with clear gap indication
- Must NOT auto-activate — always user-initiated

---

## §10 TrackIQ™ Intelligence Layer
- Scoring must be **rule-based + deterministic** — no AI inference, predictive modelling, or telemetry-based weighting
- Outputs must be **advisory only** — never trigger automated actions or modify route behaviour
- Must degrade gracefully if operating offline with incomplete data — displaying only scoring dimensions for which data is available
- Must NEVER block navigation access — always an additional layer, never a prerequisite

## §11 TrackMate™ Group Communications Module

### §11.1 Product Definition
TrackMate™ = trackaroo®'s peer-to-peer group comms + location module. Full Phase 1 deliverable. Best understood as practical digital alternative to two-way radio for groups operating in remote terrain.

### §11.2 Capability Goals
Short-range comms (P2P, no infrastructure) · Long-range comms (convoy members separated · may require optional LoRa hardware) · Battery sustainability (must not compromise ≥10-hour operation per BPS-5126) · Offline-first (no cellular/internet dependency) · **Transport transparency (no manual transport selection required by user)**.

### §11.3 Reference Transport Stack (informed reference — NOT closed spec)
| Tier | Mechanism | Range | Battery |
|---|---|---|---|
| Tier 1 Primary | BLE Mesh | ~100m device-to-device · multi-hop | Very low — continuous bg |
| Tier 1 Fallback | Wi-Fi Direct | ~200m line-of-sight | Medium — on-demand only, must NOT be held continuously |
| Tier 2 Long Range | LoRa (optional peripheral) | 5–15km open terrain | Very low |
| Phase 2 | Satellite relay | Global | TBD |

### §11.4 Absolute Prohibitions
- No adaptive bandwidth logic or telemetry-based transport weighting
- No monitoring posture — explicit or implied
- **No automated distress triggering from Group Health Envelope**
- No transport mechanism may introduce network dependency into any Survival Core path
- Wi-Fi Direct must NOT be held continuously active

### §11.5 LoRa Hardware Disclosure
Hardware dependency for long-range capability must be **disclosed to users at first use**. UI must NEVER imply long-range capability with no LoRa hardware connected. 4 WFD-5126 wireframe states required (first-use onboarding · short-range only mode · LoRa hardware detected and active · LoRa disconnected mid-session).

## §12 First Aid Reference

### §12.1 Phase 1 Scope
First Aid Reference = confirmed Phase 1 deliverable. Authority: FRM-5126 (internal governance, not vendor-distributed).

#### §12.1.1 Tier Content Model
- **Universal Baseline (all tiers, free)** — DRSABCD sequence, primary patient survey, uncontrolled bleeding, shock recognition, burns reference, fracture and spinal immobilisation reference, choking sequence. **Fully pre-loaded at install. Accessible within 2 taps from any screen state. No paywall, no tier restriction.**
- **Plus Tier — Clinical depth:** extended wound mgmt, anaphylaxis protocol, hypothermia, heat exhaustion/stroke, snake bite (PIT), extended spinal mgmt, triage timer + step-tracking, offline event log, activity-specific safety extensions
- **Pro Tier — Professional layer:** full activity-specific safety extension library, professional incident log (exportable), START triage mass-casualty sequence, altitude sickness protocol, dive injury + decompression sickness reference

#### §12.1.2 Naming Discipline
Feature name in all user-facing + App Store copy = **"First Aid Reference"**. Internal docs may use "First Aid Kit" as shorthand. **Prohibited terms** in external copy: "First Aid Kit", "medical system", "triage system", "clinical guidance", "medical tool", "diagnose", "treatment protocol", "emergency medical system".

> **CONSTRAINT — Clinical Review Required:** All First Aid Reference content + UI copy must be reviewed + approved by **qualified clinical reviewer commissioned by Trackaroo Systems** before publication. No content may be exposed to any user without written PD clearance. Authority: PRD-5126 RT-12.

### §12.3 Mandatory Disclaimer Language
Approved disclaimer must appear on **every screen** within First Aid Reference. Non-negotiable, must NOT be shortened/paraphrased/hidden behind scroll.

> *"This is reference information only. It is not professional medical advice, diagnosis, or treatment. Always seek the assistance of qualified emergency services or a medical professional where available. In an emergency, call Triple Zero (000) in Australia, or your local emergency number."*

International markets substitute last sentence: *"In an emergency, contact your local emergency services."*

### §12.4 Clinical Review Gate
Clinical review of Universal Baseline content **required before Alpha gate**. Full tier content clinical review **required before Beta-Ready gate**. Reviewer must be qualified wilderness paramedic or emergency nurse. Validates compliance with ARC Guidelines + St John Ambulance Australia published standards.

## §14 Connectivity Status Communication

### §14.1 CAL Status Indicator
CAL status indicator must be **visible at all times in navigation view**. Must communicate current connectivity without requiring any user action.
- CAL flags `satReady`, `queueEnabled`, `offlineBeacon`, `partialSignal` enforced across all comms modules
- Indicator must NOT require data connection to display — reflects device state, NOT server state
- **`satReady` remains placeholder state Phase 1 only** — must NOT imply active satellite transmission
- State changes must be reflected within **2 seconds** of actual connectivity change

## §15 Data Freshness & Transparency
- Every data layer must carry freshness indicator
- Freshness communicated via 4 OSM-5026 data systems (Track Verification Shield, Hazard Feed Freshness, Environmental Feed Freshness, PCR framework). **OSM-5026 = sole authority** for visual encoding, TTL thresholds, display rules
- Data currency must NEVER be misrepresented — stale data MUST be disclosed
- User must be able to determine age of any displayed data within 2 taps from nav view
- Freshness indicators use calm, informative language — never alarming

## §16 Onboarding & First-Use Experience
- Onboarding must complete archetype selection **before** presenting primary navigation view
- Onboarding must request **map region download** before allowing navigation
- Onboarding must set connectivity expectations — offline-first posture explicitly communicated
- For TrackMate™: hardware dependency for long-range disclosed at onboarding (LoRa wireframe state)
- Onboarding must NOT be skippable for safety-critical setup steps
- All onboarding copy: calm, non-alarming language consistent with informing-not-directing posture
