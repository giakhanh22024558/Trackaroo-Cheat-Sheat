## §4 Offline Architecture Model

> Offline-first integrity = non-negotiable Survival Core requirement. Mapping system must be fully functional within all downloaded map regions without any network dependency.

### §4.1 Region Download Model
Phase 1 = region-based offline download. Vendor must define + document:
- Bounding box or polygon selection method
- Maximum region size (area + tile count)
- Included zoom levels per region pack
- Elevation tile inclusion policy
- Satellite imagery inclusion policy (if applicable)
- Estimated storage size per representative region

### §4.2 Tile Budget Controls
Vendor must define + document:
- Maximum offline tile count per device
- Maximum zoom depth allowed in offline packs
- Storage warning thresholds + user notification behaviour
- Multi-region coexistence behaviour
- Version control mechanism for offline packs

> **CONSTRAINT:** Automatic background tile polling **prohibited**. System must NOT silently increase storage/tile costs. Satellite imagery must NOT silently increase storage or data costs.

### §4.3 Partial Download Recovery
- Detect incomplete/interrupted downloads
- Resume from interruption point safely
- Validate pack integrity before marking offline-ready
- Prevent corrupted/incomplete packs from producing degraded map state

### §4.4 Mapbox Tile Cost Governance
Mapbox offline tile usage incurs licensing costs. Vendor must:
- Document tile cost model for all offline pack configs
- Declare max tile count ceiling per device + cost implications
- Propose cost control safeguards for tile consumption at scale
- Flag any config that could result in unbounded/unpredictable tile cost at user scale

## §5 Terrain & Elevation Pipeline

Vendor must specify in full:
- DEM (Digital Elevation Model) source
- Elevation data resolution
- Hillshade rendering method + engine
- Contour source + resolution
- Maximum pitch angle for 3D tilt (recommended ≤45°)
- Low-end device fallback rules

Terrain rendering must not drop below performance envelope (§8). Fallback rules must degrade gracefully — no device shall experience catastrophic render failure.

## §6 Satellite Governance

Satellite imagery integration must be governed with explicit controls. Vendor must define:
- Satellite imagery provider
- Resolution caps + quality tiers
- Zoom cutoff limits for satellite layer
- Offline satellite inclusion policy
- Maximum satellite imagery download size per region
- Cost control safeguards to prevent unbounded satellite data consumption

Satellite transmission capability (TrackMate™ relay, BackTrack™ Emergency Escrow sync) = **Phase 2 deliverable**. NO satellite SDK references/transmission scaffolding/implied capability in Phase 1 builds. CAL must be architected to accept satellite as future transport tier — but pathway must remain non-executable Phase 1.

## §7 Overlay Interaction Rules

Basemap must support overlay layers rendered in-app without visual conflict. Overlay colour definitions + semantic rules governed exclusively by **OSM-5026**. This section governs technical interaction between basemap + overlay layers.

### §7.1 Overlay Layers (must support without basemap conflict)
- Difficulty colour system (Easy/Moderate/Difficult/Extreme/Unknown — OSM-5026 §6)
- Track Verification Shield states (No/Grey/Gold — OSM-5026 §7)
- Hazard Feed Freshness Indicator (OSM-5026 §8)
- Environmental Feed Freshness Indicator (OSM-5026 §9)
- Point Condition Report (PCR) markers (OSM-5026 §10; icon design = MAS-5126 §18)
- Active navigation route line (OSM-5026 §5A.2)
- Breadcrumb trail, incl. BackTrack™ active rendering (OSM-5026 §5A.3)

### §7.2 Vendor Technical Requirements
- Z-index hierarchy for all overlay layers — governing order: **basemap → difficulty colour → hazard overlay → breadcrumb trail → active navigation route line → PCR markers** (OSM-5026 §5A.5)
- Opacity rules for overlay rendering
- Overlay conflict resolution — PCR markers must NOT obscure difficulty grade or shield state on track line
- Confirmation no routing mutation introduced via overlay interaction
- Touch-target enforcement for PCR markers: minimum 44×44pt; recommended 60×60pt

> **CONSTRAINT:** Basemap must NOT dynamically adapt/change colour/modify behaviour based on overlay state. Difficulty colour must NEVER represent freshness or verification state. Freshness shields must NEVER represent terrain difficulty. PCR markers are separate visual layer + must NOT modify/replace difficulty colour or shield state. **Layers always independent. Violation = governance breach.**

## §8 Performance Envelope (minimum requirements)

| Performance Dimension | Target | Authority |
|---|---|---|
| Frame rate — idle/static map | **60 fps** | MAS-5126 §8 |
| Frame rate — sustained interactive (pan, zoom, layer toggle) | **≥50 fps** | MAS-5126 §8 |
| Frame rate — absolute floor | **≥45 fps** — must NOT drop below for more than 0.5s under any condition on validated device matrix | MAS-5126 §8 |
| 3D tilt frame rate | No frame drop during tilt operations | MAS-5126 §8 |
| Map tile initial render | ≤2 seconds from screen entry on all validated devices | PRD-5126 |
| Overlay layer toggle | ≤1 second response time | PRD-5126 |
| PCR marker rendering | No measurable render latency above baseline overlay toggle | MAS-5126 §8 |
| Memory footprint | Within platform norms for navigation-class apps | MAS-5126 §8 |
| Battery draw | Within navigation envelope — **BPS-5126 = governing authority** | BPS-5126 |

> **CONSTRAINT:** Maximum **3 concurrent map layers active simultaneously**. Exceeding this limit under any condition on validated device matrix = **performance governance failure**.
>
> No translucent layers permitted when hillshade is active. Translucent layers stacked over hillshade terrain = prohibited under all rendering conditions.
>
> Authority: CR02 Q4, 31 March 2026.

## §9 Failure Behaviour (deterministic, non-alarming)

| Failure Condition | Required Behaviour |
|---|---|
| Offline pack missing | Inform user calmly; prompt download. Must NOT leave user with blank screen without explanation |
| Tiles fail to render | Display degraded state with calm status indicator. Must NOT crash/freeze |
| Device storage full | Warn before failure. Prevent corrupted state. Do NOT auto-delete without user confirmation |
| Satellite imagery unavailable | Fall back to vector basemap silently. NO user-facing error required unless satellite was explicitly requested |
| Tile quota exceeded | Inform user calmly. Prevent silent cost escalation |
| Map service unreachable | Use cached tiles. Inform user of offline-only mode. Must NOT imply loss of navigation capability within downloaded regions |
| PCR sync failed (offline) | PCR submission queued locally. Inform user of queued state with offline indicator. Must NOT block submission or lose data |

> **CONSTRAINT:** All failure states must be calm, non-alarming, deterministic. No failure state may produce alarm-style language, urgency cues, or imply nav capability lost within downloaded region. **Failure states are informational, NOT directional.**

## §10 Map State Persistence

Mapping system must persist user state across app restarts:
- Last viewport position + zoom level
- Map orientation
- Active style (TRK_Light_Base / TRK_Dark_Survival / Satellite)
- Active overlay visibility state
- Breadcrumb trail visibility

> **CONSTRAINT:** Map state persistence operates in Experience & Intelligence Layer. Persisted state must NOT create dependency on Survival Core. Survival Core must initialise deterministically regardless of restored Experience Layer state. **No persisted map state may condition/delay/mutate any Survival Core function.**

## §13 Strategic Visual Identity

> "trackaroo® mapping must feel engineered, confident, premium, and calm. Must visually outperform competitors while maintaining deterministic integrity."

| Identity Principle | Implementation Meaning |
|---|---|
| **Terrain First** | Terrain must feel dimensional, immersive, engineered. Hillshade + contours = primary visual elements |
| **Clarity Under Stress** | Map must remain legible in high-glare, low-light, movement conditions. Contrast + hierarchy > aesthetics |
| **Restraint Over Saturation** | Premium = minimal colour noise. NO saturated primaries in basemap. Colour reserved for INFORMATION, not decoration |
| **Visual Calm** | NO flashing/blinking/animated features/implied urgency through basemap styling. Basemap = static visual infrastructure |
| **Professional Instrument** | Must present as **professional expedition instrument, not recreational social app**. Default Mapbox styling PROHIBITED |

## §15 Colour System

> **CONSTRAINT — Signal Orange RESERVED Colour (Non-Negotiable):** HEX `#FF5C00`. Reserved EXCLUSIVELY for Extreme difficulty overlay layer (OSM-5026 §6). **Basemap must NEVER use Signal Orange** for any basemap feature/label/UI element. Non-negotiable governance constraint shared with OSM-5026.

Basemap colour palette must avoid conflict with overlay colours (OSM-5026):
- Green `#4CAF50` — Easy difficulty
- Blue `#1976D2` — Moderate difficulty
- Black `#212121` — Difficult difficulty
- Signal Orange `#FF5C00` — Extreme difficulty (RESERVED)
- Grey `#9E9E9E` — Unknown/Ungraded

And PCR indicator colours (OSM-5026 §10.4):
- Amber `#FFA000` — Active PCR, unconfirmed
- Red `#D32F2F` — Active PCR, confirmed by multiple reporters

### §15.1 Light Mode Palette
- Background: warm neutral, sand-toned, low saturation
- Landcover: muted greens, desaturated
- Water: soft blue-grey
- Urban: light neutral, low contrast
- Contours: subtle mid-neutral
- Hillshade: soft directional shadow
- NO high-saturation primaries permitted

### §15.2 Survival Dark Mode Palette
- Background: deep charcoal (near `#121212`)
- Landcover: cool-toned muted
- Water: deep slate-blue
- Contours: soft light neutral
- Hillshade: higher contrast than light mode, but NO crushed blacks
- NO harsh black valleys

## §16 Terrain System Requirements

### §16.1 Hillshade
- Directionally consistent across map
- Balanced contrast — neither underlit nor overlit
- NO harsh black valleys
- NO overexposed ridgelines
- Dark mode: higher contrast than light, but no crushed blacks

### §16.2 Contours
- Hairline minor contours
- Slightly stronger index contours for readability
- Fade at low zoom — must NOT clutter driving zoom
- Must NOT compete visually with road hierarchy at driving zoom

### §16.3 Elevation Enhancement
Optional subtle elevation tint permitted. If applied:
- Must NOT reduce label readability
- Must NOT introduce saturated colour into basemap
- Must remain visually subordinate to hillshade

## §17 Road & Track Hierarchy

Vendor must implement zoom-dependent width scaling for all road/track types. Hierarchy must be visually intuitive without legend reference.

| Feature Type | Visual Treatment | Zoom Behaviour |
|---|---|---|
| Primary sealed roads | Strong neutral with casing | Visible all zoom levels; max weight at low zoom |
| Secondary sealed roads | Reduced contrast, lighter casing | Visible from mid zoom |
| Local sealed roads | Minimal emphasis | High zoom only |
| Unsealed / 4WD tracks | Distinct from sealed; dashed or earth-toned pattern | Must remain visible at driving zoom |
| Walking / MTB trails | Thin stroke, high contrast ratio | High zoom only; not visible at driving zoom |

> Visual distinction between sealed roads + unsealed/4WD = critical navigation requirement for 4WD Explorer archetype. Must be unambiguous without legend reference.
