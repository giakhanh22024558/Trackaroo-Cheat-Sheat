### §4.3 Comms Abstraction Layer (CAL)
CAL = sub-component of TrackMate™. Abstracts all transport from user + Survival Core paths. **No CAL operation may introduce a network dependency into any Survival Core execution path.**

| CAL state flag | Phase 1 status | Definition | Authority |
|---|---|---|---|
| `satReady` | DECLARED — NOT active | Boolean · false all Phase 1 builds · No satellite SDK · Not activatable · Signals Phase 2 satellite pathway readiness only | FSD-5126 §6.1.1 |
| `queueEnabled` | ACTIVE | Boolean · true whenever TrackMate™ session active · Offline message queue accepting outbound | FSD-5126 §6.1.1 |
| `offlineBeacon` | ACTIVE | Boolean · true when BLE Mesh active + broadcasting presence | FSD-5126 §6.1.1 |
| `partialSignal` | ACTIVE | Boolean · true when partial signal detected · Drives degraded state indicator · No automated recovery/escalation | FSD-5126 §6.1.1 · WFD-5126 §6 |

|---|---|---|---|---|

## §5 Architectural Zone Reference

### §5.1 Immutable Separation Boundary

> Safety requirement, not design preference. Enforced architecturally + validated through QA.
>
> **IMMUTABLE SEPARATION RULE:** Experience & Intelligence Layer must NOT modify, delay, conditionally suppress, or otherwise mutate any Survival Core output or execution path. Survival Core must execute identically regardless of state of any E&I Layer component. Any implementation that blurs this boundary — through shared state, conditional rendering, or unified execution paths — is a rejection trigger.
>
> Authority: UXS-5726 §4.2 · PRD-5126 §4 · VGD-5126 §4.1

### §5.2 Firebase Isolation Model

| Data category | Permitted storage | Firebase permitted? | Cloud upload Phase 1? |
|---|---|---|---|
| Breadcrumb data | Local · Firebase-independent · AES-256 · Crash-survivable | **NO — prohibited** | **NO — prohibited** |
| SOS event log | Local · Firebase-independent · Immutable after write | **NO — prohibited** | **NO — prohibited** |
| Local event log | Local · Firebase-independent · 30-day min retention | **NO — prohibited** | **NO — prohibited** |
| TrackIQ™ scoring data | Local cache | YES — non-critical only | Deferred — offline sync |
| TrackMate™ message history | Local cache | YES | Deferred — offline sync |
| HazTrack™ overlay cache | Local cache tied to map region bundle | YES | Deferred — offline sync |
| User preferences / archetype | Firestore offline persistence | YES | Deferred — offline sync |
| PCR data | Firestore offline persistence · Offline queue | YES | Queued — syncs on connectivity |

> **RECONCILIATION PROHIBITION:** A reconciliation engine that could mutate locally captured breadcrumb, SOS, or event log data is absolutely prohibited. Cloud sync must NOT overwrite local Survival Core data under any condition. Firebase-independent Survival Core storage is always authoritative.
>
> Authority: CDG-5126 §6.

### §5.3 Phase 2 Scaffold Zone — Inert
Phase 1 contains exactly 3 permitted Phase 2 scaffolds. Inert + non-executable + confirmed through TQP-5026 validation.

| Scaffold | Status | Present | Prohibited |
|---|---|---|---|
| BackTrack™ Emergency Escrow data schema | Non-executable — schema present only | Data model structured for forward migration · NO transmission logic | Any transmission reference · Satellite SDK reference · User-visible Escrow artefact |
| CAL satReady flag | Declared false — NOT active | Boolean declared in CAL state schema · Set to false · NOT activatable | Satellite SDK · Triggerable transmission logic · User-visible satellite reference |
| CAL satellite transport architectural pathway | Documented only — NOT executable | Extensible CAL interface in FSD-5126 · NO executable satellite transport path | Any compile-time branching that could activate transmission logic in Phase 1 production build |

> **PHASE 2 SCAFFOLD BOUNDARY:** Phase 2 scaffolding in Phase 1 limited exclusively to the 3 items above. A Phase 2 scaffold accessible/visible/executable in a Phase 1 build = automatic rejection trigger. Authority: PRD-5126 §9.3 · UXS-5726 §18.

## §7 Vendor Architectural Compliance Obligations

VGD-5126 §6.1 requires vendors to supply the following architectural compliance artefacts at each release gate. AOD-5026 = reference model.

1. **High-level architecture diagram** — annotated to show Survival Core isolation boundaries
2. **Deterministic state transition matrix** — covering all Survival Core state transitions
3. **Offline-first execution explanation** — how full Survival Core functionality is maintained without network dependency
4. **Module isolation mapping** — confirming E&I Layer cannot mutate Survival Core behaviour
5. **Breadcrumb classification confirmation** — confirming Local-Only + Non-Syncable
6. **CAL architecture documentation** — transport abstraction, 4 state flags, priority logic
7. **Firebase isolation confirmation** — Survival Core data stores architecturally isolated from all Firebase write contexts
8. **Phase 2 scaffold confirmation** — 3 permitted scaffolds present, inert, non-triggerable

> **ARCHITECTURAL COMPLIANCE GATE:** All 8 artefacts above are mandatory for Discovery → Alpha gate progression. Development may not commence on any subsystem without architectural compliance artefacts for that subsystem being accepted by the Project Director.
>
> Authority: VGD-5126 §6.1 · PRD-5126 §9.4
