## Role in Authority Stack

**Priority 7** (Tier 2 specialist). Subordinate to UXS-5726 / PRD-5126 / FSD-5126 / TQP-5026 / WFD-5126 / VGD-5126. References OSM-5026 (sole authority for overlay visual semantics + PCR framework definition), FRM-5126 (First Aid Reference content tier model), ESF-5026 (SOS + satellite prohibition).

> **CONSTRAINT (Executive Summary):** "The cloud is secondary and supplementary. NEVER a dependency for any Survival Core function. Survival Core data — breadcrumbs, SOS logs, local event log — must be persisted independently of Firebase and any cloud service. A Phase 1 build that requires cloud connectivity for any Survival Core operation is **architecturally non-compliant regardless of any other merit**."

CDG-5126 governs all data classification, persistence, cloud isolation, mutation prevention, and Point Condition Report data handling rules for Phase 1.

## §4 Local-Only — Non-Syncable Classification

> Highest data protection classification in Phase 1. Data carrying this classification must NEVER leave the device, NEVER be uploaded/synced/transmitted/mirrored to any cloud service, and NEVER be subject to any mutation after initial write.

### §4.1 Definition + Constraint
> **CONSTRAINT:** Any cloud pathway, sync logic, upload endpoint, or transmission pathway for Local-Only — Non-Syncable data — whether active or dormant — constitutes automatic non-compliance. Includes: Firebase cloud sync · Firebase real-time database · Firestore cloud write · third-party cloud storage · any network transmission endpoint for breadcrumb or SOS data. Presence of any such pathway in Phase 1 build = **named rejection trigger (RT-05 + RT-13 in PRD-5126)**.

### §4.2 Breadcrumb Data — Specific Rules
- Captured deterministically: standard 15m/20s; distress 5m/5s (PRD §4.1.3)
- All breadcrumbs immutable after capture — no post-capture modification, recalculation, re-weighting
- Strict chronological sequence always
- Local storage uses write-ahead log or equivalent crash-survivable mechanism
- Survive app crash, force close, device restart without data loss
- BackTrack™ reverse retrace reconstructs from local breadcrumb data only — no network request permitted (FSD-5126 §4.2)

### §4.3 Mutation Prohibition — Complete List (all phases)

Following operations **absolutely prohibited** on Local-Only — Non-Syncable data:

| Operation | Description |
|---|---|
| Merge | Combining breadcrumb sequences from multiple sessions or sources |
| Compaction | Reducing number of stored breadcrumb points by removing/consolidating |
| Reconstruction | Replacing captured points with algorithmically derived positions |
| Reordering | Changing sequence of breadcrumb points for any reason |
| Map-matching | Snapping captured coordinates to road or track geometries |
| Coordinate interpolation | Inserting estimated positions between captured points |
| Gap-filling | Adding points where captures missed due to any system state |
| Deduplication | Removing 'duplicate' points based on proximity or temporal closeness |
| Compression | Any lossy or lossless reduction of coordinate precision or point density |
| Conflict resolution | Resolving discrepancies between local + any external data source |
| Reconciliation | Synchronising local breadcrumb state against any external reference |
| Coalescing | Combining multiple nearby points into single representative point |
| Smoothing | Applying any algorithm to reduce perceived GPS noise or jitter |
| Correction | Adjusting coordinates to align with map data, prior positions, or any reference |

> **CONSTRAINT — INTEGRITY OVER OPTIMISATION:** No performance optimisation, storage efficiency objective, UX improvement goal, or vendor implementation preference may justify any mutation of Local-Only — Non-Syncable data. If optimisation requires mutation, the optimisation must be **abandoned**. This principle is absolute.

## §5 PCR (Point Condition Report) Data Governance

### §5.1 PCR Framework Authority
PCR framework defined in **OSM-5026 §10** (sole authority for PCR categories, visual treatment, supersession model, submission UI, WFD-5126 wireframes). **CDG-5126 = authority for PCR data classification, persistence model, cloud governance, mutation rules, offline-first discipline**. Complementary + non-conflicting.

> PCR data is distinct from all other categories. Breadcrumbs + SOS logs = Local-Only / Non-Syncable (must never leave device). PCR = community-sourced condition reporting with **legitimate cloud sync purpose** — PCRs submitted by one user are valuable to all users on that track segment. Challenge: ensure offline-first discipline + immutability while enabling appropriate sync behaviour.

### §5.2 PCR Data Classification — Conditional · Offline-queue-first with cloud sync on connectivity
Reflects 2 operational requirements simultaneously:
- **Offline-first supremacy** — PCR submission must function fully offline. User in low-signal must submit; offline submissions queued locally + synced on connectivity restoration
- **Community data value** — Cloud sync = mechanism by which one user's submission becomes visible to community. Permitted + appropriate — but **must NEVER become dependency** for submission or Survival Core function

### §5.3 PCR Persistence Model
- Local write-ahead queue = primary store for all PCR submissions
- PCR submissions crash-survivable — in-progress/unsynced submission must NOT be lost on crash/force close/device restart
- Offline submissions queued with local submission timestamp · uploaded on next connectivity
- Firestore = cloud persistence model for synced PCR records
- Local queue MUST NOT hold more than **90 days of unsynced PCR submissions** before user notification that connectivity required (implementation parameter, configurable by Founder)
- PCR cache for read display (PCR markers received from cloud) persisted locally as part of map region bundle, consistent with HazTrack™ overlay cache model

### §5.4 PCR Cloud Governance Rules

| Use case | Status | Rule |
|---|---|---|
| PCR submission sync — uploading user-submitted | **PERMITTED** | Must be performed on connectivity restoration only · Must fail silently if connectivity unavailable · Must not block PCR submission UI or any app function |
| PCR read cache — downloading PCR markers for display | **PERMITTED** | Tied to map region download · No real-time polling permitted · Cached PCR displayed with applicable freshness context per OSM-5026 §10.4 |
| PCR supersession sync — updating resolved/superseded records | **PERMITTED** | Supersession creates new record; does NOT mutate existing · Both new + archived synced as independent immutable entries |
| PCR sync creating Survival Core network dependency | **PROHIBITED** | Must NOT introduce network dependency into any Survival Core execution path · PCR background sync must be isolated from all Survival Core data stores |
| Real-time PCR polling / push notifications | **PROHIBITED Phase 1** | No real-time sync mechanism Phase 1 · PCR freshness governed by map region bundle download cadence · No push notification for new PCRs |
| PCR data influencing overlay state beyond PCR layer | **PROHIBITED** | PCR cloud data must NOT alter difficulty grade, verification shield, or any other overlay layer · LIR-04 + LIR-05 (OSM-5026 §11) absolute |

> **CONSTRAINT:** PCR sync MUST NEVER become dependency for map display, navigation, Survival Core, or any application capability. If cloud sync unavailable, app must display locally cached PCR data + continue operating without degradation. No error state, loading spinner, or user-visible sync indicator may be triggered by PCR sync unavailability.

### §5.5 PCR Mutation Rules

| PCR Field | Mutability |
|---|---|
| Category (PCR-OBS/CLO/INF/SRF/WTR/HAZ) | IMMUTABLE after write — no post-submission change |
| GPS coordinates at submission point | IMMUTABLE after write |
| Submission timestamp | IMMUTABLE after write |
| Optional description text (280 char max) | IMMUTABLE after write |
| Reporter identifier (anonymised) | IMMUTABLE after write |
| Resolution state (active/resolved/superseded) | Updated only by valid supersession events (OSM-5026 §10.3); never mutated directly |
| Unconfirmed-age flag (>90 days, no supersession) | Display state only — underlying record not mutated; flag derived at display time |

> 3 valid supersession events (OSM-5026 §10.3): (1) subsequent user confirms condition cleared; (2) subsequent user submits new PCR at same point; (3) original reporter marks own report resolved. In all cases, supersession creates new record or updates resolution flag. **Original PCR record is archived, NOT deleted/overwritten.** Original submitted record preserved in original form in all storage contexts.

### §5.6 PCR Data Schema Requirements

| Field | Type | Required | Notes |
|---|---|---|---|
| pcr_id | UUID | Yes | Unique identifier at submission |
| pcr_category | Enum (PCR-OBS, PCR-CLO, PCR-INF, PCR-SRF, PCR-WTR, PCR-HAZ) | Yes | OSM-5026 §10.2 |
| submission_timestamp | ISO 8601 UTC | Yes | Device timestamp at submission — immutable |
| coordinates_lat | Decimal degrees | Yes | User-confirmed submission point — immutable |
| coordinates_lng | Decimal degrees | Yes | User-confirmed submission point — immutable |
| description | String (max 280, plain text) | No | Optional free-text; no structured fields |
| reporter_archetype | Anonymised archetype token | No | Anonymised; no PII |
| resolution_state | Enum: active, resolved, superseded | Yes | Default active; updated by supersession events only |
| superseded_by | UUID (nullable) | No | Reference to superseding PCR; null if not superseded |
| sync_status | Enum: queued, synced | Yes (local only) | Local field only — not transmitted to cloud |

> **CONSTRAINT:** Prohibited PCR schema fields regardless of implementation convenience: `confirmation_count` (density metric — prohibited by OSM-5026 §4.1), `trust_score` or `quality_weight` (prohibited by OSM-5026 §4.2), `reporter_location_history` (privacy violation), `photo_attachment` (deferred Phase 2 per PRD-5126), any field derived from passive telemetry or behavioural inference.

## §6 Firebase Isolation Model

### §6.1 Firebase Scope — Permitted/Prohibited

| Use case | Status | Rule |
|---|---|---|
| Non-Survival Core data persistence (TrackIQ scoring, preferences, message history, HazTrack overlay cache, PCR data) | **PERMITTED** | Limited to non-Survival Core only |
| Survival Core data (breadcrumbs, SOS log, local event log) | **PROHIBITED** | Must use Firebase-independent device-local storage |
| Offline persistence mode for non-Survival Core | **PERMITTED** | Must be configured to operate fully offline — no cloud sync requirement for app function |
| Feature flags that affect Survival Core execution | **PROHIBITED** | Must NOT reach or alter any Survival Core execution path |
| Cloud sync when connectivity absent | **PERMITTED (deferred behaviour)** | Must fail silently — no user-visible error state, no Survival Core interruption |
| Reconciliation engine for locally captured Survival Core data | **PROHIBITED — ABSOLUTE** | Reconciliation engine that could mutate locally captured breadcrumb, SOS, event log = absolutely prohibited |
| PCR background sync thread accessing Survival Core data stores | **PROHIBITED** | PCR sync must be isolated from all Survival Core data stores — no shared write contexts |

### §6.2 Race Condition & Mutation Prevention (architectural prevention required)

6 scenarios representing race conditions that could result in unintended mutation of Survival Core data:
- Connectivity restoration triggering sync operation that overwrites local data with cloud state — must be prevented by write-once local persistence
- Firebase offline persistence cache being treated as authoritative source during conflict resolution — Survival Core local storage always authoritative
- Background Firebase sync thread accessing/locking Survival Core data stores — must be isolated from Firebase write contexts
- Feature flag update from Firebase cloud overriding Survival Core execution state — feature flags must NOT have write access to Survival Core
- App restart triggering Firebase restore operation that replaces local breadcrumb data — breadcrumb store must be Firebase-independent + restore-proof
- PCR background sync thread sharing write context with any Survival Core data store — PCR sync + Survival Core persistence must use architecturally isolated storage contexts

## §7 Encryption & Security Requirements

| Requirement | Standard | Scope |
|---|---|---|
| Encryption at rest | AES-256 | All locally persisted data — Survival Core + non-Survival Core |
| Encryption in transit | TLS 1.3 | All non-Survival Core cloud sync |
| Location data to cloud | Explicit informed consent required before transmission | Must be reviewed for APPs compliance before launch (RT-17) |
| SOS data transmission | **PROHIBITED Phase 1** | No location or SOS data may be transmitted externally |
| Breadcrumb data transmission | **PROHIBITED Phase 1** | Local-Only — Non-Syncable classification enforced |
| Pro tier incident log encryption | AES-256 — mandatory regardless of standard posture | APP 3 sensitive classification (see §10) |
| PCR data in transit | TLS 1.3 | All PCR cloud sync; no PII in PCR transmission |

## §8 Data Model Neutrality (ESF-5026 §8)

CDG-5126 enforces data model neutrality per ESF-5026 §8. Phase 1 data models may include abstract capability flags provided they do NOT name any transport medium, do NOT imply escalation capability, remain null/false in Phase 1, and do NOT alter runtime execution.

> **CONSTRAINT:** Following satellite-specific field names **prohibited** in Phase 1 data models — must NOT appear in any data schema, class definition, database table, or field name in any Phase 1 build artefact:
> `satReady` (may exist in CAL schema ONLY, set to false — NOT in application data models) · `satelliteEndpoint` · `satTransmissionStatus` · `satellitePayload` · `transmission_status` · `satellite_endpoint` · `dispatch_channel` · `fallback_mode` · `connectivity_priority`.
>
> Authority: ESF-5026 §8 + FSD-5126 §4.4.4. Prohibited field names in PCR or any other data model = non-compliance regardless of whether populated/active/annotated as dormant.

**Acceptable neutral Phase 1 placeholder fields** (where architectural separation requires):
- `externalChannelEnabled: false`
- `transportCapabilityLevel: 0`
- `escalationMode: none`
- `externalInterfaceState: inactive`

- **APPs** — internal review of all data collection/storage/transmission before Beta-Ready release (RT-17)
- No location data transmitted to any cloud service without explicit informed user consent
- Consent mechanism + disclosure language reviewed for APPs compliance by qualified legal counsel
- Data retention policies for local event log (30-day min), breadcrumb, SOS log, PCR queue documented + reviewed
- Privacy jurisdiction obligations (APPs, GDPR, CCPA, NZPA) addressed through product design before release into those jurisdictions

## §10 First Aid Reference Data Governance

Implements data governance obligations defined in FRM-5126 §6.2 (Australian Privacy Act 1988). Read with FRM-5126 §2 (Content Model) + TAA-5126 §8 (Archetype Tier Definitions).

### §10.2 Free Tier — Read-Only Reference
- Universal Baseline (FRM-5126 §2.1) available on Free tier without auth/tier gate/paywall · static read-only · NO personal data collected
- NO write operations to any local/cloud store during content browsing
- NO analytics/telemetry/usage data from individual sessions collected/transmitted
- Content pre-loaded at install · NO network request for Universal Baseline content
- APP personal-info obligations not engaged (Free tier collects no personal info) · must remain confirmed via Trackaroo Systems legal review

### §10.3 Plus Tier — Event Log
- Records session activity — timestamps + user-entered notes only
- Does NOT capture health info as defined under APP 3
- Log entries: session timestamp · content category accessed · optional free-text notes by user
- Log must NOT capture/infer/derive any health or medical info about user or third party · Notes unstructured free text · NO diagnosis/condition/clinical classification field may exist in data schema
- Log data persisted via Firestore offline persistence — standard non-Survival Core model
- Log entries immutable after write · No modification/correction/deletion permitted via app
- Plus event log NOT treated as APP 3 sensitive info (no health info) · Standard APPs obligations apply · Privacy Policy + privacy nutrition label disclosure required

### §10.4 Pro Tier — Incident Log
- Designed for Remote Professional archetype · May contain sensitive health info if user enters patient condition notes/history/clinical observations · **APP 3 sensitive information**

**Mandatory data handling for Pro incident log:**
- **Explicit informed consent** before activation · Consent recorded · Reviewed by qualified legal counsel for APP 3 compliance before enabling in production (RT-17)
- Incident log encrypted at rest **AES-256** · Applies regardless of standard encryption posture
- Incident log data must NOT be transmitted to any cloud without **explicit user consent separate from general app consent** · Cloud sync, if enabled, must use **TLS 1.3** in transit
- Primary store for incident log = device-local · Cloud sync user-optional · NOT required for any Pro tier function
- Incident log entries immutable after write · No retroactive modification/correction/deletion permitted via app · Supports evidentiary integrity for Remote Pro
- Data retention + deletion policy documented + disclosed in Privacy Policy before Pro tier enabled · Policy must address: max local retention · user-initiated deletion · cloud deletion if sync enabled · obligations on operator upon user account closure

> **RELEASE GATE:** Pro tier incident log contains APP 3 sensitive info. Consent mechanism, data schema, retention policy, deletion capability, cloud sync terms reviewed by qualified legal counsel before Pro tier enabled in production. **Beta-Ready gate condition.** Trackaroo Systems must engage legal counsel before Pro tier enabled.
