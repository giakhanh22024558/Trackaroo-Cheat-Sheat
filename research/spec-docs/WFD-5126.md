## 1. Executive Summary & Purpose

WFD-5126 defines the authoritative UI state and wireframe behaviour for Trackaroo® Phase 1. Supersedes WFD-5026 in full. Sole WFD authority document for all Phase 1 development and vendor engagement.

This document:
- Governs visual state representation and UI pathways across all Phase 1 subsystems
- Enforces deterministic, offline-first UX posture in all screen states
- Prevents UI-level behavioural reinterpretation of UXS-5726 or PRD-5126 rules
- Preserves all Phase 1 prohibitions including the **≤2-tap SOS discipline**
- Defines all wireframe states required before subsystem development may commence
- Establishes the **WFD-5126 build gate** as a non-waivable development prerequisite
- Defines the six PCR wireframe states (PCR-WF-01 through PCR-WF-06) required by OSM-5026 §10.6

### Authority Position — What WFD Does NOT Do
- Define functional execution logic (FSD-5126)
- Define acceptance criteria (PRD-5126)
- Define validation methodology (TQP-5026)
- Define overlay semantics (OSM-5026)
- Define cloud or infrastructure controls (CDG-5126)
- Define battery performance standards (BPS-5126)
- Define emergency safety framework constraints (ESF-5026)

**WFD-5126 introduces no new functional capability.**

## 4. Technical Constraints & Prohibitions

### 4.1 Deterministic Representation
- UI state transitions must map directly to FSD-5126-defined execution events
- No UI element may imply adaptive, heuristic, predictive, or inference-driven behaviour
- No animation, indicator, badge, or copy may imply background recalculation or monitoring
- Rule-based threshold triggers (e.g. stop-detection prompts) are deterministic and permitted

### 4.2 Prohibited Visual Implications
The UI must NOT represent, simulate, preview, or imply any of the following, regardless of connectivity medium, SDK presence, feature flag state, or conditional runtime pathway:

- AI inference or adaptive intelligence
- Satellite transmission or satellite signal presence (Phase 1)
- Mesh networking topology or relay routing paths
- Autonomous escalation of any kind
- Third-party dispatch or emergency services notification
- Emergency Data Escrow transmission or activation
- Gamification, points, rewards, or leaderboard mechanics
- Confidence scoring or trust weighting
- Density weighting or crowdsource-derived quality scores
- Intelligent, adaptive, or predictive routing
- Predictive hazard modelling or forward-looking risk assessment
- Dynamic recalculation engines or background optimisation
- Monitoring or surveillance of any group member

### 4.3 Overlay Semantic Protection
WFD-5126 must not redefine overlay semantics (exclusively OSM-5026), modify difficulty representation due to freshness state, introduce countdown timers, add confidence/trust scoring, blend verification with difficulty, or merge PCR state with Gold/Grey/No Shield system.

> **CONSTRAINT:** Difficulty colour must never represent freshness or verification state. Freshness shields must never represent terrain difficulty. PCR markers must never use Gold/Grey/No Shield encoding (OSM-5026 LIR-04). These layers are always independent and must never be merged, combined, or used interchangeably.

## 5. UI State & Wireframe Governance

### 5.1 Global UI Principles
- Offline is a first-class observable state — must be visibly communicated at all times
- UI must communicate deterministic execution — no automated/background decision-making
- No state may imply monitoring, supervision, or automated response
- All copy must align with UXS-5726 behavioural posture: **calm, informing, non-alarming**
- WCAG 2.1 AA compliance mandatory across all UI surfaces
- Touch targets for Survival Core controls: minimum 44×44pt; recommended 60×60pt
- All critical flows must be completable **one-handed and with gloves**

### 5.2 Home Screen Layout — Required Elements
- Primary navigation access
- Visible SOS control — accessible within **≤2 taps**
- Activity context selector (visual presentation only — does not alter routing or Survival Core logic)
- Map preview
- CAL connectivity status indicator

> **CONSTRAINT (≤2-tap SOS Immutable):** SOS must remain accessible within ≤2 taps from any screen state. This is an immutable Survival Core requirement. No home screen redesign may push SOS beyond this threshold. No escalation indicators, responder badges, or dispatch-implication elements are permitted.

### 5.3 Navigation Screen State
Must display: map layer · breadcrumb trail · orientation indicator · current location marker · CAL connectivity status indicator · **BackTrack™ access path — accessible within ≤3 taps from any navigation screen state without exception** (UXS-5726 §9.1; PRD-5126 §4.1.3).

Must NOT imply: predictive rerouting · intelligent optimisation · adaptive path selection · confidence-weighted routing · background recalculation.

### 5.4 Hazard & Overlay State
Must display: hazard iconography per OSM-5026 · source attribution · verification shield per OSM-5026 · freshness indicator per OSM-5026 · PCR markers per OSM-5026 §10.4 where active PCRs are present.

Must NOT display: expiry countdown timers · animated decay indicators · risk scoring · automated rerouting indicators · any visual element blending difficulty/freshness · PCR markers using Gold/Grey/No Shield (LIR-04 violation).

The Phase 1 valid overlay state matrix is defined in **OSM-5026 §13 (eight valid state combinations)** incorporating the PCR dimension.

### 5.5 SOS & Emergency Logging (Non-Dispatch) UI
Must display:
- Clear SOS activation control — ≤2 taps from any screen state
- Offline disclaimer confirming local log only
- QR fallback option for emergency data sharing
- Log confirmation state confirming local write complete

Must NOT imply: dispatch · escalation · notification · monitoring · satellite transmission (Phase 1) · Emergency Data Escrow transmission.

> **CONSTRAINT:** Distress breadcrumb remains local only in Phase 1. SOS feedback must use calm, factual language. It must never imply help has been dispatched, signal has been sent, or emergency services notified. ESF-5026 governs the SOS liability boundary and must be reviewed before any SOS-adjacent content is finalised.

### 5.6 Offline & Degraded State Indicators
UI must visibly indicate: offline mode (no connectivity) · limited connectivity (partial signal) · stale overlay data (per OSM-5026 freshness rules).

Indicators must be: deterministic (triggered by defined execution events, not inference) · non-alarmist · non-implying-of-pending-automated-action · never present absence of data as all-clear/safe.

### 5.7 TrackMate™ Group & Messaging UI
Must display: offline queue indicator · deterministic sync confirmation · group member presence (Last Known state with elapsed time) · **Group Health Envelope — binary visual indicator only** (all present / member outside envelope) · transport status (simple indicator only) · maximum 8 active members enforced.

Must NOT display: mesh topology indicators · peer relay icons · satellite signal icons (Phase 1) · indirect routing representation · monitoring/surveillance posture · automated distress triggers from Group Health Envelope.

> **CONSTRAINT:** Group Health Envelope is a visual indicator only. Must not trigger any escalation pathway or SOS flow. Must not imply monitoring, surveillance, or automated distress assessment. Last Known state must use neutral, non-alarming presentation. No alarm language permitted for group member absence.

### 5.8 TrackMate™ LoRa Hardware Onboarding UI — Four Mandatory States

| # | Wireframe State | UI Requirements |
|---|---|---|
| 1 | First-use onboarding with LoRa hardware expectation disclosure | Communicates long-range capability requires compatible LoRa hardware. Indicative ranges: short-range ~100–200m; long-range with hardware 5–15km open terrain. Calm, informing language only. No implication long-range is available by default. |
| 2 | Short-range-only operating mode (no LoRa detected) | UI confirms short-range-only mode. Must not imply long-range capability. Must not display error state — this is normal operating mode without hardware. Calm, informing language. |
| 3 | LoRa hardware detected and active | App acknowledges range extension. Confirms active transport. Updates visible range indicator. **No mesh topology or relay path representation.** |
| 4 | LoRa hardware disconnected mid-session | UI communicates transition to short-range-only mode. Calm, non-alarming language. Must not imply loss of safety capability. Queued messages held for delivery. No escalation or alarm triggered. |

> **RELEASE GATE:** No TrackMate™ onboarding development may commence until all four LoRa hardware wireframe states are formally approved under WFD-5126. **RT-19** (LoRa onboarding disclosure absent) is a named rejection trigger in PRD-5126. Development against unapproved wireframe states is **RT-18**.

### 5.9 First Aid Reference UI
Must display: step-by-step reference content (structured, never freeform) · calm, non-alarming copy · **DRSABCD + primary patient survey are the declared Universal Baseline reference standards. START triage is a Pro tier professional capability only** · fully offline-accessible content · clear instruction to seek emergency services where available.

Must NOT: introduce diagnostic AI or symptom inference · trigger SOS pathways · imply professional medical help has been contacted · vary content based on telemetry/location/historical data.

> **RELEASE GATE:** All First Aid Reference content must be reviewed and approved by a qualified clinical reviewer commissioned by Trackaroo Systems before exposure to any user. Authority: PRD-5126 RT-12.

> **CONSTRAINT — MANDATORY DISCLAIMER (A-14, Persistent Display):** Required text: *"This is reference information only. It is not professional medical advice, diagnosis, or treatment. Always seek the assistance of qualified emergency services or a medical professional where available. In an emergency, call Triple Zero (000) in Australia, or your local emergency number."* Must be persistently visible on every First Aid Reference screen — not modal, toast, or dismissible. Locale-appropriate emergency number must substitute dynamically; international fallback ("contact your local emergency services") universal default. (Authority: FRM-5226 — internal governance.)

> **CONSTRAINT — UNIVERSAL BASELINE ACCESSIBILITY (A-15):** Universal Baseline content (DRSABCD, primary patient survey, uncontrolled bleeding, shock recognition, burns, fracture/spinal immobilisation, choking) must be accessible without any paywall prompt, tier gate, or authentication. **A tier upgrade prompt must never appear on any Universal Baseline screen.** Upgrade prompts permitted only at: (1) category-index viewing with Plus content, (2) return to main menu after Universal Baseline completion, (3) attempted access to named Plus/Pro item. **No upgrade prompt may use language implying safety is at risk** due to current tier. **No upgrade prompt may appear if SOS active or in any Survival Core path.** (Authority: FRM-5226 §9.)

### 5.10 TrackIQ™ UI
Must display: deterministic difficulty grade per OSM-5026 colour scheme (Easy, Moderate, Difficult, Extreme, Unknown) · verification shield per OSM-5026 (None, Grey, Gold) · track metadata (distance, elevation, surface type) · stop-detection prompt at defined thresholds (fixed trigger, not adaptive).

Must NOT: imply AI scoring/grading · display telemetry-weighted grades · allow HazTrack™ event data to mutate any TrackIQ™ grade (**RT-14**) · display confidence percentages · merge PCR state with TrackIQ™ scoring (**LIR-05**).

### 5.11 Points of Interest UI
Must display: category icon · descriptive metadata · optional photo indicator (photo-enabled POIs deferred to Phase 2 — indicator only where applicable).

Trust indicators must be presentation-only. **No star ratings, leaderboards, points, rewards, or density signals permitted.**

### 5.12 Activity Context Visual Boundary
Activity selection may adjust: iconography · labelling · contextual copy and data hierarchy emphasis.

Activity selection must NOT alter: routing logic · hazard prioritisation · breadcrumb logic · telemetry behaviour · performance behaviour · overlay semantics · **SOS tap count or access path**.

Activity context is informational and visual only. Archetype/activity context definitions are authoritative in **TAA-5126**.

### 5.13 Placeholder Modules UI
All Phase 2+ placeholders must: be visible · be schema-complete · display exactly: **"Inactive in Phase 1."**

Placeholders must contain no executable logic, no preview indicators, no unlock/upgrade prompts, no teaser language, and no satellite transmission artefacts.

### 5.14 Snow & Alpine Wireframe States — Mandatory
Five mandatory states before any Snow & Alpine development may commence:
- Standard operating mode — Snow & Alpine archetype active
- Reduced-visibility UI state
- **Whiteout-condition UI state** — maximum contrast, minimal information load
- Avalanche overlay state
- Glove-compatible interaction state

> **RELEASE GATE:** WFD-5126 approval of all Snow & Alpine wireframe states required before any Snow & Alpine development commences. Applies whether or not full module activation is Phase 1 or Phase 2.5.

### 5.15 CarPlay & Android Auto Constraints
Deferred to Phase 2. Phase 1 scaffolding/placeholder constraints: simplified layout only — **≤5 visible UI elements rule** · SOS accessible within ≤2 taps · no new execution pathways introduced · no capability implied that does not exist in the mobile app · **no CarPlay-specific or Android Auto-specific development may commence in Phase 1**.

### 5.16 Local Event Log Viewer
Required wireframe state (PRD-5126 §4.1.7; FSD-5126 §4.5.3):
- Chronological event list — timestamp, event type, brief description
- Read-only display — no editing, deletion, or modification
- Fully offline — no connectivity required
- No adaptive display — deterministic rendering only
- SOS log entries displayed with calm, factual language only

### 5.17 PCR Wireframe States — Six Mandatory (PCR-WF-01 → PCR-WF-06)

PCR markers are a **distinct visual layer**. Must never use Gold/Grey/No Shield encoding (LIR-04). PCR state must never be inferred from or merged with shield state or difficulty grade (LIR-05).

| State ID | Name | UI Requirements | Authority |
|---|---|---|---|
| **PCR-WF-01** | Map view with active PCR markers | All six PCR categories (OBS, CLO, INF, SRF, WTR, HAZ) representable. Each: unique icon with **amber ring `#FFA000`** unconfirmed; **red ring `#D32F2F`** confirmed by multiple reporters. Must not obscure difficulty grade or verification shield. WCAG 2.1 AA contrast + glove-compatible. Icons visually distinct from hazard overlay icons. | OSM-5026 §10.4, LIR-04 |
| **PCR-WF-02** | PCR detail tap — expanded card | Tap opens expanded card without navigating away. Displays category, submission date, reporter archetype (anonymised), confirmation count. Must NOT display reporter identity, precise location, scoring, or social signals. Card dismissible without blocking map interaction. | OSM-5026 §10.4 |
| **PCR-WF-03** | PCR submission flow — online mode | Completable from navigation view without navigating away. Step 1: select category (6 available). Step 2: confirm point location. Step 3: optional free-text description (**max 280 chars, plain text**). Step 4: Submit. On submission: "Condition reported. Thank you." **No scoring, social signal, or additional feedback. No photo attachment in Phase 1. Submission never triggered automatically (RG-10, RT-21).** | OSM-5026 §10.5 |
| **PCR-WF-04** | PCR submission — offline queued state | Offline queue indicator displayed. Confirmation: "Condition reported. Will sync when connected." Must NOT imply report has synced. Queued state visible. On connectivity restoration, PCRs sync automatically. Timestamp preserved from submission. **PCR submissions never silently discarded.** Consistent with CAL `queueEnabled`. | OSM-5026 §10.5 |
| **PCR-WF-05** | Unconfirmed-age PCR display | PCRs not superseded for >90 days display with **muted icon treatment**. Label: "Condition age unknown." PCR remains active — not removed/suppressed/expired. **No countdown timer visible.** Muted treatment must still meet WCAG 2.1 AA + glove-compatible touch targets. 90-day threshold is an FSD-5126 §4.3.4 implementation parameter. | OSM-5026 §10.4 |
| **PCR-WF-06** | Resolved PCR — history view | Resolved PCRs **never displayed on active map view**. Accessible only in track detail/history view: category, original submission date, resolution date, resolution method. Visually distinct from active PCRs. **Resolved PCR data is retained and never deleted.** Re-activation only via new explicit user submission. | OSM-5026 §10.3, §10.4 |

> **CONSTRAINT:** PCR submission must not be triggered automatically from user behaviour, TrackIQ™ scoring, stop-detection logic, or any other automated mechanism. **Explicit user action is the only valid submission trigger. Automatic PCR triggering constitutes RG-10 and RT-21. Non-negotiable.** PCR markers must never use Gold/Grey/No Shield encoding (LIR-04 violation constitutes RG-07). PCR state must never influence difficulty grade or shield state. HazTrack™ events must never alter PCR state (LIR-05).

> **RELEASE GATE:** No PCR framework development may commence until all six PCR wireframe states (PCR-WF-01 through PCR-WF-06) are formally approved under WFD-5126 by the Founder & Project Director. Development against unapproved PCR wireframe states constitutes RT-18. Approval of all six states is **mandatory for Alpha gate progression**.

## 6. Connectivity Status & CAL Indicator

CAL status indicator must be visible at all times in navigation view. CAL state flags defined in FSD-5126.

| CAL State Flag | User-Visible State | UI Requirement |
|---|---|---|
| `offlineBeacon` | Offline / No connectivity | Calm, non-alarming offline indicator. No implied pending action. |
| `queueEnabled` | Messages queued for delivery | Queue indicator visible in TrackMate™ UI and PCR offline queue state (PCR-WF-04). Deterministic — not animated or pulsing. |
| `satReady` | Satellite pathway ready (Phase 2) | **Placeholder only in Phase 1. Must display "Inactive in Phase 1." No satellite signal icon.** |
| `partialSignal` | Partial signal | Degraded state indicator. No implication of automated recovery or escalation. |

> **CONSTRAINT:** The `satReady` flag is a Phase 1 CAL architectural scaffold. Must be declared in CAL state schema and set to **false**. No satellite SDK, transmission logic, or user-visible satellite reference may be present in Phase 1 build. **RT-01** (satellite SDK present or triggerable) is a named rejection trigger.

## 8. Phase Boundary & Deployment Logic

### 8.1 Phase Boundary
- WFD-5126 applies exclusively to Phase 1
- No clause implies Phase 2 or Phase 3 activation
- All placeholders remain inert and non-executable
- Phase 2 scaffolding present in Phase 1 builds must be confirmed inert through QA. Non-inert scaffolding is **RT-09**.

### 8.2 Gate Alignment
WFD wireframe approval applies across all Phase 1 release gates: Discovery · Alpha (30 April 2026) · Beta-Ready MVP (30 June 2026) · GA.

### 8.3 Rollback Governance
If UI behaviour diverges from UXS-5726 or PRD-5126 at any gate: release progression halts immediately · behaviour corrected to comply · deterministic compliance restored · revalidation required under TQP-5026 · rollback trigger recorded in CAR-5026.

## 10. Acceptance Structure & QA Alignment

- PRD-5126 defines acceptance criteria
- TQP-5026 defines validation methodology
- FSD-5126 defines functional implementation
- WFD-5126 defines UI state conformance only — introduces no independent acceptance logic
- Final acceptance authority resides with the Founder & Project Director

### Five-Question Cognitive Hierarchy — WFD Implications

Five-question cognitive hierarchy compliance must be validated across all six archetypes and all major screen states as a TQP-5026 user testing requirement. **Five-question hierarchy failure is RT-10.**

| # | Question | WFD-5126 UI Implication |
|---|---|---|
| 1 | Where am I? | Current location marker visible and unambiguous in all navigation states. |
| 2 | Where am I going? | Destination and route clearly presented without clutter in all navigation states. |
| 3 | What surrounds me? | HazTrack™ overlays, freshness state, and active PCR markers accessible without deliberation. PCR markers must not obscure response to this question. |
| 4 | How do I get back? | BackTrack™ accessible within **≤3 taps from any navigation screen state**. Direct, unambiguous BackTrack™ entry point must be present at all times (UXS-5726 §9.1; §5.3). |
| 5 | How do I call for help? | SOS accessible within **≤2 taps** from every screen state without exception. PCR submission flow must never interrupt SOS access. |
