## Role in Authority Stack

SFD-5026 provides the **authoritative annotated SOS flow diagram** for Trackaroo® Phase 1. Vendor pack document issued alongside RFT-5026, VRB-5126, VEG-5026, and VGD-5126. Gives the delivery partner a complete, governance-grounded visual authority for the SOS subsystem covering:
- The **3-stage log sequence**
- The **≤2-tap discipline** across all screen states
- The **non-dispatch, non-transmission posture**
- All governed state transitions and prohibitions

**Phase 1 exclusively.** No clause authorises external transmission, satellite integration, or emergency dispatch capability in Phase 1. The Phase 2 satellite relay pathway is architected but **not executable** in Phase 1 (CAL `satReady` = false).

> **CONSTRAINT:** The SOS behavioural model defined in this document is **immutable**. May not be altered by vendor architecture decisions, commercial considerations, or downstream design proposals. Vendor proposals that depart from the ≤2-tap discipline, the **non-dispatch posture**, the **absolute non-transmission rule**, the **3-stage log sequence**, or the **six-element confirmation screen** are non-conforming and constitute **rejection triggers**.
> Authority: UXS-5726 §7.1; PRD-5126 §4.1.6; ESF-5026 §5.1; FSD-5126 §4.4

## 2. Governing Rules Summary

| Rule | Requirement | Authority |
|---|---|---|
| **≤2 Tap Discipline** | SOS accessible within ≤2 taps from every screen state without exception. Absolute. Non-negotiable. | UXS-5726 §7.1; PRD-5126 §4.1.6; ESF-5026 §5.1 |
| **Non-Dispatch Posture** | trackaroo® does not contact emergency services. SOS logs distress locally only. Unconditional. | ESF-5026 §4.1; PRD-5126 §4.1.6 |
| **Absolute Non-Transmission** | **No outbound connection — HTTP, HTTPS, UDP, TCP, or SDK callback — from the application process during SOS activation or any Survival Core execution path.** | ESF-5026 §4.2; FSD-5126 §4.4.4 |
| **Log Write ≤1 Second** | Stage 1 log write and confirmation screen must render within **1 second of Tap 2** under all device states in validation matrix. | PRD-5126 §4.1.6; FSD-5126 §4.4.2 |
| **3-Stage Log Sequence** | **Stage 1:** timestamp + device ID at Tap 2 (no GPS dependency). **Stage 2:** GPS acquiring, coords field pending. **Stage 3:** coords appended on GPS fix. | 21 Mar 2026 design decision (ESF-5026 §5.4 / FSD-5126 §4.4.2 amendment pending) |
| **Six Feedback Elements** | All six elements mandatory on confirmation screen. Each is a discrete acceptance criterion. **No element may be removed, collapsed, or made dismissible.** | PRD-5126 §4.1.6; FSD-5126 §4.4.3; ESF-5026 §5.3 |
| **Copy Discipline** | All SOS copy: calm, factual, non-dispatch. No alarm language. No implication that rescue services have been notified. | ESF-5026 §5.2; UXS-5726 §7.3 |
| **Immutable Log** | No post-write modification to any SOS log entry. Non-reconcilable, crash-survivable, **Firebase-independent**. | ESF-5026 §5.4; PRD-5126 §4.1.7; CDG-5126 |
| **Satellite Field Prohibition** | No satellite-related field names in any Phase 1 data model. Prohibited field list: ESF-5026 §8. | ESF-5026 §8; FSD-5126 §4.4.4 |
| **QR Fallback** | Locally generated. No network dependency. No transmission pathway. Offline-operable. Suitable for handover to emergency responders. | ESF-5026 §5.6; FSD-5126 §4.4.5 |
| **Distress Breadcrumb** | Activates immediately on Tap 2 (parallel with log write). Elevated interval: **5m / 5s**. Available all tiers. **No paywall permitted.** | ESF-5026 §5.5; UXS-5726; TAA-5126 §8.1 |
| **SOS Onboarding** | **Click-through acknowledgement mandatory** before any user exposure to the SOS function. Pre-Alpha gate condition. **Not scroll-past.** | ESF-5026 §6.3; FSD-5126 §4.4.1 |

## 3. SOS Flow Diagram

Four swim lanes. Read top to bottom. Each node represents a discrete state or transition. Governance annotations appear in rightmost lane.

### 3.1 Swim Lane Key

| Lane | Colour | Content |
|---|---|---|
| **USER ACTION** | Blue tint | What the user does or sees at this step |
| **SYSTEM — LOG** | Amber tint | Log write operations and data model actions |
| **SYSTEM — UI** | Green tint | Screen state transitions and rendered UI elements |
| **GOVERNANCE** | Purple tint | Governing rule citations, authority references, prohibition flags |

### 3.2 Flow Diagram — Node Sequence

| Node | User Action | System — Log | System — UI | Governance |
|---|---|---|---|---|
| **1. ENTRY** | Any screen state in the application | — no log action — | SOS control persistently visible in UI chrome. Always rendered. Never hidden by any screen state. | All screen states without exception — nested modals, overlays, active navigation, TrackMate™, First Aid Reference, PCR, BackTrack™, archetype settings. UXS-5726 §7.1; PRD-5126 §4.1.6 |
| **2. TAP 1** | User taps persistent SOS control (Tap 1 of 2) | — no log action — | SOS confirmation prompt rendered immediately. **No network check. No loading state.** | ≤2 Tap Rule. This is Tap 1. Prompt must render without network dependency or loading delay. Rejection trigger if SOS not reachable within 2 taps from any screen state. UXS-5726 §7.1; PRD-5126 §4.1.6 |
| **3. TAP 2 + STAGE 1 LOG** | User confirms activation (Tap 2 of 2 — final). Prevents accidental trigger. | **STAGE 1 LOG WRITE** (executes immediately on Tap 2): • Timestamp (device clock at tap) • Device ID • Battery level • App state at activation. Must complete ≤1 second. Immutable. No post-write modification. | GPS Pending UI rendered: • Calm acknowledgement tag in progress • GPS acquiring indicator visible • Coordinates field: Acquiring... • **No alarm styling** | Stage 1 requires **NO GPS fix**. Written from device clock and system state immediately on tap. Log: device-resident, **Local-Only**, Non-Syncable (CDG-5126), crash-survivable, **Firebase-independent**, non-reconcilable. 21 Mar 2026 design decision. ESF-5026 §5.4; PRD-5126 §4.1.6; FSD-5126 §4.4.2 |
| **4. DISTRESS BREADCRUMB (parallel)** | — system-initiated in parallel with Node 3 — | **Distress Breadcrumb Mode activates immediately on Tap 2** • Interval: 5m OR 5s (elevated from standard 15m OR 20s) • Device-resident, Immutable • Non-Syncable — Local-Only • Firebase-independent • Crash-survivable | UI confirms distress breadcrumb active (Feedback Element 4 — see Node 6) | **Survival Core function. Available ALL tiers — no paywall permitted.** Runs parallel with Stage 1 log write and GPS acquisition — **NOT gated by GPS fix**. ESF-5026 §5.5; PRD-5126 §4.2.2; FSD-5126 §4.2.2; TAA-5126 §8.1 |
| **5. GPS BRANCH** | — | **Branch A — STAGE 2 LOG WRITE** (GPS fix available): Stage 2 LOG WRITE — Coordinates appended to Stage 1 log entry on GPS fix. Log entry now complete. **Branch B — STAGE 2 (GPS pending):** Stage 2 — GPS PENDING. Coords field = null/pending in log entry. Auto-completes on GPS fix. If fix never obtained, field remains null. | **Branch A:** Confirmation screen rendered with full coordinates. **Branch B:** Confirmation screen rendered with 'Acquiring...' in coords field. Updates asynchronously when fix arrives. | **3-STAGE LOG SEQUENCE: Stage 1 (Tap 2): timestamp + device ID — no GPS dependency. Stage 2 (GPS Pending): coords field null/pending. Stage 3 (On Fix): coords appended.** Confirmation screen rendered in Branch B WITHOUT waiting for GPS fix. 21 Mar 2026 design decision pending ESF-5026 §5.4 + FSD-5126 §4.4.2 amendment. |
| **6. CONFIRMATION SCREEN** | User views SOS Confirmation Screen | Log entry status: • Branch A: Complete — all fields written • Branch B: Partial — coords pending, auto-completes on GPS fix | SOS CONFIRMATION SCREEN rendered with **all 6 required feedback elements** (see §3.3) | Log write + feedback render ≤1 second from Tap 2. All 6 feedback elements are discrete acceptance criteria. PRD-5126 §4.1.6; FSD-5126 §4.4.3; ESF-5026 §5.3 |
| **7. QR FALLBACK** (optional) | User taps QR Fallback option on Confirmation Screen | — no additional log action — | QR code generated locally: • Current coordinates + timestamp • Emergency log reference • Format: handover to emergency responders | **Locally generated. No network dependency.** Fully offline. Must NOT imply emergency services notified. ESF-5026 §5.6; FSD-5126 §4.4.5 |

> **ZERO TRANSMISSION BOUNDARY — [X] PROHIBITION:** No outbound connection at any point in this flow. The **absolute prohibition** governs the trackaroo® application process: no HTTPS, UDP, TCP socket, or SDK/external callback while SOS is active or during any Survival Core execution path. Scope: application process boundary only. OS-level platform services (iOS/Android daemons, push notification infrastructure, system telemetry) are outside application control and outside the scope of this rule.
> 
> No dispatch implication in any copy · No alarm language · No satellite field names in Phase 1 data model (ESF-5026 §8) · No network dependency at any flow node · SOS must not be blocked or delayed by any UI layer · No escalation or automated external action (the indication that help is on the way).

### 3.3 SOS Confirmation Screen — Six Required Feedback Elements

All six elements mandatory on the confirmation screen. **Each is a discrete acceptance criterion per PRD-5126 §4.1.6. Absence of any element on initial render is a rejection trigger.**

| # | Feedback Element | Implementation Requirement | Acceptance Standard |
|---|---|---|---|
| **1** | **Emergency state logged** | Clear, positive confirmation that the emergency event has been logged locally. Distinct confirmation signal. | User cannot mistake whether logging has occurred. Confirmed before screen advances. |
| **2** | **Timestamp displayed** | Exact timestamp recorded at Tap 2. User's local time zone. Format suitable for verbal communication (e.g. 3:47 PM AEST 29 March 2026). | Matches Stage 1 log write time exactly. |
| **3** | **Coordinates displayed** | Exact coordinates at activation. Decimal degrees, verbally communicable. If GPS pending: display 'Acquiring...' and update asynchronously on fix. | Branch A: coords present on first render. Branch B: 'Acquiring...' shown; auto-updates on GPS fix. |
| **4** | **Distress mode active** | Confirmation distress breadcrumb mode is active at elevated capture interval (5m / 5s). Visible — not collapsed. **All tiers, no paywall.** | Distress breadcrumb status clearly visible on confirmation screen. |
| **5** | **Phase 1 limitation** | Calm statement that **external transmission is not available in Phase 1**. Satellite transmission will be available in Phase 2. | Present. Calm, factual. Must not imply transmission has failed — only that it is not available in Phase 1. |
| **6** | **User responsibility** | Explicit statement this application has not contacted anyone. User must seek help through available means. In Australia: direct to Triple Zero (000) if signal available. | Non-dispatch posture unambiguous. **No implication of monitoring or response.** |

> **CONSTRAINT — Copy Discipline (mandatory):** All SOS feedback must use **calm, factual language**. Must never imply: help is on the way · rescue services have been notified · the system is monitoring the situation · a signal has been sent.
> All six elements must be present on initial render of the confirmation screen. **No element may be removed, collapsed, made dismissible, or placed behind a scroll.**
> Authority: ESF-5026 §5.2, §5.3; UXS-5726 §7.3; PRD-5126 §4.1.6

## 5. SOS Copy Discipline Reference

### 5.1 Approved Structure

Pre-drafted SOS confirmation screen copy structure approved by Trackaroo Systems as the **governance baseline**. This copy is subject to legal confirmation under **LE-02** (SOS duty-of-care review, to be commissioned at vendor appointment — 1 May 2026). Delivery partner must implement the structure as specified. Trackaroo Systems will deliver final legally confirmed copy before the Alpha gate. **No SOS copy may be published to any user without written Project Director confirmation that LE-02 review is complete.**

**Approved structure:**
> *"Your emergency has been logged locally on this device.*
> *Timestamp: [recorded at activation]*
> *Coordinates: [recorded at activation — or 'Acquiring...' if GPS pending]*
> *Distress tracking is now active.*
> *This application has not contacted emergency services. No signal has been sent. You must seek help through available means.*
> *In Australia: call Triple Zero (000) if you have any signal. In other countries: contact your local emergency services.*
> *Satellite transmission will be available in a future update."*

**Permitted framing:** Calm, factual acknowledgement. Log confirmed. Coordinates shown or acquiring. Distress mode active. Explicit non-dispatch statement. User directed to seek help.

**Prohibited framing:** Any language implying — help is on the way · rescue services have been notified · the system is monitoring the situation · a signal has been sent · satellite transmission has occurred · emergency services have been dispatched.

> **NOTE:** The copy structure reflects governance positions in UXS-5726 §7.3 and PRD-5126 §4.1.6. It is the **internal baseline draft for LE-02 legal review**. **RT-16** (SOS liability boundary legal clearance) requires written sign-off from qualified legal counsel before the Beta-Ready gate (30 October 2026). Project Director sign-off alone does not satisfy RT-16. Authority: ESF-5026 §6.2; LRB-5026 LE-02; PRD-5126 §15.

### 5.2 SOS Onboarding Acknowledgement

Before SOS function accessible to any user (including alpha testers), explicit **click-through acknowledgement** must confirm the user has understood:
- trackaroo® does NOT contact emergency services
- SOS activation logs distress locally only
- Users retain responsibility for their own emergency communication planning
- The product is NOT a substitute for a registered PLB or satellite messenger in high-risk environments

> **CONSTRAINT:** SOS onboarding acknowledgement must be a **click-through, not a scroll-past**. No user may access the SOS function without completing this acknowledgement. **Mandatory pre-Alpha gate condition.** No user exposure to the SOS function is permitted without this acknowledgement implemented and confirmed. Authority: ESF-5026 §6.3; FSD-5126 §4.4.1
