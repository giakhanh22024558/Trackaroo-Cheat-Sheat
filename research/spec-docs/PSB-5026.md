> "Single authoritative reference for phase scope boundaries across Phase 1 delivery. Defines — for NDA-approved delivery partners — precisely what is in active scope for Phase 1, what scaffolding is permitted but must remain inert, and what is deferred to Phase 2 or beyond."

**Authority basis (§1.1):** UXS-5726 §18 (Phase 1 confirmed scope + Phase 2 deferrals) · PRD-5126 §9 · FSD-5126 §9 · BTF-5126 §8 · ESF-5026 §7 · TAA-5126 §6 · VGD-5126 §4.

## §2 Governing Principles

### §2.1 Phase 1 = Maximum Defensible Launch
> "Phase 1 is NOT a minimum viable product. It is maximum defensible launch: survival-grade, offline-first, satellite-ready in architecture, compliance-driven from outset. No Phase 2 feature may be presented, implied, or scaffolded beyond declared items in §4 — including under beta or test conditions."

### §2.2 Dual-Layer Architecture
- **Survival Core** — immutable, offline-first, deterministic, non-adaptive. Offline nav, BackTrack™, HazTrack™, SOS, local logging. **No Phase 2 scaffolding permitted in Survival Core under any circumstance — automatic rejection trigger.**
- **Experience & Intelligence Layer** — may degrade gracefully offline. TrackIQ™, TrackMate™, archetypes, First Aid, visual overlays. Must NEVER block/delay/mutate Survival Core. 3 declared scaffold items permitted (§4).

### §2.3 Scaffold Discipline Rule
> **EXHAUSTIVE LIST.** Phase 2 scaffolding in Phase 1 builds is limited EXCLUSIVELY to the three declared items in §4. NOT a general licence to pre-build Phase 2. Any scaffold accessible/visible/executable in Phase 1 production build = **RT-09** rejection trigger, halts release progression. Beyond 3 items requires explicit written PD approval + separate costing outside fixed-price engagement.

## §4 Phase 1 Inert Scaffold — Declared Items Only

> **EXHAUSTIVE LIST — no other scaffold permitted.** Phase 2 scaffolding beyond these requires explicit written PD approval. Each must satisfy all 4 QA visibility requirements. Authority: FSD-5126 §9.1; BTF-5126 §8.1; PRD-5126 §9.3.

| # | Scaffold Item | Status | Present | Absent | V-Item |
|---|---|---|---|---|---|
| S-01 | BackTrack™ Emergency Escrow data schema | INERT SCAFFOLD | Local storage schema versioned for forward migration · NO executable logic | Transmission logic · Satellite SDK refs · Any user-visible Escrow artefact/label/UI | V-05 |
| S-02 | CAL `satReady` flag | NOT ACTIVATABLE | Boolean in CAL state schema only · set to false all Phase 1 builds | Active satellite SDK · Application data model reference · Any path allowing satReady to be set true | V-08 |
| S-03 | CAL satellite transport architectural pathway | INERT SCAFFOLD | Extensible CAL interface documented in FSD-5126 to accommodate satellite as future transport tier | Executable transport logic · Satellite SDK · Any Phase 1 build path capable of activating satellite | V-08 |

### §4.1 QA Visibility Requirements for All Scaffold Items
Authority: ESF-5026 §7; VGD-5126 §4. All 3 items must satisfy without exception:

| Requirement | Description |
|---|---|
| (a) Visually surfaced to QA | Each scaffold explicitly identifiable + reviewable during QA validation |
| (b) Display text | Any user-visible placeholder must display EXACTLY: **"Inactive in Phase 1."** No other wording permitted |
| (c) Schema-complete | Data structures ready for Phase 2 activation without structural rework |
| (d) Zero executable logic | NO compile-time branch capable of activating Phase 2 logic in Phase 1 production build |

### §4.2 Satellite Field Name Prohibitions (Authority: ESF-5026 §8)
Prohibited in all Phase 1 application data models. Only permitted use of `satReady` is in CAL state schema (S-02 above), set to false.

```
satReady  (CAL state schema ONLY, set to false — NOT in any application data model)
satelliteEndpoint | satTransmissionStatus | satellitePayload
transmission_status | satellite_endpoint | dispatch_channel
fallback_mode | connectivity_priority
```
Transport-specific nomenclature reserved exclusively for Phase 2 authority documentation.

## §5 Phase 2 Deferrals

### §5.1 Absolute Satellite Prohibition (Authority: ESF-5026 §7.1)
Phase 1 includes **zero satellite capability of any kind**. ALL prohibited under all circumstances — including debug, beta, GA states:
- Satellite transmission of any kind (offline, online, debug, beta, GA)
- Satellite SDK presence (linked or dormant)
- Stubbed satellite interface classes
- Feature-flagged satellite modules
- Commented satellite code
- Conditional compilation branches targeting satellite
- Satellite fallback messaging
- Emergency dispatch via satellite
- Hybrid terrestrial/satellite switching
- Background satellite signal detection
- Satellite signal iconography implying transmission
- Satellite-specific data model fields (see §4.2)

**Detection of any prohibited satellite item = RT-01.**

### §5.2 Deferred Feature Register
| Feature | Status | Rationale | Phase 1 obligation |
|---|---|---|---|
| Satellite transmission (all forms) | PHASE 2 | Phase 2 transport — zero capability Phase 1 | CAL architecture must accommodate satellite without structural rework (V-08); Escrow schema scaffold (S-01) |
| BackTrack™ Emergency Escrow transmission/activation | PHASE 2 | Sync pathway via satellite Phase 2 | Data schema scaffold only (S-01); NO transmission logic; no user-visible artefact |
| TrackMate™ satellite relay tier | PHASE 2 | Phase 2 transport tier via BackTrack™ Emergency Escrow pathway | CAL satReady declared but not activatable (S-02, S-03) |

## §7 TrackMate™ Transport Tier Summary

> TrackMate™ transport is **transport-transparent to the user**. NO manual transport selection required. The CAL manages transport priority. Reference stack represents informed candidates — vendors must confirm/refine/propose alternatives with full technical justification. Must satisfy capability goals in UXS-5726 §11 + battery in BPS-5126.

| Tier | Mechanism | Status | Range | Battery | Notes |
|---|---|---|---|---|---|
| Tier 1 — Primary | BLE Mesh | ACTIVE | ~100m device-to-device · multi-hop | Very low | Continuous bg · device-native · V-01 |
| Tier 1 — Fallback | Wi-Fi Direct | ACTIVE | ~200m line-of-sight | Medium | System-activated only · must not be held continuously · V-02 |
| Tier 2 — Long Range | LoRa (optional hardware) | ACTIVE | 5–15 km open | Very low | GoTenna/Meshtastic-class · auto-detected · 4 WFD-5126 states · V-03 |
| Phase 2 only | Satellite relay | PHASE 2 | Global | TBD | Via BackTrack™ Emergency Escrow pathway · CAL architected · NOT executable · satReady = false |

**CAL State Flags — Phase 1 status:**
- `queueEnabled`: ACTIVE — true whenever TrackMate™ session active
- `offlineBeacon`: ACTIVE — true when BLE Mesh active + broadcasting presence
- `satReady`: **DECLARED, NOT ACTIVATABLE** — boolean in CAL state schema only · false in all Phase 1 builds · no satellite SDK · must NOT appear in application data models

> A Phase 2/3 scaffold embedded in Survival Core path — or in Experience & Intelligence Layer — that can be accidentally triggered in Phase 1 production = behavioural contamination event. Contaminated build may NOT advance until resolved + re-validated by QA.
