## §4 Technical Constraints & Prohibitions

### §4.1 Non-Dispatch Emergency Posture
trackaroo® Phase 1:
- Does not contact emergency services
- Does not transmit distress externally
- Does not notify responders
- Does not monitor user safety

> SOS remains user-triggered + device-resident. Absolute. May NOT be overridden by vendor architecture decision, commercial consideration, or future authority document without written Founder approval.

### §4.2 Absolute Non-Transmission Rule

> **CONSTRAINT:** No emergency transmission may occur under any condition, including:
> - Offline state · Online state · Feature-flag state · Debug builds · Development builds · Beta releases · GA releases
>
> Transmission capability must NOT exist. Rule governs trackaroo® app process: app process must NOT initiate any outbound connection — HTTP, HTTPS, UDP, TCP socket, SDK-level callback — while SOS is active OR during any Survival Core execution path. Applies to all connectivity media, SDK abstractions, conditional compilation states, build variants.
>
> OS-level system services operating OUTSIDE app process (iOS/Android platform daemons, push notification infrastructure, OS device telemetry) are OUTSIDE scope + control of application. Delivery partner is NOT required to suppress or account for OS-level services.

> **Zero Transmission Scope — Application Process Boundary:** Compliance obligation extends to the application process + all SDK dependencies, libraries, runtime components within it. Consistent with UXS-5726 §7.4 + PRD-5126 §4.1.6. Absolute prohibition unchanged; this defines technical perimeter.

### §4.3 Offline-First Supremacy Reinforcement
All safety-critical flows must function fully offline, generate immutable local logs, avoid any server dependency. Cloud strictly secondary + supplementary. No Survival Core path may introduce network dependency under any execution condition.

### §4.4 No Escalation / No Monitoring Clause
The system shall not:
- Escalate distress events to any external service
- Trigger third-party alerts
- Represent monitoring in progress
- Imply responder awareness
- Present Group Health Envelope or any feature as a monitoring system

### §4.5 Medium-Agnostic Prohibition Clause
All prohibitions apply regardless of connectivity medium, SDK inclusion, conditional compilation, feature flag state, or build variant. **Dormant pathways constitute violation.**

### §4.6 Placeholder Discipline & Future Activation Safeguard
Phase 1 may include structural placeholders solely to preserve architectural separation for future phases. All placeholders must:
- Be transport-agnostic — no transport medium named by name
- Not introduce SDK bindings or executable endpoints
- Not create compile-time branches or feature flags
- Not alter runtime behaviour
- Default to inert state
- Not modify UX or iconography
- Not introduce new data flows
- Not alter cloud classification under CDG-5126
- Not imply escalation capability

> **CONSTRAINT:** Phase 2 architectural hooks permitted in Phase 1 builds ONLY where explicitly specified in FSD-5126. All Phase 2 placeholders must: (a) visually surfaced to QA · (b) display **exactly the text "Inactive in Phase 1."** · (c) be schema-complete · (d) contain zero executable logic.

## §5 Emergency Operating Model — Phase 1

### §5.1 SOS Activation Discipline (≤2 Tap Constraint)
> SOS must remain accessible within **≤2 taps from any screen state**. Non-negotiable. May NOT be altered by archetype preset, activity context, subscription tier, or any UI customisation. Authority: UXS-5726 §7.1, PRD-5126 §4.1.6.

### §5.2 SOS User Feedback — Copy Discipline

> **SOS COPY RULE — NON-NEGOTIABLE:** SOS user feedback must NEVER imply help has been dispatched, rescue underway, or any emergency service notified.
>
> **Permitted framing:** Calm, factual acknowledgement that SOS event has been logged locally.
>
> **Prohibited framing:** Any language implying dispatch, rescue activation, emergency services notification, or imminent response.
>
> Authority: UXS-5726 §7.3. Violation is automatic rejection trigger — RT-06 (VGD-5126).

### §5.3 SOS User Feedback — 6 Required Elements (FSD-5126 §4.4.3)

| Element | Implementation |
|---|---|
| Emergency state logged | Clear confirmation event logged locally |
| Timestamp displayed | Exact timestamp in user's local time zone, formatted for verbal communication |
| Coordinates displayed | Exact coordinates user can communicate verbally (e.g. decimal degrees) |
| Phase 1 limitation | Calm statement that external transmission is NOT available Phase 1, and transmission will occur when satellite connectivity available Phase 2 |
| User responsibility | Clear direction: user must seek help via available means — system has NOT contacted anyone |
| Distress mode active | Confirmation distress breadcrumb mode is now active at elevated capture interval (5m / 5s) |

### §5.4 Local Emergency Logging
Upon SOS activation, immutable local log entry must be generated:
- Timestamped at moment of activation
- Device-resident — Local-Only, Non-Syncable (CDG-5126)
- Non-reconcilable — no post-write modification
- Crash-survivable — must survive app crash, force close, device restart
- Firebase-independent — no cloud pathway, active or dormant

### §5.5 Distress Breadcrumb Mode
Distress Breadcrumb Mode activates **automatically** upon SOS activation. Increases breadcrumb capture density from standard interval (15m / 20s) to distress interval (5m / 5s).

Properties:
- Generated locally — device-resident, immutable
- Crash-survivable
- Non-Syncable — Local-Only classification, Firebase-independent
- **Available on all subscription tiers** — this is a survival function and may NOT be placed behind a paywall (UXS-5726, TAA-5126 §8.1)

### §5.6 QR Fallback Mechanism
QR fallback provides locally-generated, offline-operable reference point for emergency handover.

Properties:
- Generated locally — no network dependency
- Contains no transmission pathway
- Operates fully offline
- Does NOT imply emergency services notified
- Presents user's current coordinates, timestamp, emergency log reference in format suitable for handover to emergency responders (Authority: FSD-5126 §4.4.5)

> **CONSTRAINT:** **Phase 1 — Display-only.** No export, download, or share capability permitted. QR code generated locally + displayed on device screen only. Mechanism has no output capability beyond device display. Authority: Project Director decision 1 April 2026, CR04 Q26 closed.

> **NOTE — Phase 2 pathway approved subject to 3 conditions:**
> 1. **LE-02 clearance** — legal counsel must specifically assess expanded liability of shareable static coordinate artefact (incl. stale-data risk framing in disclaimer, litigation use of downloaded files, on-screen disclaimer copy adequacy with downloaded file)
> 2. **UI treatment** — download must include explicit static-data disclosure at point of export (e.g. "These coordinates were recorded at [timestamp]. They do not update.")
> 3. **FSD-5126 §4.4.5 + ESF-5026 §5.6 formally amended** before Phase 2 implementation commences

### §5.7 User Responsibility & Environmental Awareness
trackaroo® does NOT:
- Guarantee rescue or responder response
- Guarantee responder visibility of user's location
- Guarantee signal availability

> Users retain full responsibility for communication planning + environmental awareness before/during remote travel. Responsibility model reflected in onboarding, in-app copy, Terms of Service. **Primary legal framing** for product's non-dispatch posture.

## §6 SOS Liability Boundary & Legal Review Gate

### §6.1 Liability Boundary Declaration
> trackaroo® is a **decision-support tool**. NOT an emergency communication device, personal locator beacon (PLB), or satellite messenger. SOS function logs distress locally + supports user in managing their own emergency response. Does NOT activate, notify, or contact any emergency service.

Boundary reflected in:
- Terms of Service — explicit non-dispatch declaration
- In-app SOS onboarding flow — user acknowledgement required before SOS function accessible
- App Store listing copy — no language implying emergency dispatch capability
- All SOS-adjacent marketing copy

### §6.2 RT-16 — Legal Review Gate

> **RELEASE GATE — RT-16: ESF-5026 Legal Review — Beta-Ready Gate Condition**
>
> SOS liability boundary declaration, user responsibility model, and non-dispatch posture defined in this document must be reviewed + approved by qualified legal counsel before Beta-Ready MVP gate (30 June 2026).
>
> **Required legal review outputs:**
> - Written opinion that non-dispatch posture + SOS copy discipline are legally defensible under Australian Consumer Law
> - Review of Terms of Service SOS liability clause
> - Review of in-app SOS onboarding acknowledgement language
> - Confirmation product does NOT constitute medical device or emergency communication device under applicable Australian legislation
>
> Clearance requires **written sign-off from qualified legal counsel**. Founder & Project Director sign-off alone does NOT satisfy this gate condition.
>
> Authority: PRD-5126 (RT-16), VGD-5126 §8.1.

### §6.3 SOS Onboarding Acknowledgement
Before SOS function accessible to any user (incl. alpha testers), onboarding flow must include explicit acknowledgement user has read + understood:
- trackaroo® does NOT contact emergency services
- SOS activation logs distress **locally only**
- Users retain responsibility for their own emergency communication planning
- Product is NOT substitute for registered PLB or satellite messenger in high-risk environments

> Must be **click-through**, NOT scroll-past. **Mandatory pre-Alpha requirement** — no user exposure to SOS function permitted without this acknowledgement.

## §7 Satellite Boundary Governance

### §7.1 Absolute Satellite Prohibition — Phase 1

> **CONSTRAINT:** Phase 1 includes **ZERO satellite capability**.
>
> Prohibited Phase 1 in any form — active, dormant, commented, feature-flagged, or conditionally activated:
> - Satellite SDK presence (linked or dormant)
> - Stubbed satellite interface classes
> - Feature-flagged satellite modules
> - Commented satellite code
> - Conditional compilation branches targeting satellite
> - Satellite transmission of any kind (offline, online, debug, beta, GA state)
> - Satellite fallback messaging
> - Beacon activation
> - Emergency dispatch via satellite
> - Hybrid terrestrial/satellite switching
> - Background satellite signal detection
> - Satellite signal iconography implying transmission capability
> - Satellite-specific data model fields
>
> Dormant presence constitutes non-compliance. Detection = automatic rejection trigger.

### §7.2 No Dormant SDK / No Latent Pathway Rule
Following constitute automatic rejection under VGD-5126:
- Satellite SDK present in dependency tree (linked or inactive)
- Stubbed satellite interface classes
- Feature-flagged satellite modules
- Commented satellite code
- Conditional compilation branches
- Satellite-specific schema fields in any data model

### §7.3 No Hybrid Switching
No automatic or user-initiated switching between terrestrial + satellite networks permitted. TrackMate™ transport architecture (BLE Mesh, Wi-Fi Direct, LoRa peripheral) operates entirely without satellite dependency. CAL flag `satReady` must be declared in schema but **set to false in all Phase 1 builds + not triggerable**. Authority: FSD-5126 §6.1.2.

### §7.4 No Background Signal Detection
System shall NOT detect, display, or react to satellite signal availability. No background service may scan for satellite networks. No iconography may imply satellite capability.

### §7.5 Satellite Readiness — Structural Definition
"Readiness" in Phase 1 means:
- Architectural separation maintained
- Data model not polluted with satellite logic
- No execution pathway introduced
- No conditional transport layer present

Does NOT mean partial integration, simulation, or conditional activation.

### §7.6 Procurement & Regulatory Protection
trackaroo® Phase 1 is NOT a satellite emergency device. This document exists to:
- Prevent misrepresentation of satellite capability
- Prevent vendor marketing drift
- Prevent legal exposure from implied satellite emergency capability
- Preserve emergency non-dispatch posture clarity

### §7.7 Phase 2 Satellite Activation Control Boundary
Satellite capability, if introduced in future phases, requires ALL of:
- New versioned authority document
- PRD-5126 revision · FSD-5126 revision · TQP-5026 validation update · VGD-5126 enforcement update
- Explicit procurement update
- Written Founder & Project Director approval

> No implicit activation permitted. Phase 2 Escrow pathway must remain **structurally clear in Phase 1** — not scaffolded, not hooked, not referenced as activatable.

## §8 Data Model Neutrality

Phase 1 data models may include abstract capability flags provided they:
- Do NOT name any transport medium by name
- Do NOT imply escalation capability
- Remain null or false in Phase 1
- Do NOT alter runtime execution

**Illustrative acceptable neutral fields:** `externalChannelEnabled: false` · `transportCapabilityLevel: 0` · `escalationMode: none` · `externalInterfaceState: inactive`

> **CONSTRAINT — Prohibited satellite-specific field names in Phase 1 data models:**
> - `satReady` (may exist in **CAL schema only**, set to false — NOT in application data models)
> - `satelliteEndpoint` · `satTransmissionStatus` · `satellitePayload` · `transmission_status` · `satellite_endpoint` · `dispatch_channel` · `fallback_mode` · `connectivity_priority`
>
> Transport-specific nomenclature reserved exclusively for Phase 2 authority documentation. Authority: FSD-5126 §4.4.4.

## §10 Phase Boundary & Deployment Logic

### §10.1 Phase Applicability
ESF-5026 applies exclusively to Phase 1. No clause authorises satellite transmission, hybrid routing, emergency dispatch, or telemetry intelligence in Phase 1. No Phase 2 or Phase 3 capability may be embedded, scaffolded, or implied in any Phase 1 build artefact.

### §10.2 Gate Criteria Alignment

| Gate | ESF-5026 Enforcement Focus |
|---|---|
| **Discovery** | Architecture documentation confirms satellite-free dependency tree; CAL satReady declared + set false; no transmission endpoint present |
| **Alpha (22 Aug 2026)** | SOS ≤2 tap verified; local log confirmed; distress breadcrumb mode confirmed; SOS onboarding acknowledgement in place; SOS feedback elements (all 6) verified; no satellite SDK in dependency tree; SOS copy discipline verified |
| **Beta-Ready (30 Oct 2026)** | **RT-16 legal review complete** (written sign-off from qualified legal counsel); full rejection trigger pass; written QA clearance from Founder & Project Director required (TQP-5026); Founder & Project Director sign-off required |
| **GA / Release (13 Nov 2026)** | All open rejection triggers resolved; final compliance confirmation; written QA clearance from Founder & Project Director required; Founder & Project Director sign-off required |

> **NOTE:** Gate criteria previously referenced Abante Consulting as QA clearance authority. As of 4 March 2026, Abante Consulting removed from project. **Founder & Project Director is sole gate and QA authority** for all Phase 1 delivery gates.

### §10.3 Rollback Governance

> **RELEASE GATE — Release progression halts immediately upon detection of any of the following:**
> - Transmission pathway (active or dormant)
> - Satellite SDK presence in dependency tree
> - Escalation logic of any kind
> - Monitoring posture implied in any UI state or copy
> - Transport-specific schema fields in application data models
> - SOS copy implying dispatch, rescue, or responder notification
> - Distress Breadcrumb Mode absent or gated behind a paywall
> - SOS feedback screen missing any of 6 required elements (PRD-5126 §4.1.6)
>
> Remediation of specific non-compliance required. Full prohibited capability re-scan mandatory after remediation. Revalidation under TQP-5026 required. Written clearance from Founder & Project Director required to re-enter validation.
