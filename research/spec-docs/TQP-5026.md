## §2 Validation Philosophy

Phase 1 validation must prove following properties hold **unconditionally** across all test conditions:
- **Deterministic execution** — identical inputs always produce identical outputs with no variance
- **Offline-first supremacy** — all Survival Core functions operate fully without any connectivity dependency
- **Survival Core isolation** — no mutation, contamination, or override from Experience & Intelligence Layer
- **Zero escalation logic** — no pathway exists for device to escalate to external services
- **Zero transmission capability** — SOS + all Survival Core paths generate zero outbound packets
- **Zero telemetry-weighted behaviour** — no adaptive or density-strengthened classification logic
- **Zero background monitoring posture** — no polling engine, background process, or monitoring loop

> **CONSTRAINT:** Testing must assume **hostile conditions**. Device presumed in environment with no connectivity, degraded battery, potential concurrent stress across all Survival Core functions. **No test scenario may assume favourable conditions for any Survival Core path.**

### §5.7 First Aid Reference

> Scenarios 5.7.1–5.7.2 validate offline execution + non-escalation. Scenarios 5.7.3–5.7.7 validate approved First Aid Reference requirements (PRD-5126, FSD-5126, WFD-5126, VGD-5126, CDG-5126) — incl. tier access, mandatory disclaimer, diagnostic prohibition, content gating, SOS isolation. Internal authority origin maintained by Trackaroo Systems through FRM-5126.

**Scenario 5.7.1 — Timer Activation Offline:** Activate FAR timer. Device offline. → Countdown continues uninterrupted. No monitoring logic. Local alert only on completion. **Fail:** Background health logic detected. External notification attempt. Timer failure offline.

**Scenario 5.7.2 — No Escalation on Timer Completion:** Allow FAR timer to complete. → Local alert only. No external prompt, escalation, transmission. **Fail:** Emergency prompt escalation. Any outbound communication on timer completion.

**Scenario 5.7.3 — Universal Baseline Tier Access:** Access FAR as Free-tier user. Navigate to all Universal Baseline content (DRSABCD, primary patient survey, bleeding/shock/burns/fracture/spinal/choking). → All content accessible without auth prompt, paywall gate, tier upgrade prompt. Fully offline. **Fail:** Any UB content blocked by tier gate, paywall prompt, upgrade prompt. Any UB content requiring connectivity.

**Scenario 5.7.4 — Mandatory Disclaimer Persistence:** Navigate to every screen within FAR system across all content tiers. → Mandatory disclaimer present + fully visible on every FAR screen without scrolling. Disclaimer not hidden, collapsed, or displayed on first use only. **Fail:** Any FAR screen where disclaimer absent, hidden behind scroll, collapsed, or first-use only. Any paraphrase or shortened version of approved disclaimer text.

**Scenario 5.7.5 — No Diagnostic Output:** Interact with all FAR content screens across all tiers. → Zero diagnostic inference, symptom classification, adaptive output, AI-generated clinical response. All content delivery fully deterministic + static. **Fail:** Any screen generating diagnostic inference, symptom classification, adaptive response, or AI-generated output for any user input. Any deviation from fully deterministic content delivery.

**Scenario 5.7.6 — Clinical Depth Content Gating:** As Free-tier user, attempt to access Plus-tier Clinical Depth content (extended wound mgmt, anaphylaxis, hypothermia, heat exhaustion, snake bite, spinal injury, triage timer, offline event log, activity-specific extensions). → All Clinical Depth content appropriately gated at Plus tier. Upgrade prompt displayed using compliant framing. **Fail:** Any Clinical Depth content accessible without Plus subscription. Any upgrade prompt using fear-based language, implying user's safety at risk due to tier, or appearing during active UB reference session.

**Scenario 5.7.7 — SOS Pathway Isolation from FAR:** Navigate all screens within FAR system. Inspect all interactive elements/links/nav paths for any pathway that triggers or initiates SOS execution. → No SOS pathway triggered, initiated, or implied from any FAR screen. FAR + SOS execution remain fully isolated. **Fail:** Any FAR screen containing control, link, or navigation path that triggers SOS execution or enters SOS flow. Any co-mingling of FAR + SOS execution paths.

> **CONSTRAINT:** Scenarios 5.7.3–5.7.7 = non-negotiable release conditions validating FAR implementation. Failure of any in this group = SEV-1 defect. No release build may proceed to any gate without all 5 scenarios passing. **Clinical review sign-off (PRD-5126 RT-12) must be confirmed before execution of Scenarios 5.7.4 + 5.7.5.**

### §5.9 Overlay Semantics & PCR Framework Validation

> All scenarios in §5.9 validate PCR framework + overlay state matrix defined in OSM-5026. **OSM-5026 = sole authority** for all overlay visual semantics, PCR categories, supersession model, valid state combinations. All scenarios executed in both online + offline modes where specified.

**Scenarios 5.9.1 – 5.9.6 — PCR Category Submissions** (PCR-OBS, PCR-CLO, PCR-INF, PCR-SRF, PCR-WTR, PCR-HAZ):
- Submit category report with point location. Device online.
- Expected: Marker displayed at submitted point. Confirmation message ("Condition reported. Thank you."). **No scoring, social signal, additional feedback.** Difficulty grade + shield state unchanged.
- Fail: Marker absent. Feedback beyond confirmation. Any mutation of difficulty grade or shield state.

**Scenario 5.9.7 — No Automatic PCR Trigger:** User stops on track for 60 seconds, deviates from track line. Monitor for any automatic PCR submission. → Zero PCR submissions generated. No prompt displayed. PCR submission requires **explicit user-initiated action only**. **Fail:** Any PCR generated automatically. Any submission prompt displayed without explicit user action. Stop/pause/deviation-detection triggering PCR flow.

**Scenario 5.9.8 — Offline PCR Submission & Queue:** Device in airplane mode. Submit PCR (any category). → PCR queued locally with offline indicator (PCR-WF-04). Confirmation: "Condition reported. Thank you." On connectivity restoration, PCR syncs to cloud. **Submission timestamp preserved from time of submission, NOT time of sync.** **Fail:** PCR silently discarded. Submission rejected. No offline queue state. Any error message implying report failed. Timestamp overwritten with sync time.

**Scenario 5.9.9 — PCR Supersession by Clearance:** Active PCR present on track. Second user submits report at same point confirming condition cleared. → Original PCR marked resolved. Removed from active map. Accessible in track detail/history view only. Resolution timestamp recorded. **Fail:** Original remains active. Absent from history view. Any TTL-based expiry instead of supersession event.

**Scenario 5.9.10 — PCR Supersession by New Report:** Active PCR on track. Second user submits new PCR at same point. → New PCR active on map. Prior PCR archived (NOT deleted) — accessible in history view. Prior no longer active on map. No TTL logic invoked. **Fail:** Prior deleted rather than archived. Prior remains active alongside new. Any TTL mechanism applied.

**Scenario 5.9.11 — Unconfirmed-Age PCR Display:** Advance device clock 91 days beyond PCR submission date. PCR not superseded. → PCR displays with **muted icon treatment** + "Condition age unknown" label (OSM-5026 §10.4). PCR remains active — not removed, not expired, not suppressed. **No countdown timer visible.** **Fail:** PCR removed or suppressed at 90-day mark. Display state unchanged. Any countdown timer. Any automated expiry.

**Scenario 5.9.12 — Phase 1 Valid Overlay State Matrix** (8 sub-scenarios — every valid combination defined in OSM-5026 §13):

| State 6 | Known difficulty. Verification >14 days. Active PCR | Difficulty colour + Grey Shield + PCR marker | Gold Shield. PCR marker absent. Any invalid combo |

> **CONSTRAINT:** Scenarios 5.9.1–5.9.12 mandated by OSM-5026 §14. All PCR category scenarios, submission flow, offline queue, supersession model, all 8 valid overlay state combinations must pass before release progression. Failure of any in §5.9 = SEV-1 defect. **OSM-5026 = sole authority for expected results.**

### §5.10 Overlay Rollback Governance Detection (RG-01 – RG-11)

11 rollback trigger codes defined in OSM-5026 §12. Detection of any trigger at any gate (Alpha, Beta, GA) halts release progression immediately. All 11 must be evaluated at each gate. Clear result against all 11 = mandatory condition for gate progression.

| Code | Condition | Detection |
|---|---|---|
| RG-01 | Adaptive freshness window — TTL thresholds vary at runtime | Static + runtime test |
| RG-02 | Telemetry influence on overlay state — usage/location/behavioural data alters overlay output | Static + data flow audit |
| RG-03 | Density weighting — volume of user reports weights or modifies shield state or PCR display | Static + input variation test |
| RG-04 | Composite trust logic — any scoring/ranking/trust tier applied to verification or PCR | Static analysis |
| RG-05 | Countdown timers visible to users in any overlay context | UI review all screen states |
| RG-06 | Background recalculation engine — recalculates overlay state without explicit trigger event | Static + background process audit |
| RG-07 | Blended visual semantics — difficulty + freshness combined in single visual element | UI review all screen states |
| RG-08 | Derived freshness field — any field other than verification_timestamp used to calculate shield state | Data model audit |
| RG-09 | Automatic verification trigger — verification_timestamp updated without explicit user action | Static + integration test |
| RG-10 | PCR triggered automatically — PCR submitted without explicit user action | Static + integration test |
| RG-11 | Dormant prohibited logic present in codebase — even if feature-flagged or non-executing | Static analysis of full codebase |

> **CONSTRAINT:** All 11 rollback triggers (RG-01 through RG-11) must return clear result at every release gate. Detection of any trigger condition = automatic release halt. Remediation + full revalidation of Overlay Semantics & PCR Framework domain required before gate re-entry. Written clearance from PD required before resumption.

## §7 Prohibited Capability Detection

Automated static + runtime inspection must execute to confirm complete absence of all prohibited artefacts. Inspection covers both compiled application binary + all runtime network activity. **Zero-tolerance detection:**

| Prohibited Artefact Category | Detection Method | Authority Ref |
|---|---|---|
| Satellite SDKs (Iridium, Garmin inReach, SPOT, or equivalent) | Static binary analysis | FSD-5126 |
| Escalation endpoints or transmission-capable service schemas | Static + runtime | FSD-5126 |
| Telemetry weighting logic or density-weighted classification engines | Static + code review | FSD-5126 · UXS-5726 |
| Background polling engines or continuous monitoring loops | Runtime analysis | FSD-5126 · PRD-5126 |
| Hazard reconciliation engines (adaptive TTL or decay logic) | Static + behavioural test | FSD-5126 |
| Any transport-specific schema fields or outbound protocol artefacts | Static + runtime | FSD-5126 |

> **CONSTRAINT:** Detection of any prohibited artefact in any category = **automatic rejection** of release build. No partial acceptance. Remediation + full prohibited capability re-scan + written clearance from PD required before build may re-enter validation.

## §8 Automatic Release Halt Conditions

Halt conditions cause **immediate cessation of all release progression**. Upon declaration: testing stops, build quarantined, delivery partner must notify PD immediately. Remediation + full retest of affected domain required.

| # | Halt Condition — Release Stops Immediately | Authority |
|---|---|---|
| **H1** | Any outbound SOS transmission detected during SOS validation | FSD-5126 · UXS-5726 |
| **H2** | Any satellite SDK detected in static or runtime inspection | FSD-5126 |
| **H3** | Any telemetry-weighted behavioural mutation identified | FSD-5126 · UXS-5726 |
| **H4** | Any routing mutation triggered by hazard data (reroute or breadcrumb alteration) | FSD-5126 |
| **H5** | Any background monitoring engine or polling loop detected | FSD-5126 · PRD-5126 |
| **H6** | Any Survival Core mutation caused by E&I Layer interaction | UXS-5726 |
| **H7** | Battery consumption exceeds PRD-5126 benchmark threshold during profiling | PRD-5126 |

> **CONSTRAINT:** No halt condition may be waived, deferred, or accepted-as-is. Every halt condition must be **remediated to zero** before release progression resumes. Written clearance from PD required for each closed halt condition.
