## §4 Technical Constraints & Prohibitions Enforcement

### §4.1 Architectural Guardrails
Vendors must preserve throughout all phases of delivery:
- **Offline-first execution** — Survival Core must function completely without any network dependency
- **Event-triggered state transitions only** — no periodic, decay-based, or timer-based state changes
- **Deterministic processing** — identical inputs must always produce identical outputs
- No periodic decay loops
- No background recalculation engines
- No telemetry-driven state mutation
- **PCR resolution must be supersession-based only** — no TTL logic may be applied to PCR entries (OSM-5026 §10.3)

### §4.2 Prohibited Capabilities (Active or Dormant)

> **ABSOLUTE PROHIBITION:** Following capabilities prohibited in Phase 1 in any form — active, dormant, commented, feature-flagged, or conditionally activated. **Dormant presence constitutes non-compliance. Detection = automatic rejection trigger.**

- AI or inference engines of any kind
- **Satellite SDKs** (Phase 2 — Escrow pathway must be structurally clear, not implemented)
- Mesh networking libraries outside approved TrackMate™ transport stack
- Autonomous escalation logic
- Third-party emergency dispatch integration
- Trusted Circle activation
- Emergency Data Escrow transmission
- Telemetry weighting engines
- Density modelling engines
- Confidence scoring engines
- Gamification activation modules
- Intelligent routing engines
- Dynamic overlay recalculation engines
- Background beaconing services (excluding approved TrackMate™ BLE Mesh background)
- Latent Phase 2 or Phase 3 scaffolding
- Feature-flagged future-phase logic
- Adaptive bandwidth logic or telemetry-based transport weighting in TrackMate™
- **TTL-based expiry of PCR entries** — PCR resolution is supersession-based only (OSM-5026 §10.3)

### §4.3 Dormant Capability Enforcement
**Dormant presence constitutes non-compliance.** Following = active violations:
- Inactive SDKs present in dependency tree
- Commented-out logic containing prohibited capability
- Feature-flagged modules targeting Phase 2+ functionality
- Conditional activation pathways that could enable prohibited capabilities

> **PLACEHOLDER DISCIPLINE:** Phase 2 architectural hooks permitted in Phase 1 builds **only** where explicitly specified in FSD-5126. All Phase 2 placeholders must: (a) be visually surfaced to QA, (b) display exactly the text "Inactive in Phase 1.", (c) be schema-complete, (d) contain zero executable logic.

## §5 TrackMate™ — Transport Architecture & Vendor Obligations

TrackMate™ = peer-to-peer group comms + location module. Full Phase 1 deliverable.

### §5.1 Capability Goals (Governing)

| Capability Goal | Requirement |
|---|---|
| Short-range P2P comms | P2P messaging + location sharing without any network infrastructure dependency |
| Long-range comms | Convoy members separated by significant distance (may require optional LoRa hardware) |
| Battery sustainability | TrackMate™ must NOT materially compromise ≥10-hour device operation (BPS-5126 governing) |
| Fully offline | Zero cellular or internet dependency for any TrackMate™ function under any operating condition |
| **Transport transparency** | **No manual transport selection required from user — system manages transport internally** |
| Group constraints | Max 8 active members (hard cap); invite-only groups; role-based membership |
| Offline message queue | Messages queued locally when peer unreachable; delivered on reconnection |

### §5.2 Reference Transport Stack (informed reference, NOT closed spec)

| Tier | Mechanism | Indicative Range | Battery Profile | Dependency |
|---|---|---|---|---|
| **Tier 1 — Primary** | BLE Mesh | ~100m device-to-device; extended via multi-hop | Very low — continuous background | Device-native |
| **Tier 1 — Fallback** | Wi-Fi Direct | ~200m line-of-sight | Medium — on-demand only; **must NOT be held continuously** | Device-native |
| **Tier 2 — Long Range** | LoRa (optional peripheral) | 5–15km open terrain | Very low | GoTenna/Meshtastic-class hardware; auto-detected if present |
| **Phase 2 Only** | Satellite relay | Global | TBD | Via BackTrack™ Emergency Escrow pathway — **Phase 2 only** |

### §5.3 Vendor Engagement Rule
Vendors must provide written technical response to reference transport stack. Response must:
- Confirm, refine, or propose alternative with full technical justification
- Demonstrate all capability goals in §5.1 satisfied
- Demonstrate compliance with BPS-5126 battery constraints
- Confirm CAL implementation satisfies required state flags

**Required CAL state flags — all 4 must be implemented + tested:**

| CAL Flag | Phase 1 Status | Definition |
|---|---|---|
| **satReady** | Declared — NOT active | Boolean. Declared in Phase 1 schema only. Must be set to false in all Phase 1 builds. Must NOT be reachable or triggerable. Signals satellite transport pathway availability — Phase 2 activation only (FSD-5126 §6.1.2) |
| **queueEnabled** | ACTIVE | Boolean. True whenever TrackMate™ session active. Indicates offline message queue accepting outbound messages |
| **offlineBeacon** | ACTIVE | Boolean. True when BLE Mesh active. Signals device broadcasting presence via available offline transport |
| **partialSignal** | ACTIVE | Boolean. Signals limited/intermittent connectivity. True when network interface active but signal quality or packet success rate below deterministic threshold. **No automated recovery action or escalation may be triggered by this flag.** UI indicator must be calm, non-alarming, must NOT imply any action pending or in progress. Maps to Limited Connectivity UI state in WFD-5126 §6 |

### §5.4 LoRa Hardware Onboarding Requirement
Hardware dependency for long-range capability must be **explicitly disclosed to users at first use**. UI must NEVER imply long-range comm capability to user with no LoRa hardware connected.

**4 wireframe states required in WFD-5126:**

| State ID | State Name | Description |
|---|---|---|
| TM-WF-01 | First-use onboarding | Hardware expectation + range context disclosure — LoRa hardware dependency stated explicitly |
| TM-WF-02 | Short-range only mode | Operating without LoRa — UI must NOT imply long-range capability |
| TM-WF-03 | LoRa hardware detected + active | Hardware connected — long-range capability available + accurately represented |
| TM-WF-04 | LoRa hardware disconnected mid-session | Hardware removed during active session — UI transitions to short-range mode; no silent capability degradation |

> **CONSTRAINT:** LoRa onboarding disclosure must accurately represent range limitations + hardware dependencies. States TM-WF-01 + TM-WF-02 must NOT imply long-range capability in absence of connected LoRa hardware. Omission = **Rejection Trigger RT-17**.

### §5.5 TrackMate™ Prohibitions
- No adaptive bandwidth logic or telemetry-based transport weighting
- No monitoring posture — explicit or implied — in any TrackMate™ UI state or copy
- No automated distress triggering from Group Health Envelope
- No transport mechanism may introduce a network dependency into any Survival Core execution path
- Wi-Fi Direct must NOT be held continuously active — on-demand activation only
- **Group Health Envelope is a visual indicator only — must NOT be presented as monitoring system**
- Satellite relay must NOT be implemented or scaffolded in Phase 1

### §5.6 SOS Copy & Feedback Obligations

> **SOS COPY RULE — NON-NEGOTIABLE:** SOS user feedback must NEVER imply help has been dispatched, rescue underway, or any emergency service notified.
>
> Applies to: SOS activation confirmation screens · all SOS feedback messages · post-activation state copy · any user-facing text in SOS flow.
>
> **Permitted framing:** Calm, factual acknowledgement that SOS event has been logged locally.
> **Prohibited framing:** Any language implying dispatch, rescue activation, emergency services notification, or imminent response.
>
> Authority: UXS-5726 §7.3. Violation = automatic rejection trigger — **RT-06** (monitoring posture implied).

> **CONSTRAINT:** SOS-triggered distress breadcrumb mode (5m/5s) = **Survival Core function**. Activates immediately upon SOS confirmation for ALL users on ALL tiers — including Free. **No tier boundary may restrict, delay, or degrade any SOS-triggered behaviour.** Any implementation applying tier restriction to SOS-triggered distress mode = named rejection trigger **RT-22** + halts release progression.
>
> Tier-gated feature = manual distress density recording — user's ability to enable elevated breadcrumb capture density without SOS activation. Experience & Intelligence Layer control (Plus + Pro tiers only). **Distinct features + must NOT be conflated** in any architecture or implementation decision.
>
> Authority: FSD-5126 §4.2.2, BTF-5126 §6.1.4, CR04 Q8 (1 April 2026).

## §6 PCR Framework Enforcement

### §6.1 PCR Definition & Governing Authority
**OSM-5026 = sole + exclusive authority** for PCR framework: categories, resolution model, visual treatment, submission UI requirements. VGD-5126 enforces vendor compliance with those rules only.

> PCR entries represent **discrete physical-state events** at a point on a track (e.g. Bridge Out, Track Damage, Surface Change). Resolved by **supersession** — a new PCR at the same point + category supersedes the prior one. **No TTL logic may be applied to PCR entries** (OSM-5026 §10.3).

### §6.2 PCR Vendor Obligations
Vendors must:
- Implement PCR data classification in full conformance with **CDG-5126 §5** (PCR Data Governance)
- Implement **supersession-based resolution model** defined in OSM-5026 §10.3 — **no TTL logic permitted**
- Implement PCR visual treatment in full conformance with OSM-5026 §10.4
- Implement PCR submission UI in full conformance with OSM-5026 §10.5
- Implement all PCR wireframe states defined in WFD-5126 (see OSM-5026 §10.6 for required states)
- Confirm PCR immutable submission history — prior PCR archived on supersession, **NOT deleted**
- Confirm PCR independence from all TTL layers — no TTL logic may be applied to PCR entries under any condition
- Confirm PCR excluded from Firebase-dependent sync paths in conformance with CDG-5126 §5.3

### §6.3 PCR Artefact Requirements at Gate
Vendors must supply at each relevant gate:
- PCR data schema documentation confirming classification per CDG-5126 §5.6
- PCR resolution logic documentation confirming supersession model — no TTL logic
- PCR visual treatment implementation evidence against OSM-5026 §10.4
- PCR submission UI evidence against OSM-5026 §10.5
- Confirmation that prior PCR records archived (NOT deleted) on supersession

## §8 Breadcrumb Forensic Integrity Enforcement

### §8.1 Local-Only and Non-Syncable Classification
Breadcrumb data classification = **Local-Only — Non-Syncable** under CDG-5126. Must NEVER leave device, must NEVER be uploaded/synced/transmitted/mirrored to any cloud service.

### §8.2 Prohibited Mutation Behaviours (14 named operations)
All operations **absolutely prohibited** on breadcrumb data:

`Merge · Compaction · Reconstruction · Reordering · Map-matching · Coordinate interpolation · Gap-filling · Deduplication · Compression · Conflict resolution · Reconciliation · Coalescing · Smoothing · Correction`

> Static analysis must scan for every named item. Absence of all 14 confirmed before any release gate.

### §8.3 Deterministic Reverse-Order Preservation
BackTrack™ retrace must:
- Reverse breadcrumbs strictly in stored order
- Use recorded coordinates only — without transformation
- Render complete trail within ≤3s
- Display chronologically (oldest to most recent)
- Function fully offline

### §8.4 Integrity Over Optimisation Principle
> No performance optimisation, storage efficiency objective, UX improvement goal, or vendor implementation preference may justify any mutation of breadcrumb data. If optimisation requires mutation, **the optimisation must be abandoned**. Absolute.
