## Role in Authority Stack

**Priority 8** (Tier 2 specialist). Subordinate to UXS-5726 / PRD-5126 / FSD-5126 / TQP-5026 / WFD-5126 / VGD-5126. References OSM-5026 (sole authority for overlay visual semantics, TTL thresholds, PCR framework).

> **AUTHORITY RESTRICTION:** HazTrack™ may only ingest feeds that pass all 5 pillars of the HFG-5026 inclusion filter. A feed that fails any single pillar is ineligible regardless of data quality, coverage, or commercial availability. **Restriction is absolute and may NOT be relaxed by vendor proposal or delivery decision.**

HFG-5026 governs which hazard data feeds may be ingested by HazTrack™. Updated to align with OSM-5026 (20 March 2026, which introduced PCR framework + calibrated multi-type TTL model). All TTL threshold references cite **OSM-5026 §8 as sole authority**. No independent threshold values declared in HFG-5026.

### Relationship to FSD-5126
> FSD-5126 §4.3 declares HazTrack™ ingestion sources + references OSM-5026 for TTL threshold authority. HFG-5026 = governing authority for feed eligibility. OSM-5026 = sole authority for overlay visual semantics + TTL values + PCR framework. In conflict between FSD §4.3 and HFG-5026 → HFG-5026 prevails within its domain. In conflict between HFG-5026 and OSM-5026 regarding freshness display or TTL values → OSM-5026 prevails.

## §3 The Five-Pillar Inclusion Filter

Every candidate hazard feed must satisfy ALL 5 pillars before HazTrack™ ingestion. Pillars evaluated sequentially. Failure at any pillar ends evaluation — feed = ineligible.

| Pillar | Name | Requirement | Failure Condition |
|---|---|---|---|
| 1 | **Authority Origin** | Feed must originate from recognised statutory, government, or emergency management authority. **No commercial, aggregated, crowdsourced, or user-generated feed may be admitted.** | Feed originates from non-government or non-statutory source — automatic ineligibility |
| 2 | **Australian Jurisdiction** | Feed must cover Australian geographic territory under jurisdiction of an eligible Australian authority body (Commonwealth, state, or territory) | Feed covers only non-Australian territory or operated by non-Australian authority without formal Australian jurisdiction mandate |
| 3 | **Machine-Readable Format** | Machine-readable, structured format. Accepted: **GeoJSON, CAP** (Common Alerting Protocol), **WFS**, or equivalent open standard. PDF, HTML scrape, unstructured = ineligible | Available only in PDF/HTML/unstructured requiring parsing beyond standard open processing |
| 4 | **Deterministic Compatibility** | Update cadence, data schema, delivery mechanism must be compatible with trackaroo®'s deterministic offline-first architecture. Feed must NOT require persistent polling/real-time streaming/continuous connection as functional dependency | Feed requires continuous connection, real-time socket dependency, or persistent polling to function correctly |
| 5 | **Stability & SLA** | Feed must have published or formally confirmed availability commitment (uptime target, SLA, or equivalent). Experimental, beta, or unstable feeds = ineligible Phase 1 | No published availability commitment or classified as experimental/beta by provider |

## §5 Absolute Prohibitions

### §5.1 Aggregator Prohibition (absolute, non-waivable)
> HazTrack™ must NOT ingest data from any aggregator service — service that consolidates/normalises/repackages/redistributes data originally sourced from one or more authority bodies. Applies regardless of how closely aggregator's output mirrors authority-origin data.
>
> **AGGREGATOR PROHIBITION — ABSOLUTE:** Authority-origin chain must be unbroken. If data passes through any intermediary, normalisation engine, third-party API, or commercial redistribution layer before reaching trackaroo® → it is aggregated and ineligible — regardless of original source. Non-negotiable. May NOT be waived by vendor proposal.

### §5.2 Crowdsourced & User-Generated Data Prohibition
- No crowdsourced hazard report may be ingested by HazTrack™ Phase 1
- No user-generated content (reports, pins, alerts, notes) displayed through HazTrack™ overlay system
- Community reporting features — if developed in future phases — must operate on strictly separate data layer with explicit user disclosure

> **PCR FRAMEWORK — SEPARATE DATA LAYER:** Point Condition Report framework (OSM-5026 §10) provides user-reporting mechanism for discrete physical-state events (fallen tree, locked gate, washed-out crossing). PCRs are distinct data layer governed by OSM-5026. They do NOT constitute crowdsourced hazard data — submitted by authenticated trackaroo® users, operate through separate visual system, resolved by supersession rather than TTL decay. **Crowdsourced prohibition in §5.2 applies to HazTrack™ hazard feed ingestion only.** PCR governance defined exclusively in OSM-5026 §10 + CDG-5126 §5.

### §5.3 Commercial Weather & Hazard API Prohibition
- Commercial weather APIs (e.g. OpenWeatherMap, Weather.com, Tomorrow.io) ineligible — do NOT satisfy Pillar 1 (authority origin) or Pillar 2 (Australian jurisdiction)
- Commercial road condition / traffic APIs ineligible for same reason
- Prohibition applies even where commercial API uses BOM or SES data as underlying source — **aggregation chain is broken**

### §5.4 Route Intersection Eligibility Filter
Not all eligible HazTrack™ feeds may trigger route intersection notification (PRD §4.1.2). Feed eligibility for route intersection = separate determination. Only feeds meeting ALL additional criteria may trigger route notification:
- Feed event has defined **geographic boundary (polygon or radius)** — point events ineligible for route intersection
- Feed event type directly relevant to user route safety (fire, flood, severe weather, road closure)
- Feed data within **Gold or Grey Shield freshness window** as defined in OSM-5026 §8 — stale (No Indicator equivalent) MAY NOT trigger route notification

## §6 Deterministic Compatibility Requirements

All eligible feeds must be compatible with trackaroo®'s offline-first deterministic architecture.

| Rule | Requirement |
|---|---|
| No persistent connection dependency | Feed integration must operate via opportunistic polling only · No feed may require persistent socket/streaming/continuous polling interval as functional dependency |
| Cache-first display | Cached feed data must be displayed when connectivity unavailable · Overlay must NOT be suppressed/emptied/replaced with error state due to feed unavailability |
| Silent failure mode | Feed timeout/error/unavailability must result in **silent staleness indicator update only** · NO alert/error state/notification/user prompt may be generated by feed failure |
| No all-clear implication | Empty overlay (region with no cached data or feed failure) must NEVER be presented as all-clear or safe condition · Absence of hazard data ≠ confirmation of hazard absence |
| TTL compliance | All cached feed data governed by deterministic TTL thresholds defined in **OSM-5026 §8 (sole authority)**. OSM-5026 §8 declares calibrated thresholds per hazard feed type. **TTL may NOT be adapted by feed cadence or connectivity** |
| No adaptive ingestion | Feed ingestion frequency may NOT be weighted/adjusted/prioritised based on telemetry/user behaviour/location density or any signal. **Ingestion is opportunistic + uniform** |

> **TTL THRESHOLD AUTHORITY:** All TTL threshold values for hazard feed freshness display defined exclusively in OSM-5026 §8 (sole authority). HFG-5026 does NOT independently declare threshold values. No implementation may set/override/extend TTL thresholds without formal amendment to OSM-5026 recorded in CAR-5026. Any revision to OSM-5026 §8 threshold values automatically governs HFG-5026 without separate HFG-5026 amendment, provided OSM-5026 updated in CAR-5026.

## §7 PCR (Point Condition Report) Framework — Relationship to HFG-5026

PCR framework introduced in OSM-5026 §10 (20 March 2026). PCRs = user-submitted reports of specific physical-state events at defined points on tracks/trails/paths. **Architecturally + visually distinct from HazTrack™ hazard feed overlays.**

### §7.1 Architectural Separation
- HazTrack™ feed events must NOT alter PCR state · PCR submissions must NOT alter HazTrack™ feed state or freshness indicators
- PCRs use **supersession resolution model** — NOT TTL decay · A PCR remains active until explicitly superseded by subsequent user action
- PCRs use distinct icon + visual system (OSM-5026 §10.4) · Must NEVER use Gold/Grey/No Indicator freshness system applied to hazard feed data
- PCR data governance defined in OSM-5026 §10 + CDG-5126 §5 · HFG-5026 has no authority over PCR governance

### §7.2 Vendor Obligation
Delivery partner must confirm in vendor feed evaluation response (Section 8) that **PCR data layer is implemented as architecturally independent from HazTrack™ feed ingestion**. Shared state, shared visual indicators, or shared resolution logic between PCRs and HazTrack™ feeds = **rejection trigger (PRD §14.4)**.

> **LAYER INDEPENDENCE — MANDATORY:** HazTrack™ feed events + PCR state must remain architecturally independent always. No HazTrack™ feed event may alter PCR display state. No PCR submission may alter HazTrack™ freshness state, overlay display, or feed ingestion behaviour. **Independence requirement non-negotiable + named rejection trigger in PRD-5126.** Authority: OSM-5026 LIR-04 + LIR-05.

## §8 Vendor Feed Evaluation Obligations

Delivery partner must provide formal feed evaluation response for each proposed feed integration:
- Confirm which feeds from Phase 1 reference list (§4) being integrated
- Provide endpoint URLs, schema documentation, update cadence
- Confirm Five-Pillar Filter compliance for each feed, with justification where compliance requires explanation
- Confirm TTL compatibility with OSM-5026 §8 threshold values (sole authority)
- Confirm cache-first offline display implementation approach
- Confirm PCR data layer is architecturally independent from HazTrack™ feed ingestion (Section 7)
- Identify any feeds NOT in Phase 1 reference list being proposed + provide full Five-Pillar evaluation for each

> **LEGAL / LIABILITY REVIEW REQUIRED:** HazTrack™ ingests authority-origin feeds only, displays freshness state transparently per OSM-5026 §8, does NOT imply dispatch/guarantee currency. These architectural decisions substantially mitigate duty-of-care exposure. Residual legal review obligation for Trackaroo Systems covers 2 items: (1) **notification copy + freshness disclosure wording** — confirm language ceiling within posture established; (2) **route intersection consent model** (PRD §4.1.2) — confirm opt-in mechanism sufficient. Both owned + managed by Trackaroo Systems. Neither imposes delivery obligation on vendor. Both = **named Beta-Ready gate conditions (RT-16)**.
