**Priority 7** (Tier 2 specialist). Subordinate to UXS-5726 / PRD-5126 / FSD-5126 / TQP-5026 / WFD-5126 / VGD-5126.

> "BackTrack™ is a retrace mechanism only. NOT a navigation engine, optimisation layer, or intelligence module. No new capability beyond FSD-5126 is introduced by this document."

BackTrack™ is:
- Reverse chronological playback only
- Fully offline-capable
- Deterministic in execution
- Non-optimising · Non-adaptive · Non-inferential

This framework: preserves breadcrumb evidentiary integrity · prevents routing reinterpretation · prevents smoothing/interpolation/correction logic · prevents telemetry-driven mutation · prevents architectural contamination from optimisation engines.

## §3 Scope

### §3.1 In Scope (Phase 1)
- Reverse chronological breadcrumb playback
- Deterministic dual-trigger breadcrumb capture (distance + time)
- Deterministic waypoint sequencing
- Offline retrace capability — fully available without connectivity
- Distress breadcrumb mode — higher-density capture immediately upon SOS activation
- Immutable breadcrumb storage with crash survivability
- Recovery audit event generation on app restart following unclean termination
- Phase 2 Emergency Escrow architectural readiness — non-executable schema scaffold only (§6.2)

### §3.2 Explicitly Out of Scope (Phase 1 — NON-NEGOTIABLE)
> EXHAUSTIVE LIST. Any capability not in §3.1 = out of scope unless explicitly approved in writing by PD.

- Route optimisation of any kind
- Map-matching or path correction
- GPS stabilisation overlays
- Path smoothing or interpolation
- Heuristic interpolation
- Terrain inference
- Hazard-aware rerouting or hazard-aware breadcrumb modification
- Environmental-aware rerouting
- Telemetry-based rerouting / telemetry-based breadcrumb modification
- Density weighting / confidence scoring
- AI inference of any kind / adaptive intelligence (PRD §9)
- Adaptive recalculation
- TrackIQ™ intelligence within BackTrack™ execution
- Gamification logic — scaffolding permitted ONLY if inert + non-executable
- BackTrack™ Emergency Escrow transmission/activation/user-visible artefact
- Satellite transmission / satellite SDK refs within BackTrack™ module
- CarPlay/Android Auto — deferred Phase 2; BackTrack™ unaffected, functions identically on all Phase 1 platforms

> Presence of any prohibited mechanism — even dormant, feature-flagged, SDK-linked — = non-compliance. Static analysis must confirm absence before any release gate.

### §3.3 Five-Question Cognitive Hierarchy — BackTrack™ Compliance Anchor

BackTrack™ = sole system responsible for **Question 4: "How do I get back?"**

| # | Question | BackTrack™ Responsibility | Failure Condition |
|---|---|---|---|
| 4 | How do I get back? | Reconstruct exact reverse trail from locally stored breadcrumb data, fully offline, within ≤3s of user initiation | Breadcrumb loss on crash; retrace unavailable offline; trail load > 3s; BackTrack™ not accessible within ≤3 taps from any nav screen |

> **CONSTRAINT:** BackTrack™ accessible within **≤3 taps from any navigation screen state** without exception (UXS §9.1, WFD §5.3, PRD §4.1.3). Failure = **RT-10**.

Compliance obligations:
- Reconstruct return path from local breadcrumb data only — no network request at any stage
- Return path clearly distinguished from forward route on map at all times
- Functional if breadcrumb trail partial/gapped — display available with clear gap indication
- Must NOT auto-activate — always + only user-initiated
- Must NOT require connectivity for any phase

## §5 Constraints

### §5.1 Technical Constraints & Prohibitions (absolute)
- Event-triggered execution only — no background recalculation engines
- No periodic decay loops
- No adaptive weighting
- No telemetry-driven mutation
- No density modelling
- No server-dependent computation
- All breadcrumb data Firebase-independent — local persistence only Phase 1

### §5.2 Map-Matching & Path Correction Prohibition (absolute)
> Map-matching engines, GPS stabilisation overlays, third-party path correction SDKs, coordinate smoothing libraries = strictly prohibited regardless of dormant/feature-flagged state.

**Complete prohibited mutation taxonomy (authoritative for static analysis scan; authority: VGD-5126 §7.2):**

`Merge · Compaction · Reconstruction · Reordering · Map-matching · Coordinate interpolation · Gap-filling · Deduplication · Compression · Conflict resolution · Reconciliation · Coalescing · Smoothing · Correction`

Static analysis must scan for every named item. Absence of all 14 operations confirmed before any release gate.

### §5.3 Battery & Performance Constraints (defers to BPS-5126)
- Battery consumption — active nav: **≤8%/hr** (BPS-5126 sole authority)
- Offline endurance — active nav: **≥10 hours** (BPS-5126)
- BackTrack™ retrace trail load: **≤3 seconds to full render** (PRD/FSD)

BPS-5126 §3.4 establishes PCR overlay rendering as included component of standardised battery test load. No retrace behaviour may introduce CPU-intensive recomputation, background loops, or persistent wake locks.

### §5.4 WFD-5126 Build Gate
BackTrack™ wireframe states required before development:
- Standard retrace state — trail rendered on offline map
- Partial trail state — gap clearly indicated; no interpolation implied
- Distress mode active state — activated on SOS; no transmission artefacts visible
- Data absent or corrupt state — graceful handling

## §6 Content — Functional Framework

### §6.1.1 Breadcrumb Capture Baseline
Breadcrumbs must be:
- Timestamped — UTC millisecond precision
- Sequentially ordered — strict chronological always
- Stored locally — Firebase-independent
- AES-256 encrypted at rest
- Crash-survivable — write-ahead log or equivalent
- Immutable once committed

**Record fields (6, all mandatory):** Timestamp · Lat/Long WGS84 ≥6 decimal · GNSS accuracy estimate (m) · Altitude (m ASL from GNSS where available — must not be inferred) · Session identifier · Capture mode flag (standard | distress).

### §6.1.2 Dual-Trigger Logic
| Mode | Distance | Time | Activation |
|---|---|---|---|
| Standard | 15 m | 20 s | Active all session |
| Distress (SOS active) | 5 m | 5 s | Immediately on SOS, persists remainder of session |

> **CONSTRAINT (CR04 Q8, 1 April 2026):** SOS-triggered distress mode = Survival Core function, **tier-agnostic** — activates for ALL users on ALL tiers including Free. Tier-gated control = manual distress density recording (Experience Layer, Plus/Pro). Conflating = named rejection trigger. Authority: UXS-5726 · PRD-5126 · FSD-5126 §4.2.2.

> Capture intervals fixed + deterministic. Cannot be dynamically altered by any system component, Experience Layer feature, navigation state, device speed, telemetry signal, runtime condition. Dynamic interval modification = named rejection trigger equivalent to RT-07.

### §6.1.3 Reverse Chronological Playback
BackTrack™ retrace must:
- Reverse breadcrumbs strictly in stored order
- Use recorded coordinates only — without transformation
- Avoid optimisation/correction/recomputation
- Fully offline always
- Render complete trail within ≤3s of user initiation
- Display chronologically (oldest to most recent)
- Overlay on offline map — no connectivity required

Playback must NOT:
- Generate alternate route suggestions
- Snap to mapped roads or any geometry layer
- Recalculate based on terrain/hazard/environmental overlays
- Re-weight breadcrumb significance based on any signal

### §6.1.4 Distress Breadcrumb Mode
When SOS activated:
- No adaptive runtime modulation
- No telemetry-based adjustment
- No background beaconing
- Local-only Phase 1 — NO transmission
- One-way transition within session — does NOT revert to standard mode for remainder
- Distress mode density is pre-defined + static (FSD §4.2.2) — not runtime-adjustable
- Validated under TQP-5026 Scenario 5.2.2
- Must remain within ≤8%/hr battery (BPS-5126)

### §6.1.5 Session Lifecycle Model

| State | Description | Breadcrumb behaviour |
|---|---|---|
| Session Start | New journey · unique session ID assigned | Standard mode (15m/20s) immediately · always starts standard regardless of prior |
| Active — Standard Mode | Nav in progress; SOS not active | 15m/20s · session ID in every record |
| Active — Distress Mode | SOS activated · ONE-WAY transition (cannot revert/cancel within session) | 5m/5s · capture mode flag = distress on all subsequent · does NOT revert |
| Session End (Clean) | User explicitly ends · session closes normally | Capture ceases · data remains locally accessible for retrace |
| Session End (Crash) | App terminates unexpectedly mid-session | All committed breadcrumbs survive via write-ahead log · Recovery audit event on next start |
| Recovery State | App restart following unclean termination | Recovery audit event written to local event log · Prior session data accessible · New session ID for any new journey |

> **CONSTRAINT:** Distress mode = one-way within session. NOT reversible by user/nav state/SOS deactivation/runtime condition. New session always starts standard. Session IDs unique, not reused.

### §6.1.6 Immutable Integrity Guarantees
- Preserve historical breadcrumb sequence exactly — no reordering, duplication, gap
- Resist silent overwrite — no retroactive mutation
- Restore safely after crash — survive unexpected termination
- Generate recovery audit event on restart following unclean termination
- Use WAL or equivalent crash-survivable local storage

### §6.1.7 Non-Interference Rules
BackTrack™ is execution-contained + representational only. Must NOT:
- Influence routing logic
- Influence hazard ingestion (HFG-5026)
- Influence overlay semantics (OSM-5026)
- Influence PCR state/submission flows/visual treatment (OSM-5026 §10; LIR-04; LIR-05)
- Influence verification freshness logic
- Influence SOS execution
- Trigger notifications
- Trigger escalation logic
- Mutate any E&I Layer component state

### §6.1.8 Architectural Isolation Requirement
BackTrack™ execution logic must remain modularly isolated from:
- Navigation Core optimisation logic
- Overlay ingestion modules
- SOS escalation pathways
- Cloud reconciliation engines
- Telemetry aggregation services
- TrackIQ™ scoring/grading logic

> No shared mutable state between BackTrack™ and any module capable of behavioural mutation. Any architectural coupling creating pathway for Experience Layer state to modify BackTrack™ execution = Survival Core contamination event, halts release immediately.

> **RELEASE GATE:** Delivery partner written confirmation that Phase 1 creates no structural barrier to Phase 2 Escrow = **named Beta-Ready gate condition (PRD §9.2)**. Full QA validation sign-off from PD additionally required at Beta-Ready, confirming exit criteria XC1–XC10 (TQP-5026) satisfied.

| Gate | BackTrack™ requirement |

### §6.3.3 Rollback Governance — release halts if detected at any gate:
- Route optimisation logic of any kind
- Map-matching engines or coordinate smoothing

- Dynamic breadcrumb capture interval modification
- Coalescing/compression/deduplication/reconciliation/conflict resolution on breadcrumb
