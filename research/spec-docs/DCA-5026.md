## §7 Project Administration, Records, Reporting

- **Live registers required (continuous PD access):** project plan · risk register · decision register · defect register · dependency register · OSS register · SDK register · **AI-tool register** · change register
- **Weekly written status report** covering: Gate progress · risks · blockers · decisions required · upcoming gate evidence · defects · security events · personnel changes · variation exposure
- **No verbal obligations** — all material decisions documented in writing
- **Record retention** — minimum **5 years from GA**; **7 years** for IP assignment, OSS/AI registers, security incidents, privacy/access records, gate evidence, handover records, legal/clinical clearance mapping

## §8 Repository, Source Code, Build, Credential Control

- **§8.1 Repository control:** Client-owned OR Client must have continuous unrestricted admin access. Contractor NEVER holds the only functional copy of source/build/config. Established **before Discovery Gate Clearance**.
- **§8.2 Branching/commit/build discipline:** named branches · meaningful commits · PR review · release tagging · reproducible builds · env segregation (prod/staging/test/dev)
- **§8.3 Credentials & Secrets:** controlled credentials register · secure secret-management · no plaintext · no commit to source · no individual-only holders
- **§8.4 Platform account ownership:** Apple App Store, Google Play, Mapbox, Firebase, domain, hosting, CMS, analytics — all **Client-owned**, vendor access **role-based + time-limited + immediately revocable**
- **§8.5 Build continuity:** At each Gate, updated build instructions enabling a competent team to reproduce/test/deploy without Contractor-specific tacit knowledge

## §9 Security, Privacy, Data Governance

- **§9.1 Secure SDL** — threat modelling · dependency scanning · static analysis · runtime testing · secure code review · prohibited-capability scans **at each Gate**
- **§9.2 Access Control** — MFA · least-privilege · named accounts · RBAC · revocation on personnel change · no shared accounts (without PD approval)
- **§9.3 Survival Core data isolation** — Local-Only / Non-Syncable / Firebase-independent / immutable / non-transmission (breadcrumb, SOS event log, local event log) per CDG-5126 + BTF-5126
- **§9.4 Security incident notify** — **within 24 hours** for any actual/suspected breach (description: nature, affected systems, impact, containment)
- **§9.5 Privacy compliance** — APPs · Privacy Act 1988 (Cth)
- **§9.6 Overseas access** — Contractor must not allow access from any country/subcontractor/group not disclosed and pre-approved in writing
- **§9.7 Schedule 9 Annex** — **strict condition precedent to environment access**; must be fully populated + executed before any PII / location / OCS data access

## §10 Intellectual Property

- **§10.1–10.4 Absolute assignment** of all Works + Repository Materials IP, with moral rights consents (per-individual, fully-executed deed = condition precedent to repo access)
- **§10.5 Pre-existing/OSS** — no incorporation without written disclosure + approval; App Store + Play compatibility confirmed
- **§10.6 AI-Generated Code** — no upload of Client Confidential / source / credentials / PII / architecture to AI services without PD prior written approval. Production code with AI assistance → disclose in AI-tool register · human review · security review · licence-clean confirmation
- **§10.7 OSS Licence Audit** — named **Discovery Gate artefact (D9)**; updated each Gate; GPL/AGPL/SSPL needs prior written PD approval
- **§10.8 Subcontractor chain of title** — no less protective IP/moral/AI terms

## §11 Delivery Standards, Quality, Validation

- **Authority Stack precedence** (Schedule 1) — UXS-5726 highest
- **Rejection Triggers (RT-01..23)** — automatic non-compliance, full revalidation before Gate Clearance
- **TQP-5026 validation** — evidence produced each Gate
- **Hard rules called out in §11:**
  - SOS ≤2 taps reachable, ≤1s log write+feedback, distress breadcrumb 5m/5s, **no tier restriction even for Free**
  - BackTrack™ deterministic + fully offline + immutable + Local-Only + Firebase-independent (no mutation/smoothing/map-matching/interpolation/dedup/cloud)
  - OCS never reads Survival Core data (breadcrumb, SOS, event log)
  - **WCAG 2.1 AA mandatory @ Beta-Ready** + independent audit deliverable

## §11A Design Quality Obligation (NEW — premium-consumer standard)

> Functional compliance with Authority Stack is **necessary but not sufficient** for Gate Clearance on any Experience & Intelligence Layer surface. **PD may withhold Gate Clearance on design-quality grounds independently of functional compliance.**

Applies to: TrackIQ™ displays · TrackMate™ group/messaging · First Aid screens · onboarding/archetype selection · map chrome · PCR submission/detail · Safe Anchor screens · POI screens · Settings · CAL status panel.

Survival Core UI (main map, SOS, BackTrack™, HazTrack™) subject to obligation but UXS-5726/OSM-5026/ESF-5026 takes absolute precedence. **No design direction may override a Survival Core behavioural rule.**

**Contest window:** 5 BD written. Standard = premium consumer safety app, **not parity with minimum spec**. OSM-5026 compliance is NOT a defence against RT-23.

## §14 Key Personnel & Subcontracting

- **§14.1 Key Personnel** (Schedule 5 — 6 named roles):
  - Substitution = **30 days advance notice + PD written approval** (Security/DevOps Lead = 14 days)
  - Substitute must have equal-or-superior skill, experience, project familiarity, availability
  - **Continuity Plan required before Discovery Gate Clearance** — backup personnel · knowledge transfer · repo/build continuity · escalation when any Key Personnel unavailable >5 consecutive BD or materially reduced allocation
  - Senior Squad cannot be replaced with junior personnel after execution without PD approval
- **§14.2 Subcontracting** — no material subcontract without prior PD written approval; subcontractors bound by equivalent or stricter obligations
- **§14.3 Personnel vetting & training** before access granted

## §15 Defects, Warranty, Hypercare

|---|---|---|

| **SEV-2** | Gate-blocking; major feature failure; significant perf breach; WCAG blocker; major OCS failure | **Fix or workaround 3 BD** |
| **SEV-3** | Material non-critical; user-facing; doc defect; minor perf | **Fix 10 BD or next sprint** |
| **SEV-4** | Cosmetic; non-material | Prioritised by agreement |

- **Hypercare 90 days** — concurrent with warranty (NOT consecutive). Launch monitoring · priority triage · SEV-1/2 · App Store deployment · companion website go-live · handover · operational readiness
- **§15.5** — Gate Clearance does NOT waive latent defects

## §20 Termination

## §21 Variation

- **§21.1 Grounds** — material change to scope/cost/schedule/delivery/risk OR Specification ambiguity. **Contractor non-compliance with clear spec is NOT grounds**
- **§21.2 Proposal format** — specific clause/spec doc · ambiguity/change · proposed solution · cost · schedule · risk · affected gate · evidence
- **§21.3 Approval** — only effective with PD written approval + CAR-5026 record
- **§21.4 No implied variation** — no discussion/email/meeting/suggestion/invoice/delay/omission constitutes a Variation

## Key takeaways

- **Hard launch date:** 13 Nov 2026 (liquidated damages apply pre-GA, exclusions in §5.5.4)
- **Survival Core integrity is the central non-negotiable** — many clauses (8, 9.3, 11, 11A, 14, 22) reinforce this
- **Live registers (§7)** are contractual obligations — 9 registers required, all PD-accessible at all times
- **AI-tool register + AI code controls (§10.6)** = new control surface most teams don't pre-establish
- **Schedule 9 = condition precedent to environment access** — onboarding cannot proceed without it
- **IP assignment + Moral Rights Consent Deed = condition precedent per-individual** to repo access
- **Hypercare is mandatory** (not optional), included in Contract Sum, runs concurrent with Warranty
- **§11A Design Quality Obligation** = NEW gate — PD can withhold Clearance on E&I Layer design grounds even if functional spec passes (RT-23 introduced)
- **§14 Senior Squad lock-in** — substitution requires 30 days advance + PD approval (14d for Security/DevOps Lead)
- **§22 Audit response 5 BD** — must be operationally ready to satisfy
