# OSM-5026 · Overlay Semantics Map

**Source:** `04_SUPPORTING GUIDES FRAMEWORK DOCUMENTS/trackaroo OSM-5026 Overlay Semantics Map 20260408.pdf`
**Date issued:** 1 Apr 2026 · Ian Moore — Founder & Project Director
**Subtitle:** Phase 1 — Authoritative Visual Semantics Standard
**Supersedes:** OSM-3026 (23 Feb 2026) — fully superseded

> "OSM-5026 is the **sole and exclusive authority** governing all overlay visual semantics for trackaroo® Phase 1. No other document in the governance stack may define, reinterpret, or override visual encoding rules declared in this document."

**Role in Authority Stack:** Priority 6 specialist (Tier 2). Subordinate to UXS-5726, PRD-5126, FSD-5126, TQP-5026, WFD-5126. References from MAS-5126 (basemap compatibility), HFG-5026 (feed ingestion), CAR-5026 (amendment register), VGD-5126 (vendor interpretation).

This document establishes:
- **3 independent data classification systems** underpinning all overlay states: Hazard Feed Data, Environmental Feed Data, Track Verification Data
- **Point Condition Report (PCR) framework** — 4th supersession-based data system for discrete physical-state events
- **TTL threshold values** for each data system, calibrated to volatility + update cadence
- **Difficulty grade visual encoding rules**
- **Freshness shield visual encoding** + separation from difficulty state
- **PCR visual treatment + interaction model**
- **Phase 1 valid overlay state matrix**

> Material changes from OSM-3026: replacement of single 90-day TTL with 3-system calibrated TTL model · introduction of PCR framework · expansion from 2 overlay layers to 4 data systems · updated document ID series.

> **CONSTRAINT:** No document may define/reinterpret/override overlay visual semantics without formal amendment to OSM-5026 recorded in CAR-5026. Downstream docs that reference overlay states must cite OSM-5026 as authority + may NOT introduce independent definitions.

### §3.4 Road Classification Scope (PD decision · 1 April 2026)
- **Primary sealed roads** (bitumen/concrete) — basemap rendering only · NO TrackIQ™ difficulty grade · NO Track Verification Shield · PCR submission for PCR-CLO (closure) + PCR-SRF (surface) only
- **Secondary unsealed roads** (gravel/dirt) — in scope for TrackIQ™ difficulty grading · default = Ungraded/Grey until processed through scoring pipeline · all 6 PCR categories applicable
- **Track Verification Shield** applies to secondary unsealed roads on same basis as tracks

## §4 Technical Constraints & Prohibitions (absolute)

Apply regardless of connectivity state, device class, runtime pathway, SDK presence, dormant module presence, feature-flag state, or conditional compilation branch. Non-negotiable.

> **CONSTRAINT:** Presence of prohibited logic — even **dormant**, feature-flagged, or in non-executing branch — = non-compliance. Static analysis must confirm absence before acceptance.

### §4.1 Overlay Logic MUST Be:
- **Deterministic** — identical inputs produce identical outputs at all times
- **Event-triggered only** — state transitions occur only on defined trigger events, never on polling or background recalculation
- **Offline-capable** — all state transitions must occur without any network dependency
- **Telemetry-independent** — no usage/behaviour data may influence overlay state
- **Density-independent** — volume/frequency of user reports may not weight or modify overlay state
- **Geography-independent** — location/region may not alter TTL thresholds or state logic
- **Device-independent** — device class/OS version/screen type may not alter overlay behaviour

### §4.2 The Overlay System MUST NOT Include:
- Adaptive decay curves or variable freshness windows
- Telemetry-weighted freshness or trust scoring
- Density-weighted trust or crowdsource-derived quality scores
- Confidence scoring or composite trust metrics
- Heuristic recalculation of any overlay state
- AI inference or machine learning applied to overlay logic
- Background recalculation engines or polling loops
- Countdown timers visible to user
- Animated decay indicators or visual degradation effects
- Blended difficulty/freshness semantics or composite safety indication
- Derived freshness variables, weighted averages, shadow freshness fields
- Automatic verification triggered by proximity, map viewing, or behavioural detection

> **RELEASE GATE:** Static analysis validation must confirm absence of telemetry-weighted, density-weighted, adaptive, heuristic, or AI-based freshness mutation logic within all overlay modules. **Mandatory pre-release check at every gate — Alpha, Beta, GA.** Any detection halts release progression.

## §5 The Four Overlay Data Systems

| System | Data Owner | Update Mechanism | Volatility | Resolution Model |
|---|---|---|---|---|
| **1. Hazard Feed Data** | Government authority (BOM, AFAC, state agencies) | Automated feed ingestion | High — hours | **TTL decay** (time-based) |
| **2. Environmental Feed Data** | Government authority (BOM, parks, fire agencies) | Automated feed ingestion | Medium — hours to days | **TTL decay** (time-based) |
| **3. Track Verification Data** | trackaroo® user community | User submission on passage | Low — days to weeks | **TTL decay** (time-based) |
| **4. Point Condition Reports (PCR)** | trackaroo® user community | User submission — discrete event | Event-driven — indeterminate | **Supersession** (event-based) |

> 4 systems must be treated as **architecturally independent**. No visual encoding may imply a relationship between systems that does not exist.

## §5A Navigational Instrument Overlays

Navigational instrument overlays are map canvas elements that communicate user's current navigation state. **Distinct from the 4 data systems in §5** — they carry no data-system semantics, no difficulty state, no freshness signal, no hazard/condition info. They exist solely to orient user in context of current journey.

2 navigational instrument overlays defined for Phase 1:
- **(a) Active Navigation Route Line** — communicates user's intended forward path
- **(b) Breadcrumb Trail** — communicates user's traversed path. In BackTrack™ mode, breadcrumb trail becomes primary navigational instrument + rendering changes to signal active reverse navigation

### §5A.2 Active Navigation Route Line
- **Semantic rule:** Communicates forward navigation intent only · NOT difficulty, freshness, hazard, verification, PCR, or any data attribute
- **Rendering:** 3-pass centred overlay — (1) Difficulty colour polyline widest pass ~10px · (2) Dark casing #1A1A1A centre pass ~7px · (3) **Yellow fill #FFD600** innermost pass ~4px
- **Z-index position:** Above difficulty colour layer, hazard overlay, breadcrumb trail · Below PCR markers
- **BackTrack™ mode behaviour:** Route line **clears** when BackTrack™ activated. Breadcrumb trail becomes primary navigational instrument. **Yellow is NEVER simultaneously present as both solid route line and dashed retrace trail.**

### §5A.3 Breadcrumb Trail

**Standard mode (navigation active or idle — BackTrack™ not activated):**
- **Rendering:** (1) Difficulty colour polyline ~10px shared outer pass · (2) Dark casing #1A1A1A ~4px dashed centre · (3) **White dashes #FFFFFF ~2px innermost** (5px on, 4px off)
- **Direction:** Static — direction-neutral

**BackTrack™ active mode:**
- **Rendering:** Same (1) + (2) · (3) **Yellow dashes #FFD600 ~2px innermost** (5px on, 4px off)
- **Direction:** Animated — most recent breadcrumb to oldest · reinforcing direction of travel back toward origin
- **Z-index:** Above difficulty/hazard · Below PCR markers (active nav route cleared in BackTrack™ mode — no z-index conflict)

**Gap treatment (both modes):**
A gap = elapsed time between consecutive breadcrumb records exceeding **3× standard capture interval** (>60 seconds in standard mode; >15 seconds in distress mode). Both dark casing + inner dashes cease at gap boundary — produces **visible void** in trail. User must NEVER be uncertain whether section of trail is absent vs traversed.

> **CONSTRAINT:** Yellow must NEVER appear simultaneously as both active navigation route line fill (solid §5A.2) AND BackTrack™ retrace trail fill (dashed §5A.3 BackTrack™ active mode). Route line clears on BackTrack™ activation. **Yellow exclusively communicates direction of active navigation — forward when solid, reverse when dashed. Non-negotiable.**

### §5A.5 Z-Index Summary Table (Phase 1 canonical rendering order)

| Z-Index | Layer | Notes |
|---|---|---|
| 1 (bottom) | Basemap | Base cartographic render (MAS-5126) |
| 2 | Difficulty colour polyline | Widest pass — shared outer for breadcrumb + route line |
| 3 | Hazard overlay | HazTrack™ feed indicators (HFG-5026) |
| 4 | Breadcrumb trail | Above difficulty + hazard; below route line + PCR markers |
| 5 | Active nav route line | Above breadcrumb; clears in BackTrack™ mode |
| **6 (top)** | **PCR markers** | **Always topmost navigational layer; MUST NOT obscure difficulty or shield on track line** |

> PCR markers occupy topmost position to ensure tappability + visibility. Icon touch targets: **min 44×44pt; recommended 60×60pt** (WCAG 2.1 AA + glove-compatibility requirement).

## §6 Difficulty Grade Layer

### §6.2 Colour Encoding

| Grade | Colour Name | Hex Code | Application Notes |
|---|---|---|---|
| Easy | Green | **#4CAF50** | Suitable for all vehicles/fitness within archetype context |
| Moderate | Blue | **#1976D2** | Requires appropriate vehicle/fitness; conditions within normal expectations |
| Difficult | Black | **#212121** | Requires specific capability; challenges expected |
| **Extreme** | **Signal Orange** | **#FF5C00** | Maximum difficulty; expert only; **reserved exclusively for this grade** |
| Unknown / Ungraded | Grey | **#9E9E9E** | No authoritative grade data available |

> **CONSTRAINT:** **Signal Orange (#FF5C00) reserved exclusively for Extreme difficulty grade.** MUST NOT appear in any other visual context. Difficulty colour must NEVER represent freshness, verification state, hazard status, or any other data attribute. Governance breach if violated.

### §6.3 Grade Change Authority
Difficulty grade changes require **verified grading review by authorised contributor**. Automated inference, TrackIQ™ scoring output, or hazard feed events must NEVER alter a difficulty grade. The grade persists until explicitly superseded by an authorised review.

### §6.4 Display Rules
- Colour applied to track polyline on base map
- Consistent across all zoom levels within downloaded region
- Where WCAG 2.1 AA contrast conflicts with colour table, **secondary non-colour indicator** (icon, label, pattern) added — colour itself MUST NOT be changed
- Unknown/Grey tracks MUST be accompanied by disclosure indicator confirming grade unavailable
- No grade may be inferred/estimated/interpolated from adjacent track data — **Unknown remains Unknown** until authoritative scoring applied

## §7 Track Verification Shield Layer

### §7.2 Shield States

| Shield State | Visual Treatment | TTL Threshold | Meaning |
|---|---|---|---|
| **Gold Shield** | Gold badge icon on track segment | **≤14 days** since last user verification | Track verified by trackaroo® user within past 2 weeks; conditions likely representative |
| **Grey Shield** | Grey badge icon | 15–28 days since last verification | Track verified but verification becoming stale; conditions may have changed |
| **No Shield** | No badge displayed | >28 days, or never verified | Track not yet verified, or verification too old to be reliably representative |

> **CONSTRAINT:** Colour must NEVER represent freshness. **Shield state is sole visual carrier of verification freshness.** A gold/grey shield must NEVER change colour to indicate difficulty or hazard state. Shield state transitions driven **exclusively** by elapsed time since last verification submission — no other trigger permitted.

> **NOTE — Snow & Alpine seasonal consideration:** In alpine environments, track conditions can change within 24–48 hours due to snowpack, access, weather. Standard 14-day Gold window may be misleading. Flagged as Phase 2 design consideration — archetype-contextualised TTL thresholds. **Phase 1 applies uniform thresholds across all archetypes.**

### §7.3 Valid Verification Event Definition

A valid verification event = sole mechanism by which track segment's shield state resets to Gold + TTL clock restarts.

**A valid verification event MUST:**
- Originate from intentional, in-app user action performed while at or near track segment
- Be explicitly confirmed by user — passive proximity or map interaction NOT sufficient
- Be recorded with single authoritative timestamp: `verification_timestamp`

**A verification event MUST NOT be triggered by:**
- Passive telemetry or background location detection
- GPS proximity alone without explicit user confirmation
- Background app presence or foreground map viewing
- Density modelling or inferred repeat visit patterns
- Behavioural revisit detection or stop-detection logic
- **System inference of any kind**

> **CONSTRAINT:** Verification must remain **fully user-controlled** at all times. Any mechanism that could automatically update `verification_timestamp` without explicit user action = critical governance violation + release blocker.

### §7.4 Freshness Data Model
Freshness evaluation relies exclusively on single authoritative data field: **`verification_timestamp`**. Sole source of truth for shield state calculation.

**Absolute data model rules:**
- `verification_timestamp` overwritten only on valid new verification event (§7.3)
- No derived freshness variables permitted
- No weighted averages across multiple verification events permitted
- No shadow freshness fields may exist in any data model
- Shield state calculated solely from: `current_time − verification_timestamp`, compared to TTL thresholds in §7.2

## §8 Hazard Feed Freshness Indicator

### §8.2 TTL Threshold Table — Hazard Feed Types

| Hazard Feed Type | Gold Threshold | Grey Threshold | No Indicator | Rationale |
|---|---|---|---|---|
| **Active fire / total fire ban** | ≤6 hours | 6–12 hours | >12 hours | High volatility; fire conditions can change within hours |
| **Flood / waterway warning** | ≤6 hours | 6–12 hours | >12 hours | Dynamic events; conditions change rapidly |
| **Road / track closure (emergency)** | ≤12 hours | 12–24 hours | >24 hours | Emergency closures typically persist; updates on event cycle |
| **Storm / severe weather watch** | ≤6 hours | 6–18 hours | >18 hours | BOM severe weather update cycles; typically 3–6 hours |
| **Earthquake / geophysical event** | ≤24 hours | 24–48 hours | >48 hours | Aftershock sequences taper; response window hours to days |
| **General alerts and advisories** (fallback — uncategorised) | ≤12 hours | 12–24 hours | >24 hours | Fallback category. Authority: CR04 Q21, 1 April 2026 |

### §8.3 Display Rules
- Hazard feed freshness indicators use **distinct visual treatment** — separate from Gold/Grey/No Shield track verification system
- Freshness indicator displayed adjacent to hazard overlay, **NOT on track line**
- **No alert or notification generated on staleness** — cached hazard data displayed with freshness indicator
- Empty hazard overlay regions DO NOT imply all-clear conditions
- Stale hazard data MUST NOT be suppressed or hidden — displayed with appropriate staleness indicator until superseded

## §10 Point Condition Report (PCR) Framework

### §10.1 Definition & Rationale
A **Point Condition Report (PCR)** = discrete, user-submitted report of specific physical-state event at defined point on track/trail/path. PCRs describe events that are **binary in nature** — a condition is either present or has been cleared — whose duration is indeterminate. They DO NOT conform to TTL decay model because elapsed time since report provides no reliable signal about whether the condition still exists.

**Examples:** fallen tree across trail · locked or blocked gate · washed-out creek crossing · landslide deposit · damaged bridge/infrastructure · track submerged or flooded · road closed by emergency services tape.

PCRs are **distinct from Track Verification Shields**. A shield communicates age of general condition assessment for a track segment. A PCR communicates presence of specific, named physical condition at specific point.

> **CONSTRAINT:** PCRs must NEVER use Gold/Grey/No Shield visual system. They are separate data layer with own visual treatment. Merging PCR state with shield system = false signal — shield transitioning to Grey would NOT indicate whether fallen tree had been cleared.

### §10.2 PCR Categories

| Category Code | Name | Examples | Archetype Relevance |
|---|---|---|---|
| **PCR-OBS** | Obstruction | Fallen tree, rockfall, debris across track | 4WD, Bushwalker, MTB, Snow & Alpine |
| **PCR-CLO** | Closure | Locked gate, emergency services closure, private access revoked | 4WD, Remote Pro, Fish & Hunt |
| **PCR-INF** | Infrastructure Damage | Damaged bridge, washed-out crossing, broken signage, missing marker | 4WD, Bushwalker, Remote Pro |
| **PCR-SRF** | Surface Condition | Deep mud, submerged track, ice, extreme corrugation, sand trap | 4WD, MTB, Snow & Alpine |
| **PCR-WTR** | Water Crossing | Flooded crossing, risen creek, altered ford depth or approach | 4WD, Bushwalker, Fish & Hunt |
| **PCR-HAZ** | Localised Hazard | Fallen powerline, unstable embankment, fire damage adjacent to track | 4WD, Bushwalker, Remote Pro |

### §10.3 Resolution Model — Supersession

PCRs resolved by **supersession**, NOT by TTL decay. A PCR remains active + visible until one of the following supersession events occurs:
- Subsequent user reports same point + confirms condition cleared — PCR marked resolved with reporting timestamp
- Subsequent user submits new PCR at same point — new report supersedes prior; prior is **archived, NOT deleted**
- PCR reporter marks their own report as resolved (self-resolution by original reporter)

> No automated expiry, clock-based suppression, or TTL-driven removal of PCRs permitted. **A PCR that has not been superseded remains active regardless of age.**

> **NOTE — Unconfirmed PCR ageing:** A PCR not confirmed or superseded for **>90 days** may be visually flagged as "unconfirmed — condition age unknown". Display state only — PCR remains active. 90-day threshold = Phase 1 default implementation parameter (FSD-5126 + TQP-5026 validated).

> **CONSTRAINT — PCR Confirmation Threshold (CR04 Q18, 1 April 2026):** Threshold for transition from unconfirmed state (**amber ring #FFA000**) to confirmed state (**red ring #D32F2F**) is exactly **2 independent reporter submissions** at the same PCR point.
> - First submission = unconfirmed PCR (amber ring)
> - Second independent submission from different anonymised archetype token at same location = transition to confirmed (red ring)
> - Threshold = exactly 2 — NOT 3 or more
> - **Token distinctness required** — same reporter submitting twice does NOT trigger confirmation transition. 2 submissions from same token at same point count as 1 submission for confirmation purposes.

### §10.4 Visual Treatment
PCRs displayed as point markers at reported location:
- **Distinct icon system** separate from difficulty + shield layers
- Each PCR category (§10.2) has unique icon · icon design governed by **MAS-5126 Part B**
- Active PCRs display category icon with coloured indicator ring: **amber #FFA000** for active unconfirmed · **red #D32F2F** for active confirmed (≥2 independent reporter submissions)
- Resolved PCRs NOT displayed on active map view · accessible in track detail/history view only
- Unconfirmed-age PCRs (>90 days, no supersession) display with **muted icon treatment + "Condition age unknown" label**
- PCR markers MUST NOT obscure difficulty grade or shield state on track line

### §10.5 PCR Submission UI Requirements

PCR submission flow = user-initiated action, triggered by user while viewing track or upon completing passage. UI requirements:
- Submission requires user to **select PCR category + confirm point location on map**
- Free-text description field: optional, max 280 characters, plain text only
- **No photo attachment Phase 1** (photo-enabled POIs deferred to Phase 2)
- Submission must be possible in **fully offline mode** — offline submissions queued + synced on next connectivity
- UI must NOT infer PCR from user behaviour (stopping, pausing, deviating from track). **PCR submission always explicitly user-initiated**
- On submission, user receives brief confirmation: **"Condition reported. Thank you."** No additional info, scoring, or social signal provided

> **CONSTRAINT:** PCR submission must NOT be triggered automatically from user behaviour, TrackIQ™ scoring, stop-detection logic, or any other automated mechanism. **Explicit user action = only valid submission trigger.** Non-negotiable, applies in all operating modes.

### §10.6 WFD-5126 Wireframe Requirements

Required wireframe states in WFD-5126 to support PCR framework:
- **PCR-WF-01:** Map view with active PCR marker displayed — all categories
- **PCR-WF-02:** PCR detail tap — expanded card showing category, submission date, reporter archetype (anonymised), confirmation count
- **PCR-WF-03:** PCR submission flow — category selection, point confirmation, optional description, submit
- **PCR-WF-04:** PCR submission in offline mode — queued state with offline indicator
- **PCR-WF-05:** Unconfirmed-age PCR display state ("Condition age unknown")
- **PCR-WF-06:** Resolved PCR — track detail/history view only

## §11 Layer Independence Rules (LIR-01 → LIR-06)

> Non-negotiable governance constraints applying to every implementation decision, UI state, vendor deliverable.

| Rule | Description |
|---|---|
| **LIR-01** | Difficulty colour must NEVER represent freshness, verification state, hazard status, environmental condition, or PCR state |
| **LIR-02** | Freshness shield state must NEVER represent terrain difficulty, hazard status, environmental condition, or PCR state |
| **LIR-03** | Hazard feed freshness must NEVER be expressed using Gold/Grey/No Shield visual system applied to track verification. Separate indicators with separate visual treatments |
| **LIR-04** | PCR markers must NEVER use the shield icon system. PCR state must NEVER be inferred from shield state or difficulty grade |
| **LIR-05** | **No overlay layer may mutate any other overlay layer.** HazTrack™ feed events must NOT alter difficulty grade or shield state. PCR submissions must NOT alter difficulty grade or shield state |
| **LIR-06** | Visual representation must NOT rely on colour alone for any safety-relevant distinction (WCAG 2.1 AA). All overlay states must be distinguishable by shape, icon, label in addition to colour |

## §12 Rollback Governance (RG-01 → RG-11)

The 11 rollback trigger codes are detection conditions for prohibited overlay mutation logic. Detection of any trigger at any release gate (Alpha, Beta, GA) halts release progression immediately. **All 11 must be evaluated at each gate** — clear result against all 11 = mandatory condition for gate progression.

(Full RG table extracted in TQP-5026.md §5.10. Authority for the definitions = this section of OSM-5026.)

| Code | Condition | Detection |
|---|---|---|
| RG-01 | Adaptive freshness window — TTL varies at runtime based on any input | Static + runtime test |
| RG-02 | Telemetry influence on overlay state — usage/location/behavioural data alters overlay | Static + data flow audit |
| RG-03 | Density weighting — volume of user reports weights/modifies shield or PCR | Static + input variation test |
| RG-04 | Composite trust logic — any scoring/ranking/trust tier applied to verification or PCR | Static analysis |
| RG-05 | Countdown timers visible to user in any overlay context | UI review all screen states |
| RG-06 | Background recalculation engine — overlay state recalculates without explicit trigger | Static + background process audit |
| RG-07 | Blended visual semantics — difficulty + freshness combined in single visual element | UI review all screen states |
| RG-08 | Derived freshness field — any field other than `verification_timestamp` used | Data model audit |
| RG-09 | Automatic verification trigger — `verification_timestamp` updated without explicit user action | Static + integration test |
| RG-10 | PCR triggered automatically — PCR submitted without explicit user action | Static + integration test |
| RG-11 | Dormant prohibited logic in codebase — even if feature-flagged or non-executing | Static analysis of full codebase |

> **CONSTRAINT:** All 11 rollback triggers must return clear result at every release gate. Detection = automatic release halt. Remediation + full revalidation of Overlay Semantics & PCR Framework domain required before gate re-entry. Written clearance from PD required before resumption.

## §13 Phase 1 Valid Overlay State Matrix (8 valid combinations)

Sole authority for permitted state combinations across Difficulty Layer × Verification Shield × PCR.

| # | Difficulty | Verification Shield | PCR | Meaning |
|---|---|---|---|---|
| 1 | Known | No Shield | None | Graded; not yet verified |
| 2 | Known | Grey Shield | None | Graded + verified; verification stale |
| 3 | Known | Gold Shield | None | Graded + verified; verification fresh |
| 4 | Unknown (Grey) | No Shield | None | No authoritative grade data; not yet verified |
| 5 | Known | No Shield | PCR active | Graded; not verified; active condition report |
| 6 | Known | Grey Shield | PCR active | Graded; stale verification; active condition report |
| 7 | Known | Gold Shield | PCR active | Graded; fresh verification; active condition report |
| 8 | Unknown (Grey) | No Shield | PCR active | Ungraded; active condition report present |

> Any overlay state combination NOT listed in this 8-state matrix = **QA failure**. All 8 must be validated at every release gate (TQP-5026 §5.9.12).
