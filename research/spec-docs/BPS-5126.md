## 3. Core Battery Benchmarks

### 3.1 Primary Benchmark — Active Navigation

> **PRIMARY BENCHMARK (non-negotiable):** **≤8% battery consumption per hour during active navigation**, measured under BPS-5126 standardised conditions (§5). Acceptance criterion across all devices in validated device envelope. Device-specific exceptions require written Founder & Project Director approval with justification.

### 3.2 Endurance Benchmark

> **ENDURANCE BENCHMARK:** **≥10 continuous hours of active navigation in a zero-connectivity (airplane mode) state**, with all active Phase 1 features running (navigation engine, HazTrack™ overlays, breadcrumb logging, TrackMate™ BLE Mesh background, PCR overlay passive display). Device must **not fall below 20% battery at the 10-hour mark when starting from 100%**. Deterministic posture — no favourable condition assumptions.

### 3.3 TrackMate™ Battery Constraint

TrackMate™ BLE Mesh must operate as continuous background process without materially compromising the ≥10-hour endurance benchmark.

| Component | Battery Model | Benchmark Compliance Rule |
|---|---|---|
| **BLE Mesh (Tier 1 — Primary)** | Very low — continuous background | Must not add more than **0.5% per hour** to baseline navigation consumption. Vendor must quantify and confirm. |
| **Wi-Fi Direct (Tier 1 — Fallback)** | Medium — on-demand only | Must not be held continuously active. On-demand activation must not compromise endurance at reasonable frequency. Vendor must model worst-case activation scenario. |
| **LoRa peripheral (Tier 2 — optional)** | Very low — hardware-managed | Hardware power consumption is peripheral-managed. App-layer integration must not introduce material drain beyond peripheral self-reporting. |

### 3.4 PCR Overlay Rendering Load

OSM-5026 establishes the **Point Condition Report (PCR) framework as a fourth active overlay data system** in Phase 1. PCR icons rendered as discrete point markers on the map with real-time state transitions (active unconfirmed, active confirmed, resolved-suppressed). Rendering contributes to Phase 1 active feature load and must be accounted for in standardised battery test conditions (§5).

| PCR Feature Component | Battery Model | Inclusion in Test Load |
|---|---|---|
| PCR icon rendering (passive display) | Negligible — static icon render | Included in standard active feature set (§5 Active Features row). PCR overlay module active with representative PCR density loaded. |
| PCR submission UI (user-initiated) | Negligible — event-driven only | Not included in sustained endurance load. Tested separately as event spike scenario. |
| PCR state polling / sync | Zero — offline mode, no polling | Airplane mode test conditions eliminate network-based PCR sync. Offline PCR state is static during test. |

> **CONSTRAINT:** PCR sync, refresh, or state-update operations must not introduce any connectivity dependency in offline mode. PCR overlay state during offline test runs must be derived **exclusively from the last locally cached dataset**. Any implementation that polls for PCR updates during offline navigation **violates the Survival Core offline-first rule and is a rejection trigger under VGD-5126**.

### 3.5 SOS / Distress Mode Battery Scenario

SOS activation triggers a **materially different system load profile** from standard navigation. GPS acquisition shifts from dual-trigger interval (15m / 20s) to **continuous high-frequency acquisition (5m / 5s)**. Breadcrumb writes increase to every record. AES-256 encryption and WAL write operations sustained at elevated frequency. Structurally distinct from §3.1 primary benchmark — requires separate acceptance scenario. **Authority: CR02 Q1, 31 March 2026.**

| Parameter | Condition | Notes |
|---|---|---|
| GPS acquisition | Continuous high-frequency — 5-second cycle | Mirrors distress mode breadcrumb interval |
| Breadcrumb capture | 5m / 5s dual-trigger — distress mode rate | Maximum capture density; WAL write on every record |
| AES-256 encryption | Active and continuous | Every breadcrumb record encrypted at write |
| Screen state | On — SOS confirmation screen active and persistent | Represents field use; screen remains active during distress |
| All other features | As per §5 standardised conditions | Navigation engine, HazTrack™, TrackMate™ BLE background active |
| Connectivity | Airplane mode — zero connectivity | Zero Transmission posture; ESF-5026 §4.2 |
| Duration | Minimum 2 continuous hours per test run | Validates sustained distress operation |
| Sample size | Minimum 3 test runs per device | Consistent with primary benchmark methodology |

> **SOS / DISTRESS MODE ACCEPTANCE THRESHOLD:** Battery consumption during sustained SOS / distress mode operation must **not exceed ≤20% per hour**, averaged across 3 runs per device on all devices within the approved validation matrix. Acknowledges elevated load relative to standard navigation — **not a relaxation of the §3.1 primary benchmark**. Separate test scenarios with separate acceptance criteria.

> **NOTE:** SOS / distress mode tests sustained operation after SOS activation — does not modify or supersede §3.1. Both benchmarks must be independently satisfied. The ≤20%/hr threshold ensures a device at 80% battery at SOS activation sustains a minimum of 4 hours of distress mode operation — the governing field survival envelope.

> **CONSTRAINT:** SOS-triggered distress breadcrumb mode (5m / 5s) is a **Survival Core function. It activates unconditionally upon SOS confirmation for all users on all tiers — including Free tier. No tier boundary may restrict, delay, or degrade distress mode activation or operation.** The ≤20% per hour distress mode battery threshold applies equally across all tiers. Authority: FSD-5126 §4.2.2; CR04 Q8, 1 April 2026.

## 4. Device Validation Envelope

### 4.1 Hardware Generation Scope

| Platform | Scope | Excluded |
|---|---|---|
| **iOS** | Current generation + 2 prior generations at time of Alpha gate (end-June 2026) | Devices older than current − 2 generations. iPads unless explicitly added. |
| **Android** | Current generation flagship + 2 prior generations of 3 nominated representative manufacturers at time of Alpha gate | Low-tier or budget Android devices unless explicitly added. Older than current − 2 generations. |

> **DEVICE MATRIX CONFIRMATION:** The specific device list constituting the Phase 1 validation matrix is a **vendor response obligation (V-07, PRD-5126 §12)**. Delivery partner proposes list for Founder & Project Director approval before battery testing commences. Approved list documented in TQP-5026 and ratified against BPS-5126 scope rules. **Unapproved devices do not count toward benchmark satisfaction.**

### 4.2 Device Exclusion Rules
- Devices with degraded batteries (below 80% of original rated capacity) excluded
- Devices running non-standard OS builds or beta OS versions excluded
- Devices with active background processes outside test control set excluded
- Battery consumption results from excluded devices must not be used to satisfy any BPS-5126 acceptance criterion

## 5. Standardised Test Conditions

| Test Parameter | Standardised Condition | Rationale |
|---|---|---|
| Connectivity | **Airplane mode** — zero cellular, Wi-Fi, and satellite connectivity throughout | Validates survival-grade offline endurance |
| Starting battery level | 100% — fully charged before each test run | Ensures consistent baseline |
| Screen state | **Screen on** — navigation view active and visible throughout | Represents real-world use; screen is primary drain contributor |
| Screen brightness | **50% brightness** — consistent across all devices | Neutralises screen brightness as a variable |
| Active features | Navigation engine + breadcrumb logging + HazTrack™ overlays + TrackMate™ BLE Mesh background + TrackIQ™ scoring display (passive background) + First Aid Reference module (passive — no active timer) + **PCR overlay module (passive — representative PCR density loaded, no active submission or sync)** | Represents full Phase 1 confirmed scope active session load. **First Aid Reference active timer state and PCR submission events tested separately as battery spike scenarios — not included in sustained endurance load.** |
| Temperature | **18°C–24°C ambient** — standard laboratory conditions | Eliminates temperature as a variable; cold/heat degrade performance |
| Location simulation | GPS signal simulation with continuous position updates — no static GPS state | Ensures GNSS is active as in field conditions |
| Duration | **Minimum 10 continuous hours per test run** | Validates endurance benchmark directly |
| Sample size | **Minimum 3 test runs per device** — results averaged | Eliminates single-run anomalies |
| Measurement method | OS battery API + external measurement logging — both must agree within **±1%** | Prevents single-source measurement error |

## 6. Performance Validation Envelope

### 6.1 Acceptance Thresholds

| Metric | Acceptance Threshold | Failure Condition |
|---|---|---|
| Battery consumption — active navigation | **≤8% per hour** (averaged across 3 runs per device) | Any single device in approved matrix exceeds 8%/hr on average — rejection trigger |
| Offline endurance | **≥10 hours** before falling below 20% remaining | Battery falls below 20% before 10 hours on any approved device |
| BLE Mesh overhead | **≤0.5% per hour** additional above baseline navigation | BLE Mesh overhead exceeds this threshold on any approved device |
| **≤20% per hour during sustained SOS / distress mode** | ≤20% per hour during sustained distress mode operation | Any single device exceeds 20%/hr on average under §3.5 distress mode test conditions |
| Cold start — navigation ready (map bundle ≤500MB) | ≤6 seconds to navigation-ready state | Exceeds 6s on any approved device with ≤500MB bundle in standard test conditions |
| Cold start — navigation ready (map bundle >500MB) | ≤8 seconds to navigation-ready state | Exceeds 8s on any approved device with >500MB bundle in standard test conditions. Cold start exceedances attributable to coverage scope decisions by Trackaroo Systems are not treated as vendor benchmark failures — see FSD-5126 §10. |
| Map tile render | ≤2 seconds initial tile layer render | Exceeds 2s on any approved device in standard test conditions |
| **SOS — UI response** (visual transition to active SOS state, including confirmation feedback display) | **≤0.5 seconds from final confirmation tap** — unconditional | UI response exceeds 0.5s on any approved device under any battery state within test envelope |
| **SOS — First GPS fix + AES-256 encryption + WAL write** (first immutable log record) | **≤3 seconds from final confirmation tap — GPS warm state** | First log record exceeds 3s on any approved device in GPS warm state within test envelope. GPS cold-state acquisition time excluded — see FSD-5126 §10 for cold-state UI indicator obligation. |
| PCR overlay render at load | No measurable delta above HazTrack™ overlay baseline render time | PCR rendering causes detectable sustained increase in battery drain rate above baseline |

### 6.2 Vendor Evidence Requirements

Delivery partner must provide as part of Beta-Ready gate submission:
- Battery test results for all approved devices — 3 runs per device, per-hour consumption data
- Test environment configuration documentation confirming all BPS standardised conditions met
- **BLE Mesh overhead isolation** — separate measurement isolating BLE Mesh contribution
- **Wi-Fi Direct on-demand activation modelling** — battery impact at reasonable activation frequency
- **PCR overlay render confirmation** — delta measurement confirming PCR icon rendering does not create detectable sustained increase above HazTrack™ baseline
- **SOS / distress mode battery test results** — 3 runs per device under §3.5 conditions, confirming ≤20%/hr threshold met
- Founder & Project Director review and approval of battery test plan before testing commences (TQP-5026 §4.1 EC6)

> **BATTERY BENCHMARK — GATE CONDITION:** Battery benchmark (≤8%/hr, ≥10hr endurance) is a named exit criterion (**XC9 in TQP-5026**) and a Beta → GA gate condition. **SOS / distress mode benchmark (≤20%/hr) is a co-located gate condition.** Written confirmation from the Project Director, Trackaroo Systems Pty Ltd, required. **No release build may proceed to GA without this confirmation.**

## 8. Closing Statement

BPS-5126 establishes:
- Primary benchmark — **≤8% battery consumption per hour** under standardised conditions
- Endurance benchmark — **≥10 hours** continuous offline active navigation, device not below 20% at 10-hour mark
- TrackMate™ battery constraints — BLE Mesh ≤0.5%/hr overhead; Wi-Fi Direct on-demand only
- PCR overlay rendering load — passive display included in standard test load; submission/sync events excluded from sustained endurance measurement; offline-first prohibits network polling
- **SOS / Distress mode battery scenario — ≤20%/hr threshold; tier-agnostic; independently satisfiable alongside primary benchmark**
- Device validation envelope — current + 2 prior generations, iOS and Android; device matrix requires F&PD approval
- Standardised test conditions — ten defined parameters, all mandatory for valid measurement, updated to include PCR passive display in active feature set
- Performance validation envelope — acceptance thresholds for seven metrics including PCR render confirmation, SOS response, map render, cold start
- Vendor evidence requirements — battery test report, BLE isolation, PCR render delta, test environment documentation

All conflicts with Tier 1 authority documents escalate immediately to the Founder & Project Director. This document grants no implementation discretion — it defines governing rules within its declared domain.
