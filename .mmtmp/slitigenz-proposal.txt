# **2. TECHNICAL ARCHITECTURE RESPONSE**

## **2.1. Dual-Layer Architectural Strategy**

Slitigenz proposes a rigid **Dual-Layer Architecture** to satisfy the **"Survival-Grade"** mandate. This architecture establishes an **Immutable Separation Boundary** between the safety-critical **Survival Core** and the **Experience & Intelligence (E\&I) Layer**.

  - **Survival Core (Safety Pillar):** Contains Navigation, BackTrack™, HazTrack™, SOS, and Local Logging. It operates under strict deterministic, non-adaptive, and offline-first supremacy.
  - **Experience & Intelligence Layer (Capability Pillar):** Contains TrackMate™, TrackIQ™, Archetypes, and First Aid Reference Reference. This layer is designed to degrade gracefully when offline and is architecturally prohibited from blocking or mutating any Survival Core flow.

### **2.1.1. Evidentiary Integrity: Prohibited Mutation Taxonomy**

In accordance with **VGD-5126 §8.2** and **BTF-5126 §5.2**, Slitigenz confirms that our Survival Core implementation will architecturally prohibit the following fourteen (14) operations to ensure absolute "Evidentiary Integrity" of breadcrumb data:

1.  **Merge:** Combining breadcrumb sequences from multiple sessions.
2.  **Compaction:** Reducing stored points by consolidation.
3.  **Reconstruction:** Replacing captured points with derived positions.
4.  **Reordering:** Changing chronological sequences.
5.  **Map-matching:** Snapping breadcrumbs to road/track geometry.
6.  **Coordinate interpolation:** Filling gaps with estimated data.
7.  **Gap-filling:** Heuristic correction of signal loss areas.
8.  **Deduplication:** Removing redundant points regardless of proximity.
9.  **Compression:** Lossy coordinate reduction.
10. **Conflict resolution engines:** Algorithmic "reconciliation" of path data.
11. **Reconciliation engines:** Cloud-based overwrite of local data.
12. **Coalescing logic:** Grouping points into average clusters.
13. **Smoothing:** Applying noise-reduction filters to raw coordinates.
14. **Correction:** Retroactive coordinate modification post-write.

## **2.2. Survival Core Isolation Approach**

We enforce isolation through a **unidirectional dependency model**. The E\&I Layer may read states from the Survival Core, but no mechanism exists for the E\&I Layer to inject state, override triggers, or mutate the execution path of core functions.

  - **Execution Discipline:** All core modules execute deterministically. We confirm that no AI inference, telemetry-based weighting, or adaptive logic is present within the Survival Core.
  - **Contamination Prevention:** We will employ automated static analysis during the CI/CD pipeline to detect and block any shared mutable state or coupled execution paths that blur the boundary between layers.

## **2.3. Firebase Isolation Model**

Slitigenz confirms a **Firebase-Independent** posture for the Survival Core.

  - **Safe-Store Persistence:** All Survival Core data—including position history, breadcrumbs, and SOS logs—will be persisted using device-native storage (e.g., encrypted SQLite with WAL) that is entirely independent of the Firebase Firestore SDK.
  - **Zero Transmission Boundary:** We confirm that all Survival Core paths generate **zero outbound packets** in Phase 1. No Firebase sync processes, cloud functions, or background polling engines will have access to core data stores.

## **2.4. Offline-First Implementation Strategy**

The offline state is the **primary operating mode** for the Slitigenz solution.

  - **Navigation & Mapping:** We will utilize the **Mapbox SDK with OSM vector tiles**. Initial map tiles will render within **≤2 seconds** from local region bundles, with no server-side handshake required for route guidance.
  - **Deterministic Logic:** System behaviors are event-triggered only; we confirm the absence of any periodic decay loops or background recalculation engines that depend on connectivity.

## **2.5. Breadcrumb Persistence & Forensic Integrity**

In compliance with **BTF-5126** and **VGD-5126**, Slitigenz classifies all breadcrumb data as **Local-Only and Non-Syncable**.

  - **Persistence Mechanism:** We will implement **AES-256 encryption at rest** for all locally persisted coordinates. To ensure **crash-survivability**, we will utilize a **write-ahead log (WAL)** to prevent data loss during forced app termination.
  - **Forensic Immutability:** Once written, breadcrumb records are immutable. We explicitly confirm the absence of the **14 Prohibited Mutation Behaviours**, including map-matching, coordinate interpolation, path smoothing, and gap-filling.
  - **Performance:** The system will capture breadcrumbs using **Dual-Trigger Logic** (15m or 20s in standard mode; 5m or 5s in distress mode). We commit to rendering 10-hour retrace trails within **≤3 seconds**.

## **2.6. Architectural Readiness (Phase 2 Scaffolding)**

While maintaining strict Phase 1 boundaries, our architecture includes exactly **three permitted inert scaffolds** to support Phase 2 expansion without structural rework:

1.  **Emergency Escrow Schema:** A versioned data model structured for forward migration.

# **APPENDIX C: GLOSSARY OF TERMS**

To ensure interpretative alignment with the **trackaroo® Phase 1 Authority Stack**, the following definitions apply to this proposal:

  - **Survival Core:** The safety-critical feature set (Navigation, BackTrack™, HazTrack™, SOS) that must execute 100% offline with zero network dependency.
  - **Experience & Intelligence Layer:** Functional enhancements (TrackMate™, TrackIQ™) that utilize Firebase and may degrade gracefully when connectivity is absent.
  - **CAL (Comms Abstraction Layer):** The subsystem managing all peer-to-peer transport priority and the four mandatory state flags (satReady, queueEnabled, offlineBeacon, partialSignal).
  - **PCR (Point Condition Report):** A discrete physical-state report (e.g., "Bridge Out") resolved by supersession rather than time-decay TTL.
  - **Inert Scaffold:** Non-executable source code structures included in Phase 1 solely to preserve future activation pathways (e.g., satellite relay hooks) without structural rework.
  - **Deterministic Execution:** A system state where identical inputs consistently produce identical outputs with no probabilistic or adaptive variance.
